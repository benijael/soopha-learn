Le Sample « update-checker » arrive sur ton bureau

Un faux utilitaire Linux, reçu après une alerte EDR sur un poste développeur

📥 Ce que tu sais au départ
  • Nom du fichier : update-checker — présenté comme un outil interne de diagnostic
  • Hash inconnu dans l'environnement — aucun historique de déploiement légitime
  • L'EDR signale une exécution suivie d'un appel réseau suspect puis d'une sortie rapide
  • Pas de source, pas de symboles utiles, pas de documentation
  • Ton rôle : comprendre le comportement sans brûler les preuves
🎯 Mission du capstone
  • Trier le binaire sans l'exécuter au hasard
  • Formuler des hypothèses à partir d'artefacts tangibles
  • Confirmer / infirmer avec Ghidra et GDB
  • Contourner une couche d'obfuscation et d'anti-debug
  • Extraire des IOCs et produire un rapport clair
  • Tu dois agir comme un analyste, pas comme un touriste du RE
"Le vrai reverse engineering commence quand tu ne cherches plus juste à comprendre le binaire, mais à décider intelligemment de la prochaine action. — SooPHA Learn"
// Slide 03 · Triage initial · Réflexes minimums

Triage — Les 5 premières minutes décident du reste

Avant Ghidra, avant GDB, tu dois cadrer le sample

analyst@lab:~$ sha256sum update-checker
9f14...d3c1 update-checker

analyst@lab:~$ file update-checker
ELF 64-bit LSB pie executable, x86-64, dynamically linked, not stripped

analyst@lab:~$ strings -n 6 update-checker | head -20
checking update source...
Debugger detected
/api/v1/ping
libcurl.so.4

analyst@lab:~$ readelf -h -S update-checker | head -25
✅ Ce que ce triage te dit déjà
  • ELF x86-64 PIE — attention aux adresses runtime et à l'ASLR
  • Dynamically linked — imports intéressants à explorer rapidement
  • Not stripped — avantage énorme pour la première passe
  • Strings réseau + anti-debug — double piste comportementale immédiate
🚫 Ce qu'il ne faut pas faire
  • Double-cliquer et "voir ce que ça fait" sur une machine normale
  • Conclure qu'il est bénin parce qu'il n'est pas stripped
  • Se jeter dans le décompilateur sans hypothèse initiale
  • Ignorer les imports et les sections avant de debugger

Construire un Plan d'Analyse Avant d'ouvrir les outils

Un bon reverse engineer choisit son ordre d'attaque

1
Identifier les points d'entrée — main, imports réseau, messages d'erreur, fonctions de validation, wrappers anti-debug

// Slide 05 · Ghidra · Cartographie du sample · QCM requis

Première Passe Ghidra — Cartographier sans se perdre

Tu cherches la structure, pas encore tous les détails

🔬 Ce que tu repères dans Ghidra
  • main appelle d'abord une fonction de vérification d'environnement
  • Puis une routine renommée build_payload() construit un buffer octet par octet
  • Une fonction compare ensuite le résultat à une chaîne attendue avant d'appeler libcurl
  • Une branche d'échec imprime Debugger detected puis sort immédiatement
  • Le décompilateur te donne la carte globale en quelques minutes
🧭 Workflow Ghidra minimal
  • Importer et laisser tourner l'auto-analyse
  • Localiser main puis remonter les appels
  • Explorer les strings et leurs XREF
  • Renommer immédiatement les fonctions clés
  • Comparer listing asm et pseudo-code quand quelque chose sent mauvais
"Si Ghidra te montre une fonction confuse, ne lui fais pas un procès : il t'avertit peut-être qu'il y a de l'obfuscation, un mauvais typage, ou une logique volontairement tordue."
// Slide 06 · Pseudo-code · Hypothèse validation

Le Cœur de l'Échantillon — Une Chaîne Reconstruite à la Main

Quand le sample évite les strings statiques, il faut suivre la logique

// pseudo-code retypé dans Ghidra
int build_payload(char *buf) {
  buf[0] = 0x2f ^ 0x42;
  buf[1] = 0x23 ^ 0x42;
  ...
  for (i=0; i<0x12; i++) tmp[i] = buf[i] ^ key;
  if (check_debugger() != 0) return -1;
  return compare_magic(tmp);
}

// intuition : construction d'une chaîne obfusquée avant un check réseau
Signal 1
Construction caractère par caractère → probable stack string / XOR simple
pas de string claire dans .rodata
Signal 2
Fonction check_debugger() appelée avant la suite logique
garder un breakpoint dessus
Signal 3
Appel à compare_magic — parfait pour un breakpoint plus tard
avant strcmp ou memcmp
Signal 4
libcurl apparaît seulement après validation → comportement conditionnel
runtime indispensable
// Slide 07 · GDB · Exécution contrôlée

Passer en Dynamique au Bon Moment

La statique t'a donné une hypothèse ; GDB doit la prouver

🎯 Breakpoints prioritaires
  • main — vérifier le point de départ réel
  • check_debugger — voir le mécanisme exact
  • compare_magic — récupérer la valeur comparée
  • strcmp / memcmp — lire les buffers juste avant la comparaison
  • curl_easy_perform — confirmer l'appel réseau final
🧪 Ce que tu dois observer
  • Valeurs dans RDI / RSI avant les appels critiques
  • Contenu réel du buffer après décodage runtime
  • La condition qui fait bifurquer vers Debugger detected
  • Éventuelle différence entre exécution normale et exécution sous debugger
  • La dynamique sert ici à révéler ce que la statique suggère sans certitude
"Le meilleur breakpoint n'est pas forcément le premier appel trouvé. C'est celui qui te donne la vérité la plus vite."
// Slide 08 · GDB pratique · Lecture runtime · QCM requis

La Découverte Clé dans GDB

Une seule commande bien placée peut économiser 30 minutes de souffrance

pwndbg> b *compare_magic
pwndbg> run
Breakpoint hit at compare_magic

pwndbg> i r rdi rsi
RDI → 0x7fffffffdc20
RSI → 0x5555555581c0

pwndbg> x/s $rdi
"/api/v1/ping"
pwndbg> x/s $rsi
"/api/v1/ping"

// hypothèse confirmée : le buffer XOR reconstruit une route HTTP cachée
✅ Ce que tu viens de prouver
  • Le buffer reconstruit n'était pas arbitraire : il cachait bien une ressource réseau
  • La comparaison sert probablement de garde logique avant l'appel curl
  • Le runtime t'a permis de lire la vérité sans recoder manuellement toute la routine
⚡ Réflexe à mémoriser
  • x/s $rdi et x/s $rsi avant strcmp/memcmp
  • Inspecter avant de patcher
  • Confirmer la donnée avant de théoriser sur son usage
// Slide 09 · Obfuscation · Décodage raisonné

Décoder Sans Surréagir

Toute donnée encodée n'est pas du chiffrement avancé

🔓 Ici, l'obfuscation sert à quoi ?
  • Éviter une détection triviale par strings
  • Masquer la route HTTP et la logique de validation
  • Retarder un analyste pressé de quelques minutes
  • Ce n'est pas du chiffrement robuste ; c'est du camouflage opérationnel
🛠️ La bonne méthode
  • Repérer la clé ou l'opération répétée
  • Vérifier si le runtime révèle déjà le résultat
  • Sinon, recoder la routine en Python propre
  • Documenter la transformation pour le rapport
# reconstitution rapide
data = bytes([0x2f,0x23,0x32,0x2b,0x6d])
key = 0x42
print(bytes([b ^ key for b in data]))
b'/api/'
// Slide 10 · Anti-debug · Contournement raisonné · QCM requis

La Protection Anti-Debug n'est pas un Mur

Elle t'impose juste de comprendre ce qu'elle teste

🧨 Ce que révèle l'analyse
  • check_debugger() appelle un wrapper autour de ptrace
  • Si l'appel échoue, le programme considère qu'il est observé
  • Le sample affiche Debugger detected puis court-circuite le flux principal
  • Le but n'est pas de te stopper définitivement, mais de casser ton rythme
🩹 Stratégies propres de bypass
  • Breakpoint après le test puis forcer le registre de retour à 0
  • Patch temporaire du saut de la branche anti-debug
  • LD_PRELOAD ou instrumentation si le contexte s'y prête
  • Documenter la technique utilisée et ses limites
"Un anti-debug simple ne se combat pas avec l'ego. Il se combat avec une hypothèse, un test, et un bypass minimal."

// Slide 12 · Rapport · Restitution claire

Le Rapport Fait la Différence entre un Hacker et un Pro

Si tu ne sais pas restituer, tu n'as pas fini l'analyse

1
Résumé exécutif — ce que fait probablement le sample, niveau de risque, pourquoi on s'en soucie
2
Contexte technique — type de fichier, architecture, dépendances, présence d'obfuscation / anti-debug
3
Findings prouvés — route cachée, flux conditionnel, appel réseau, mécanisme ptrace
4
IOCs — hash, chaînes, imports, chemins, comportements observés
5
Recommandations — isolement, blocage, règles de détection, enrichissement TI, surveillance complémentaire
Prouvé Hypothèse À vérifier en sandbox Toujours séparer les trois
// Slide 13 · Synthèse méthodologique · Parcours complet

La Méthode Complète du Reverse Engineer

Le vrai livrable de ce capstone : un process réutilisable sur n'importe quel sample

Phase 01
Triage
Hash, file, strings, readelf, imports, sections. On cadre le terrain avant toute exécution.
Phase 02
Hypothèses
On note ce qui semble réseau, validation, obfuscation, anti-debug. On prépare le plan.
Phase 03
Statique
Ghidra pour cartographier, renommer, comprendre le flux principal et localiser les fonctions clés.
Phase 04
Dynamique
GDB pour lire la vérité runtime, casser au bon endroit, inspecter registres et mémoire.
Phase 05
Contournement
Bypass minimal d'obfuscation et d'anti-debug — seulement ce qui est nécessaire pour comprendre.
Phase 06
Restitution
IOCs, findings, risques, recommandations. On transforme l'analyse en valeur pour la défense.
Ne jamais improviser l'ordre Toujours documenter Toujours distinguer preuve et intuition
