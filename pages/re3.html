Analyse Statique — Comprendre Sans Exécuter

La première phase de toute analyse RE — examiner le binaire au repos

🔍 Analyse Statique
  • Examiner le binaire sans jamais l'exécuter — sécurité maximale pour les malwares
  • Lire le code assembleur, identifier les fonctions, reconstruire la logique
  • Outils : Ghidra, IDA Pro, radare2, Binary Ninja
  • Avantage : vue globale complète — accès à tout le code d'un coup
  • Limite : ne voit pas les comportements runtime (déchiffrement, réseau, packing)
⚡ Analyse Dynamique
  • Exécuter le binaire dans un environnement contrôlé (VM isolée)
  • Observer les comportements en temps réel : mémoire, réseau, fichiers
  • Outils : GDB, pwndbg, ltrace, strace, Wireshark
  • Avantage : valeurs réelles — voir ce qui se passe vraiment à runtime
  • Limite : ne voit que ce qui est exécuté — branches non prises ignorées
"La règle d'or : toujours commencer par l'analyse statique. Si elle ne suffit pas (packing, chiffrement runtime), on passe à la dynamique. Les deux se complètent — le pro maîtrise les deux."

Ghidra — L'Outil Libre de la NSA

Le décompilateur open-source le plus puissant — gratuit, multi-plateforme

📖 Qu'est-ce que Ghidra ?
  • Développé par la NSA (National Security Agency), open-sourcé en 2019
  • Supporte : x86, x86-64, ARM, MIPS, PowerPC, RISC-V et des dizaines d'autres architectures
  • Formats : ELF, PE (Windows), Mach-O, DEX (Android), firmware binaires
  • Moteur de décompilation Sleigh → génère du pseudo-code C lisible
  • Extensible via des scripts Java/Python (Jython)
  • Remplace IDA Pro (propriétaire, ~700$/an) pour la plupart des usages
⚙️ Installation

Interface Ghidra — Maîtriser le CodeBrowser

Les 6 panneaux essentiels et leur rôle dans l'analyse

📋 Symbol Tree
Gauche — Liste de toutes les fonctions, labels, imports, exports. Point d'entrée pour naviguer. Chercher une fonction par nom.
Raccourci : G → "Go to address/function"
🔧 Listing (Assembleur)
Centre — Vue désassemblée instruction par instruction. Adresses, opcodes, mnémoniques, commentaires. C'est ta référence absolue.
; → ajouter un commentaire sur la ligne
🧩 Decompiler
Droite — Pseudo-code C généré par le moteur Sleigh. Synchronisé avec le Listing. Cliquer sur une variable = highlight dans les 2 vues.
Raccourci : Ctrl+Shift+E → exporter le C
📊 Data Type Manager
Droite — Tous les types de données disponibles : structs, enums, typedefs. Importer des types depuis Windows API, POSIX, etc.
Glisser un type sur une variable pour la caster
🗺️ Function Graph
Window → Function Graph — Vue graphique du Control Flow Graph (CFG). Chaque bloc de base est un nœud, branches = arêtes. Indispensable pour les conditions.
Raccourci : V pour basculer Listing/Graph
🔗 References (XREF)
Clic droit → References → Show References → listes tous les endroits qui appellent ou accèdent à une fonction/variable.
Ctrl+Shift+F → trouver toutes les références
# Navigation rapide dans Ghidra
G        → Go to address       | L        → Rename (label)
T        → Retype variable      | Ctrl+Z → Undo annotation
Alt+← → Back in history    | Alt+→ → Forward in history

Renommer Variables & Fonctions — La Compétence Clé

Transformer un binaire illisible en code documenté — l'art du reverse engineer

🏷️ Renommer (L)
  • Clic sur variable → L → taper le nouveau nom → Enter
  • DAT_00104020 → encoded_flag
  • FUN_00101280 → check_flag
  • param_1 → user_input
  • Le renommage se propage dans tout le projet automatiquement
  • Toujours renommer main en premier → lit les appels de fonctions
📐 Retypier (T)
  • Clic sur variable → T → choisir le type correct
  • undefined8 → char * (pointeur de chaîne)
  • undefined4 → int (entier 32-bit)
  • Clic droit → Edit Function Signature → corriger le prototype
  • Importer des types Windows : File → Parse C Headers → windows.h
  • Le bon type = le bon comportement dans le décompilateur
💡 Stratégie de renommage — ordre optimal
ÉTAPE 1
main()
Identifier la structure globale, les appels de fonctions importantes, les constantes visibles
ÉTAPE 2
Fonctions filles
Suivre les appels depuis main → renommer chaque fonction selon son rôle apparent
ÉTAPE 3
Variables
Dans chaque fonction : param_1 → rôle réel, local_10 → counter/buffer/key selon l'usage
ÉTAPE 4
Constantes
Les magic numbers : 0x1d → FLAG_LENGTH, 0x42 → XOR_KEY — les nommer rend le code évident
// Slide 08 · Cross-References · XREF · QCM requis

Cross-Références (XREF) — Cartographier les Appels

Comprendre qui appelle quoi — la carte routière d'un binaire inconnu

🔗 Comment utiliser les XREF dans Ghidra
1
Clic droit sur une fonction → References → Show References to → liste tous les appelants
→ check_flag appelée depuis : main (1 référence)
2
Clic droit sur une donnée globale → References → Show References → accès en lecture/écriture
→ encoded_flag lue depuis : check_flag (29 références)
3
Dans le Listing, les XREF s'affichent au-dessus de chaque label : ; XREF[1]: main:001010a2(c)
→ (c) = call · (r) = read · (w) = write · (d) = data
4
Window → Call Trees → graphe complet des appels entrants ET sortants d'une fonction
→ Indispensable pour les binaires complexes avec 100+ fonctions
🎯 Cas d'usage CTF
  • Chercher strcmp → voir quelles strings sont comparées
  • Chercher printf → trouver les messages d'erreur/succès
  • Fonctions sans XREF entrant = code mort ou entry points alternatifs
⚡ Raccourcis XREF
  • Ctrl+Shift+F — Find References (toutes)
  • Flèche haut sur un XREF label — naviguer vers l'appelant
  • Alt+← — revenir au point de départ après navigation

Structures de Données dans Ghidra

Reconstruire les structs, unions et tableaux pour rendre le code lisible

📦 Créer une Structure
  • Data Type Manager → clic droit → New → Structure
  • Ajouter les champs : nom + type + taille
  • Exemple : struct User { char name[32]; int id; int role; }
  • Appliquer la struct sur un pointeur dans le décompilateur → T → User *
  • Le code devient : user→name, user→role au lieu de *(ptr+0x20)
📊 Types courants à corriger
// Avant renommage
undefined8 param_1
undefined4 local_10
undefined1 *puVar1

// Après retypation
char *user_input
int flag_length
byte *encoded_ptr
🔍 Identifier les tableaux — Array Definition
; Dans le Listing, une donnée à l'adresse 0x00104020 :
00104020 23 31 26 33 20 26 21 3b 26 2e ; bytes bruts

; Clic droit → Data → Create Array → byte[10]
00104020 byte[10] encoded_flag ; bien plus lisible !

; Dans le décompilateur :
if (input[i] ^ 0x42) != encoded_flag[i])

Patcher un Binaire dans Ghidra

Modifier les instructions d'un binaire pour changer son comportement — technique CTF fondamentale

⚡ Workflow de patching — bypass d'une condition
1
Identifier la condition à bypasser — ex: JNE LAB_fail après la comparaison du flag
→ Si wrong = JNE saute vers fail · On veut forcer le saut vers done
2
Clic droit sur l'instruction JNE → Patch Instruction → remplacer JNE par JE ou NOP
→ NOP (0x90) = No Operation = l'instruction est ignorée
3
File → Export Program → Format : Original File → sauvegarder le binaire patché
→ Le binaire exporté aura tes modifications appliquées
4
chmod +x binary_patched && ./binary_patched — tester le binaire modifié
→ "Congratulations! Flag: SOOPHA{...}" — patching réussi ✅
🔄 Patches courants CTF
  • JNE → JE — inverser une condition
  • JNE → NOP NOP — supprimer le saut (2 NOP car JNE = 2 bytes)
  • JMP SHORT — forcer un saut inconditionnel
  • MOV EAX,0 → MOV EAX,1 — forcer un return true
⚠️ Limites du patching
  • Checksums d'intégrité — le binaire vérifie son propre hash
  • Signatures cryptographiques — le binaire est signé
  • Anti-tampering — détection de modification en mémoire
  • Dans un CTF → patching fonctionne 90% du temps sur les challenges basiques

Workflow Ghidra en 7 Étapes

Une méthode simple pour éviter de te noyer dans 500 fonctions anonymes

STEP 01
Importer
Créer un projet propre, lancer l'auto-analysis, vérifier l'architecture et les options détectées.
STEP 02
Repérer main
Trouver l'entrée logique : main, init, start, ou une fonction orchestratrice équivalente.
STEP 03
Scanner les strings
Chercher les messages d'erreur, prompts, URLs, noms de fichiers, mutex, indices fonctionnels.
STEP 04
Suivre les XREF
Remonter des indices vers les fonctions qui les utilisent réellement.
STEP 05
Renommer
Nommer fonctions, variables, buffers, structures. Le renommage est une partie de l'analyse.
STEP 06
Comparer asm/C
Valider le pseudo-code avec la vue assembleur quand un détail paraît suspect.
STEP 07
Préparer GDB
Noter les adresses, offsets et hypothèses à confirmer dynamiquement.
Ghidra n'est pas juste un lecteur de pseudo-code : c'est un outil de cartographie. Ton job est de réduire l'inconnu.

// Slide 13 · Quand le décompilateur ment · Validation

Quand le Décompilateur Te Trompe

Le pseudo-code est pratique, mais il ne remplace jamais la vue assembleur

🚨 Signaux de pseudo-code douteux
  • Variables au type absurde ou tailles incohérentes
  • Tableaux transformés en pointeurs incompréhensibles
  • Conditions simplifiées à l'excès
  • Fonctions inconnues mal interprétées par les signatures
✅ Réaction professionnelle
  • Revenir au listing assembleur
  • Vérifier les appels, registres et tailles mémoire
  • Corriger les types manuellement
  • Documenter ce qui reste incertain avant de passer en dynamique
# Règle d'or
pseudo-code = accélérateur de lecture
assembleur = source de vérité
GDB = arbitre final à runtime
