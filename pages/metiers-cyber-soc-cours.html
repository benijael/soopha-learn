SIEM — Security Information & Event Management

Le cerveau du SOC. Collecte, normalise, corrèle et analyse les logs de tous les équipements (AD, firewall, EDR, proxy, VPN…). Génère des alertes quand un comportement anormal est détecté.

SplunkMicrosoft SentinelElastic SIEMIBM QRadar
SOAR — Security Orchestration, Automation & Response

L'automatisation de la réponse. Quand le SIEM détecte une alerte, le SOAR exécute automatiquement un playbook : bloquer une IP, isoler un poste, notifier le manager, ouvrir un ticket. Multiplie par 10 la capacité de réponse d'une équipe.

Palo Alto XSOARSplunk SOARTheHive + Cortex
EDR / XDR — Endpoint & Extended Detection

EDR : agent sur chaque poste/serveur qui capture comportements, processus, connexions réseau, fichiers. Détecte le malware sans signature. XDR étend la visibilité au réseau, cloud, identités — vision unifiée de toute l'infrastructure.

CrowdStrikeSentinelOneMicrosoft Defender XDR
Flux complet d'une alerte SOC
Événement → Log envoyé au SIEM
SIEM → Corrélation → Alerte générée
N1 → Triage : Faux positif ou réel ?
N2 → Investigation + qualification
SOAR → Playbook auto (blocage, isolation)
N3/RSSI → Escalade si incident confirmé

IOC vs IOA — Deux types d'indicateurs
IOC — Indicator of Compromise

Preuve après coup qu'une compromission a eu lieu. Hash de fichier malveillant, IP C2, domaine malveillant, clé de registre. Réactif.

IOA — Indicator of Attack

Comportement en cours d'attaque : PowerShell encodé, création de service inhabituel, LSASS access. Proactif, détecte avant l'IOC.

// IOC = chercher la trace du crime
// IOA = attraper le criminel en flagrant délit
Pyramide de la douleur · David Bianco

Plus on bloque haut dans la pyramide, plus on coûte cher à l'attaquant :

TTPs ← Douleur max. Oblige à rechanger de méthode
Outils ← Doit recompiler, changer d'outil
Réseau/Host ← Changer d'infra, coûteux
Domaines ← Changer de domaine, facile
Hash / IP ← Trivial à changer, douleur min.
Processus de triage — OODA Loop adaptée
Observe Lire l'alerte SIEM sans préjugé
Orient Contextualiser (qui, quoi, quand, où)
Decide Faux positif / Vrai incident / Escalade
Act Containment → Eradication → Recovery

// 04 · Cas Concret
🔴 CAS CONCRET #1

Connexion suspecte compte admin à 3h du matin

Alerte SIEM · Impossible Travel · Credential Compromise
L'alerte qui arrive
[03:17:42] ALERT HIGH — Impossible Travel
User : admin.sys@corp.net
Login 1 : 22:45 UTC — Paris, FR (VPN corp)
Login 2 : 03:17 UTC — Lagos, NG (IP résidentielle)
Delta : 4h32 — impossible physiquement
Cible : Accès réussi · portail Azure AD
MFA : Contourné via Adversary-in-the-Middle
L'attaquant a utilisé un kit de phishing Evilginx2 qui proxifie la vraie page de login Microsoft et vole le cookie de session post-MFA. Le MFA ne protège plus.
Investigation N2 — Questions à résoudre
  • Le Login 1 était-il légitime ? → Vérifier les logs AD de la soirée, valider avec l'utilisateur
  • Que s'est-il passé entre 22h45 et 03h17 ? → Analyser les actions sur Azure AD pendant cette fenêtre
  • Quelles ressources ont été accédées ? → Logs Azure Audit, Graph API, SharePoint, Exchange
  • Y a-t-il eu exfiltration ? → DLP logs, volume de téléchargements, emails sortants
  • Y a-t-il persistence ? → Nouveaux comptes créés, règles de transport Exchange, MFA modifiée
Réponse SOC — Procédure d'urgence
01
Containment immédiat — Révoquer toutes les sessions actives du compte (Azure AD : Revoke Sign-In Sessions). Réinitialiser le mot de passe. Désactiver le compte temporairement.
02
Notification — Alerter le RSSI et le manager de l'utilisateur. Ouvrir un ticket P1 dans TheHive. Déclencher le playbook SOAR "Compte compromis admin".
03
Investigation Forensique — Extraire les logs Sign-In Azure (30j), analyser les actions via Microsoft Graph API, chercher règles Exchange créées, applications OAuth autorisées.
04
Scope de la compromission — L'attaquant avait-il accès à d'autres comptes ? Mouvement latéral vers d'autres services ? Données exfiltrées ?
05
Remédiation — Réinitialisation du compte, renforcement MFA (FIDO2/clé physique), déploiement Conditional Access bloquant les pays non autorisés.
Requête KQL — Microsoft Sentinel
// Détecter les Impossible Travels sur comptes admin
SigninLogs
| where UserPrincipalName contains "admin"
| where TimeGenerated > ago(24h)
| summarize Locations=make_set(Location),
  IPs=make_set(IPAddress) by UserPrincipalName
| where array_length(Locations) > 1
| extend Risk = "IMPOSSIBLE_TRAVEL"
| order by UserPrincipalName asc

Ransomware détecté en cours de propagation

Incident P0 · Réponse temps réel · Toutes mains sur le pont
Timeline de l'attaque reconstruite
J-21
Initial access — Phishing ciblé RH. Pièce jointe Excel macro. Exécution Qakbot loader.
J-18
Persistence — Création tâche planifiée, clé Run registre. Beacon Cobalt Strike installé.
J-7
Discovery & Lateral movement — BloodHound AD recon, Pass-the-Hash, accès DC.
J-3
Exfiltration — 2TB de données copiées vers Mega.nz via rclone en HTTPS.
J0 03h
Chiffrement — LockBit 3.0 déployé via GPO. Fichiers chiffrés sur 40 serveurs en 47 minutes.
J0 07h
Détection SOC — Alertes EDR massives. L'équipe de nuit est déjà sur le bridge.
Pourquoi le SOC n'a pas détecté plus tôt ?
  • L'attaquant a "lived off the land" : utilisation de outils Windows légitimes (PsExec, WMI, PowerShell) pour éviter les signatures
  • Pas de règles Sigma sur Cobalt Strike en environnement sans EDR à l'époque
  • Exfiltration HTTPS vers Mega.nz non bloqué par le proxy — trafic chiffré non inspecté
  • Pas de monitoring C2 beacon : connexions régulières toutes les 60s vers IP externe non repérées
Plan de réponse à l'incident (IRP)
P0
Isolation réseau immédiate — Firewall : couper toutes les connexions sortantes des segments impactés. Ne pas éteindre les machines (préserver la RAM pour forensics).
P1
War room activé — RSSI, DSI, DG, CERT, juriste, communication. Bridge permanent 24/7. Ouverture d'un canal Slack dédié.
P2
Périmètre de contamination — Lister les machines chiffrées vs saines. Identifier le patient zéro. Cartographier les mouvements latéraux via les logs AD et l'EDR.
P3
Forensics — Dump RAM des machines clés avant isolement. Image disque des serveurs critiques. Analyse avec Volatility + Autopsy.
P4
Restoration — Reconstruction depuis les derniers backups sains (hors ligne). Patch de la CVE exploitée. Rebuild AD si compromis.
P5
Notification légale — RGPD : notification CNIL sous 72h si données personnelles exfiltrées. Dépôt de plainte.
Leçons apprises — améliorations SOC
  • Déployer EDR sur 100% des assets (endpoint + serveurs)
  • Ajouter règle SIEM : beacon régulier vers IP externe inconnue
  • Inspection SSL/TLS sur le proxy pour Mega.nz / DropBox
  • Sauvegardes immuables hors ligne testées chaque trimestre
// 06 · Cas Concret
🔴 CAS CONCRET #3

BEC — Business Email Compromise

La fraude la plus coûteuse · 2,9 Mds$ de pertes mondiales en 2023 · FBI IC3
Le scénario
La comptable reçoit un mail du "CEO" depuis ceo.nom@corp-entreprise.com
(domaine typosquat enregistré 48h plus tôt).
Objet : "Virement urgent confidentiel — ne pas en parler"
Elle effectue un virement de 187 000€ vers un compte lituanien.
La banque alerte 3h plus tard. Argent introuvable.
  • Domaine usurpé non détecté (pas de surveillance DMARC)
  • Aucun processus de validation des virements >50K€
  • Absence de sensibilisation des équipes finance
  • Pas de règle SIEM sur les emails de domaines similaires
Investigation SOC — Comment détecter le BEC
01
Analyse de l'en-tête email — Vérifier SPF, DKIM, DMARC. Comparer le domaine d'envoi vs le domaine de l'entreprise. Détecter les typosquats.
02
Recherche du typosquat — corp-entreprise.com enregistré 2j avant. Whois : Registrar Namecheap + Privacy guard + IP hébergement Russie.
03
Vérifier si d'autres employés visés — Chercher dans les logs Exchange/M365 tous les emails reçus depuis ce domaine ces 30 derniers jours.
04
Vérifier la compromission réelle du CEO — Est-ce que le vrai compte CEO est compromis ou juste usurpé ? Vérifier ses logs Azure AD, Exchange, sessions actives.
Règle SIEM — Détection typosquatting
// Splunk SPL — Emails domaines similaires
index=email sourcetype=exchange_mail
| eval sender_domain=split(sender,"@")[-1]
| where sender_domain != "corp.net"
  AND (like(sender_domain,"%corp%")
  OR like(sender_domain,"%c0rp%")
  OR like(sender_domain,"%corp-%"))
| stats count by sender, sender_domain, recipient
| where count > 0
| sort -count
Remédiation et prévention
  • DMARC p=reject sur tous les domaines de l'entreprise — bloque l'usurpation
  • DMARC monitoring — Alertes si un domaine similaire envoie en votre nom
  • Bannière mail externe — Tag visible sur tout email provenant de l'extérieur
  • Process financier — Double validation téléphonique obligatoire pour tout virement >10K€
  • Sensibilisation — Simulation de phishing mensuelle pour les équipes finance/RH
// DMARC record — Protection anti-usurpation
v=DMARC1; p=reject; rua=mailto:dmarc@corp.net
// Leçon clé :
Le BEC est un problème humain autant que technique.
La meilleure règle SIEM ne remplace pas
un processus de validation des virements.

| where EventID == 4688
| where CommandLine contains "-enc"
or CommandLine contains "-EncodedCommand"

CrowdStrike Falcon
Expert

EDR/XDR cloud-native de référence. Détection comportementale (IA), threat graph, investigation rapide. Utilisé par les MDR et grandes entreprises. Standard pour les incidents majeurs.

  • Détection comportementale excellente, peu de faux positifs
  • Threat Graph : visualisation des chaînes d'attaque
  • Coûteux, formation longue pour le plein potentiel
TheHive + Cortex
Accessible

Plateforme open source de gestion d'incidents + orchestration. TheHive = cas d'incidents, timeline, tâches. Cortex = analyse automatisée des observables (IP, hash, URL) via 100+ analyzers.

  • 100% open source et gratuit — parfait pour débuter
  • Intégration MISP, Sigma, VirusTotal native
  • Nécessite une infra à maintenir
MISP — Threat Intelligence
Open Source

Plateforme de partage d'IOC. Reçoit des feeds de menaces (ANSSI, CIRCL, communautés) et les injecte dans le SIEM pour enrichir les alertes. Standard des SOC gouvernementaux africains et européens.

// 08 · Threat Hunting

Threat Hunting & Règles de détection

Le Threat Hunting — définition
Le Threat Hunting ne attend pas les alertes.
Il part d'une hypothèse (un attaquant utilisant T1055 est présent)
et cherche activement les preuves dans les données — avant tout déclenchement d'alerte.
01
Hypothèse — "Un groupe APT utilisant BloodHound pour AD recon est peut-être dans notre réseau"
02
Recherche data — Requêtes SIEM ciblées sur les comportements connus de BloodHound (LDAP queries massives, enumération SPN)
03
Analyse — Identifier les anomalies, corrélation avec d'autres sources (EDR, réseau)
04
Output — Soit un incident confirmé, soit une nouvelle règle de détection pour le SIEM
Format Sigma — Règle de détection universelle

Sigma = langage YAML universel pour écrire des règles de détection convertibles vers tout SIEM (Splunk, Sentinel, QRadar, Elastic). Le standard de facto de la communauté.

title: LSASS Memory Access via ProcDump
status: stable
description: Détecte l'accès à LSASS pour dump credentials
tags: [attack.credential_access, T1003.001]
logsource:
  category: process_access
detection:
  selection:
    TargetImage|endswith: '\lsass.exe'
    GrantedAccess|contains: '0x1010'
  condition: selection
falsepositives: [AV scanners, some EDR agents]
level: high
Format YARA — Détection de malware

YARA = règles de description de patterns pour identifier des fichiers malveillants. Basé sur des chaînes, des patterns binaires ou des conditions logiques. Intégré dans la plupart des EDR et sandbox.

rule Detect_Mimikatz_Strings
{
  meta:
    description = "Détecte les strings Mimikatz"
    author = "SOC SooPHA"
  strings:
    $s1 = "sekurlsa::logonpasswords"
    $s2 = "privilege::debug"
    $s3 = "lsadump::dcsync"
  condition:
    2 of ($s*)
}

Ce qu'il faut retenir

✅ LES 5 RÉFLEXES DE L'ANALYSTE SOC
  • Qualifier avant d'agir — Faux positif ou vrai incident ? Ne jamais paniquer sur une alerte brute
  • Contextualiser — Qui ? Sur quelle machine ? À quelle heure ? Depuis où ? C'est normal pour ce user ?
  • Documenter tout — Chaque action, chaque décision, chaque log analysé trace l'investigation
  • Contenir d'abord — Isoler avant d'analyser. Stopper l'hémorragie, puis comprendre
  • Tirer les leçons — Chaque incident produit une nouvelle règle de détection ou un playbook amélioré
⚠ LES 3 ERREURS LES PLUS FRÉQUENTES
  • Alert fatigue — Trop d'alertes → l'analyste se désensibilise → un vrai incident passe inaperçu
  • Éteindre la machine infectée — On perd la RAM, les processus actifs, les connexions C2. Toujours isoler sans éteindre.
  • Pas documenter en temps réel — La mémoire est faillible sous pression. Le journal d'incident en temps réel est indispensable.
