// 02 · Fondamentaux

Les 4 piliers de la Cloud Security

🚫 Zero Trust Architecture
"Never trust, always verify."
Aucun utilisateur, aucun service, aucun réseau n'est implicitement de confiance — même en interne.
  • Vérifier l'identité à chaque requête — MFA fort, certificates, SPIFFE/SPIRE
  • Valider l'état du device — posture check avant tout accès (MDM, compliance)
  • Least Privilege — accès juste-à-temps (JIT), juste assez (JAE)
  • Micro-segmentation — isolation des workloads, East-West traffic control
  • Chiffrement de bout en bout — in transit (TLS 1.3) et at rest (AES-256)
🏗 Defense in Depth — Couches de sécurité
Périmètre WAF · DDoS protection · CDN
Réseau VPC · Security Groups · NACL · Private Link
Compute EDR Cloud · Hardening OS · Patch mgmt
Application SAST · DAST · SCA · API Gateway
Données Chiffrement · DLP · Classification
Identités IAM · PAM · MFA · SSO · CIEM
Logs & SIEM CloudTrail · GuardDuty · Sentinel
🔑 IAM Cloud — Les pièges classiques
  • Root account non protégé — Le compte root AWS / super-admin Azure doit avoir un MFA hardware et ne jamais générer d'access keys
  • Access keys en dur dans le code — La fuite #1 sur GitHub. AWS Secret Scanner, GitGuardian détectent ça
  • Rôles IAM trop permissifs — AdministratorAccess pour un Lambda qui lit juste un fichier S3
  • Service accounts partagés — Pas de traçabilité des actions. Un service = un rôle IAM dédié
  • Solution — IAM Roles (pas d'Access Keys), Instance Profiles, IRSA (Kubernetes), Workload Identity
🚀 DevSecOps — Shift Left Security

Intégrer la sécurité dès le début du cycle de développement, pas en fin de livraison. Coût d'une faille détectée en prod = 30× celui détecté pendant le dev.

Plan → Threat modeling dès la conception
Code → SAST (Semgrep, SonarQube)
Build → SCA (Snyk), scan d'image Docker
Test → DAST (OWASP ZAP, Burp API scan)
Deploy → IaC scan (tfsec, Checkov)
Operate → CSPM + CWPP + runtime protection

// 04 · Cas Concret
🔴 CAS CONCRET #1

Bucket S3 public — données sensibles exposées

La misconfiguration cloud la plus fréquente · Des millions de records exposés chaque année
Le scénario
Un développeur crée un bucket S3 pour stocker les exports RH.
Il coche "Public Access" pour tester rapidement depuis son laptop.
Il oublie de le désactiver. Le bucket contient 45 000 fiches employés.
Un chercheur en sécurité le découvre via GrayhatWarfare 3 semaines plus tard.
L'entreprise l'apprend par un tweet public.
  • Cause : "Block Public Access" désactivé au niveau bucket ET au niveau account
  • Cause : Bucket policy trop permissive "Principal": "*"
  • Cause : Aucun CSPM en place pour détecter les ressources publiques
  • Cause : Données non chiffrées avec KMS (SSE-KMS désactivé)
  • Impact : RGPD — notification CNIL 72h · risque amende 4% CA · perte de confiance
Détection et remédiation
01
Détection CSPM — Prisma Cloud génère un finding "S3 Bucket is Publicly Accessible" — sévérité CRITICAL. En l'absence de CSPM, on le détecte grâce à AWS Security Hub + AWS Config rule.
02
Containment immédiat — Activer "Block All Public Access" en une commande CLI. Le bucket devient privé instantanément, même si la bucket policy reste permissive.
03
Analyse d'exposition — Vérifier les CloudTrail logs : qui a accédé au bucket ? Depuis quand ? Quels objets ? Volume de données potentiellement exfiltrées.
04
Chiffrement retroactif — Activer SSE-KMS sur tous les objets existants. Rotation de clé KMS. S3 Object Lock pour les données sensibles.
05
Prévention systémique — SCP (Service Control Policy) au niveau AWS Organizations bloquant tout bucket public dans tous les comptes de l'organisation.
AWS CLI — Remédiation en 30 secondes
# 1. Bloquer tout accès public
aws s3api put-public-access-block \
  --bucket mon-bucket-rh \
  --public-access-block-configuration \
  "BlockPublicAcls=true,IgnorePublicAcls=true,
  BlockPublicPolicy=true,RestrictPublicBuckets=true"
 
# 2. Activer le chiffrement SSE-KMS
aws s3api put-bucket-encryption \
  --bucket mon-bucket-rh \
  --server-side-encryption-configuration \
  '{"Rules":[{"ApplyServerSideEncryption
  ByDefault":{"SSEAlgorithm":"aws:kms"}}]}'
Terraform — Bucket S3 sécurisé by design
resource "aws_s3_bucket" "rh_data" {
  bucket = "corp-rh-exports-prod"
}
resource "aws_s3_bucket_public_access_block" "rh" {
  bucket = aws_s3_bucket.rh_data.id
  block_public_acls = true
  block_public_policy = true
  ignore_public_acls = true
  restrict_public_buckets = true
}
// Leçon clé :
Infrastructure as Code + CSPM = Sécurité proactive.
Détecter en 5 minutes ce qui prenait
3 semaines sans outillage.

🔴 CAS CONCRET #2

Escalade de privilèges AWS IAM

La technique préférée des attaquants en cloud · Souvent possible en 3 étapes
Le vecteur — Access Key exposée sur GitHub
# Dans un commit public GitHub — détecté par un bot
AWS_ACCESS_KEY_ID=AKIA4XXXXXXXXXXX
AWS_SECRET_ACCESS_KEY=abc123xxxxxxxxxxxxxxxxxxxxxxxxxxxx
# L'attaquant clone le repo 4 minutes après le push
# AWS détecte l'utilisation 6h plus tard (trop tard)
Chaîne d'escalade de privilèges
01
Énumération — iam:GetUser, iam:ListAttachedUserPolicies → l'attaquant voit que l'utilisateur a iam:AttachUserPolicy
02
Escalade — Il s'attache la policy AdministratorAccess à son propre utilisateur. En une commande, il devient admin.
03
Persistence — Crée un nouvel utilisateur IAM avec des credentials frais. Même si la clé initiale est révoquée, il conserve l'accès.
04
Action — Lance des centaines d'instances EC2 pour du crypto-mining. Facture cloud : 47 000€ en 72h.
Les permissions IAM dangereuses à surveiller
iam:AttachUserPolicy → Escalade directe
iam:CreateAccessKey → Persistence
iam:PassRole → Escalade via services
sts:AssumeRole → Pivot inter-comptes
lambda:CreateFunction → Code exec avec rôle
iam:CreatePolicyVersion → Modifier une policy
ec2:RunInstances → Crypto-mining
Détection & Réponse
01
AWS GuardDuty — Finding "UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration" + "Policy:IAMUser/RootCredentialUsage". Alerte en quasi-temps réel.
02
CloudTrail forensics — Requêter tous les événements IAM des dernières 24h depuis l'access key compromise : quel API calls, depuis quelle IP, à quelle heure.
03
Containment — Désactiver l'access key, attacher une Deny policy explicite à l'utilisateur, révoquer toutes les sessions STS actives (IAM : DeleteServiceSpecificCredential).
04
Scope — L'attaquant a-t-il créé de nouveaux utilisateurs/rôles ? Accédé à d'autres services (S3, RDS, Secrets Manager) ? Exfiltré des données ?
05
Prévention — AWS Organizations SCP bloquant iam:CreateAccessKey pour tous les comptes. GitHub secret scanning activé. Rotation automatique des secrets via AWS Secrets Manager.
AWS Config — Règle de détection des keys non rotées
# Lister les access keys de plus de 90 jours
aws iam generate-credential-report
aws iam get-credential-report \
  --query 'Content' --output text | base64 -d \
  | awk -F',' '$9=="true" &&
    (systime()-mktime(...)>7776000)'
# AWS Config Rule native :
access-keys-rotated (maxAccessKeyAge: 90)

🔴 CAS CONCRET #3

Container Escape & Kubernetes Misconfiguration

L'attaque qui sort du container pour compromettre le nœud — et tout le cluster
Le scénario — Container mal configuré
Un pod Kubernetes est déployé avec privileged: true et un montage du socket Docker.
L'attaquant exploite une vulnérabilité dans l'app web.
Depuis le container, il monte le filesystem du nœud hôte,
lit les credentials de tous les autres pods du cluster,
puis rebondit vers AWS via l'IMDS pour voler les credentials EC2.
Les misconfigurations Kubernetes les plus dangereuses
privileged: true → Accès kernel hôte
hostPID: true → Voir tous les process hôte
hostNetwork: true → Réseau hôte, bypass Network Policy
volumeMounts: /var/run/docker.sock→ Docker escape
automountServiceAccount: true→ Token RBAC exposé
RBAC: ClusterAdmin all SA → Compromission totale
L'attaque IMDS — Instance Metadata Service
01
Depuis le container — Appel IMDS v1 : curl http://169.254.169.254/latest/meta-data/iam/security-credentials/node-role
02
Récupération credentials — AccessKeyId + SecretAccessKey + Token temporaire du rôle IAM de l'instance EC2.
03
Pivot cloud — L'attaquant utilise ces credentials depuis n'importe où. Il accède à tous les services AWS accessibles au nœud EC2.
Pod Security — Remédiation
securityContext:
  runAsNonRoot: true # jamais root
  runAsUser: 10001 # UID non-root
  readOnlyRootFilesystem: true # FS lecture seule
  allowPrivilegeEscalation: false
  capabilities:
    drop: ["ALL"] # aucune capability Linux
automountServiceAccountToken: false # pas de token RBAC
01
Admission Controllers — OPA/Gatekeeper ou Kyverno pour bloquer les pods privileged avant déploiement.
02
IMDSv2 obligatoire — Forcer IMDSv2 (requiert un token HTTP) sur toutes les instances EC2. Bloque les appels IMDS depuis les containers sans configuration explicite.
03
Network Policies — Bloquer le trafic sortant vers 169.254.169.254 depuis les namespaces applicatifs.
04
IRSA / Workload Identity — Donner des credentials IAM directement aux pods via OIDC — sans passer par les credentials du nœud EC2.
Outils Kubernetes Security
kube-bench · CIS checks Falco · Runtime detection Trivy · Image scanning Kyverno · Policy engine OPA/Gatekeeper KubeAudit Checkov · IaC scan

DevSecOps — Intégrer la sécurité dans le pipeline CI/CD

Pipeline CI/CD sécurisé — Vue d'ensemble
git push → GitHub Actions déclenché
├── SAST Semgrep · SonarQube (vulnérabilités code)
├── Secrets GitGuardian · Gitleaks (credentials exposés)
├── SCA Snyk · Dependabot (CVE dépendances)
├── IaC Scan Checkov · tfsec (Terraform misconfiguration)
├── Image Trivy · Grype (vulnérabilités Docker)
├── DAST OWASP ZAP (injection, XSS, auth)
└── Deploy → Seulement si ZERO finding CRIT/HIGH
Le vrai problème : les secrets dans le code
GitHub a détecté 12,8 millions de secrets dans les dépôts publics en 2023.
Un secret pushé sur GitHub est scrapé en moins de 60 secondes
par des bots automatisés.
# JAMAIS dans le code
DB_PASSWORD = "Azerty123!"
AWS_SECRET = "abc123xxxxx"
# TOUJOURS via des variables d'env sécurisées
DB_PASSWORD = os.environ["DB_PASSWORD"]
# Ou via Vault / AWS Secrets Manager
GitHub Actions — Security Gate complet
name: Security Pipeline
on: [push, pull_request]
jobs:
  security:
    runs-on: ubuntu-latest
    steps:
    - name: Secret scanning
      uses: gitleaks/gitleaks-action@v2
    - name: SAST — Semgrep
      uses: semgrep/semgrep-action@v1
    - name: IaC scan — Checkov
      uses: bridgecrewio/checkov-action@master
    - name: Docker image — Trivy
      uses: aquasecurity/trivy-action@master
      with: {severity: 'CRITICAL,HIGH'}

// Récap · Cloud Security

Ce qu'il faut retenir

✅ LES 5 COMMANDEMENTS DU CLOUD SECURITY
  • Responsabilité partagée — Le provider sécurise l'infra, toi tu sécurises tout le reste
  • Least Privilege IAM — Aucun rôle, aucune politique avec * en Action ou Resource sans justification
  • Tout en IaC — Aucune ressource créée manuellement en production. Terraform + Git = traçabilité totale
  • Secrets dans un vault — Zéro secret statique dans le code, les variables d'env des CI/CD, les images Docker
  • CSPM activé 24/7 — Tu ne peux pas sécuriser ce que tu ne vois pas. Scanning continu, pas trimestriel
⚠ LES 3 ERREURS QUI COÛTENT UNE FORTUNE
  • Bucket S3 / Blob Storage public — Données sensibles exposées, RGPD, amende. La misconfiguration #1 depuis 10 ans.
  • Access Keys IAM longue durée — Sur GitHub = compromission en 60 secondes. Utiliser des rôles IAM, jamais des keys.
  • Containers privileged en prod — Container escape → accès hôte → cluster compromis. Toujours enforcer une Pod Security policy.
