Techniques Anti-Debug — Arsenal Complet

Les méthodes utilisées par les malwares pour détecter et tuer les debuggers

ptrace(PTRACE_TRACEME)
Linux. Un processus ne peut être tracé que par un seul parent. Si ptrace(PTRACE_TRACEME) échoue → GDB déjà attaché → quitter. La technique la plus répandue sur Linux.
if (ptrace(PTRACE_TRACEME,0,0,0) == -1) { exit(1); }
Timing checks
Mesurer le temps entre deux instructions. Sous debugger, le stepping ralentit drastiquement l'exécution. Si delta > seuil → debugger détecté → comportement altéré.
clock_gettime avant → stepping GDB → clock_gettime après
delta > 1000ms → debug détecté
IsDebuggerPresent (Windows)
API Win32 qui lit le flag BeingDebugged dans le PEB (Process Environment Block). Simple et très courant dans les malwares Windows.
if (IsDebuggerPresent()) { FakeDecoy(); }
// PEB+2 = BeingDebugged byte
INT3 / INT 0x2D scan
Scanner sa propre mémoire à la recherche des breakpoints GDB (opcode 0xCC = INT3). Si trouvé → debugger présent → exit ou comportement alternatif.
if (*(byte*)func == 0xCC) exit(1);
// Scan du code en mémoire
VM / Sandbox detection
Détecter l'environnement d'analyse : présence VMware/VBox (CPUID, fichiers /sys/), hostname par défaut, peu de RAM, pas de vraie activité utilisateur.
dmidecode | grep -i "virtual\|vmware\|vbox"
cat /sys/class/dmi/id/product_name
Fork + ptrace parent
Fork() → le child trace le parent avec ptrace → si un autre debugger essaie de s'attacher, il échoue. Technique "self-debugging" pour occuper le slot ptrace.
pid = fork();
if (child) ptrace(ATTACH, parent, ...);
"Face à l'anti-debug, la règle d'or : identifier la technique dans Ghidra AVANT de lancer GDB. Patcher la détection statiquement est toujours plus propre que de la contourner dynamiquement."
// Slide 03 · Bypass anti-debug · Patcher · GDB catch syscall

Contourner l'Anti-Debug — Techniques de Bypass

Pour chaque technique de détection, une technique de contournement

🔓 Bypasses par technique
1
ptrace(PTRACE_TRACEME) → Patcher le résultat
Ghidra : trouver le CALL ptrace → NOP les instructions de test du retour (TEST EAX, EAX / JS exit) → le binaire ne peut plus détecter le debugger
Ou : GDB → catch syscall ptrace → si appelé, set $rax = 0 (succès forcé)
2
Timing checks → Patcher le seuil ou le branchement
Ghidra : trouver la comparaison CMP delta, threshold / JG exit → changer le seuil (très grande valeur) ou NOP le JG
Ou : GDB → modifier le résultat de clock_gettime pour simuler un delta court
3
IsDebuggerPresent (Win) → Patch PEB ou valeur de retour
x64dbg : modifier manuellement le byte PEB+2 (BeingDebugged) → 0x00 après chargement
Ou : patcher la valeur de retour de IsDebuggerPresent → MOV EAX, 0 / RET
4
VM detection → Masquer l'environnement
Modifier les valeurs DMI/CPUID (VMware Tools), changer le hostname, simuler de l'activité utilisateur, augmenter la RAM de la VM
Ou : utiliser une bare-metal machine pour l'analyse dynamique

Analyse Statique de Malware

Extraire les IOCs et comprendre le comportement sans exécuter le danger

🔍 Triage statique — workflow
1
Hash — md5sum, sha256sum → chercher sur VirusTotal → connu ou inconnu ?
2
file + strings — architecture, strings visibles, URLs, IPs, registry keys
3
Entropie — sections packées ? Sections suspects (UPX0, .ndata) ?
4
Imports (readelf -d ou dumpbin) — quelles fonctions système → comportement déduit
5
Ghidra — analyser la logique de main(), persistence, C2 communication
🎯 IOCs à extraire
  • Réseau — URLs, IPs, domaines C2, ports, user-agents custom
  • Fichiers — chemins créés/modifiés/supprimés, noms de fichiers déposés
  • Registre (Win) — clés de run, services créés, clés de config
  • Processus — processus lancés, injection de code, noms de mutex
  • Hashes — MD5/SHA256 du malware et des fichiers qu'il dépose
  • IOCs → alimenter SIEM, EDR, firewall, threat intelligence
# Triage rapide d'un binaire suspect
$ sha256sum malware.bin | tee hash.txt
$ file malware.bin
$ strings malware.bin | grep -E 'http|\.exe|HKEY|cmd\.exe|powershell' | sort -u
$ readelf -d malware.elf | grep NEEDED  # libs → comportement réseau/crypto ?
$ curl -s "https://www.virustotal.com/api/v3/files/$(cat hash.txt | cut -d' ' -f1)" -H "x-apikey: YOUR_KEY"
// Slide 05 · YARA · Règles · Détection de malwares

YARA — Écrire des Règles de Détection

Le langage de signature de malwares — créer des règles qui détectent des familles entières

📝 Anatomie d'une règle YARA
rule SooPHA_XOR_Backdoor {
  meta:
    author = "Béni-Jaël MBATA"
    description = "Détecte le backdoor XOR 0x42 SooPHA CTF"
    date = "2025-01"
    severity = "HIGH"

  strings:
    $xor_key = { 80 70 42 }                  // XOR byte [rax], 0x42
    $c2_encoded = { 11 0D 0D 12 0A 03 }         // "SOOPHA" XOR 0x42
    $wrong_msg = "Wrong flag, try again" nocase
    $flag_fmt = /SOOPHA\{[a-z0-9_]+\}/            // regex flag

  condition:
    uint32(0) == 0x464C457F                  // magic ELF
    and ($xor_key or $c2_encoded)
    and $wrong_msg
}

# Utilisation
$ yara SooPHA_XOR_Backdoor.yar ./samples/
SooPHA_XOR_Backdoor ./samples/crackme_05

Sandbox — Analyse Dynamique Sans Risque

Observer le comportement d'un malware dans un environnement contrôlé et isolé

🌐 Any.run
Sandbox interactive en ligne. Voir le malware s'exécuter en temps réel. Rapport automatique : processus, réseau, fichiers, IOCs. Plan gratuit disponible.
app.any.run → Upload → Analyser → Rapport JSON
🔬 VirusTotal
70+ moteurs AV. Analyse comportementale automatique. Relations avec d'autres samples. Idéal pour les triage rapide et la chasse aux variants.
virustotal.com → Hash lookup → Behavior tab
🦆 Cuckoo Sandbox
Sandbox open source auto-hébergée. Rapport complet : syscalls, réseau PCAP, dump mémoire, screenshots. Idéal pour les équipes SOC avec données sensibles.
pip install cuckoo → cuckoo submit malware.bin
🖥️ REMnux VM
Distribution Linux spécialisée analyse malware. 100+ outils préconfigurés : Ghidra, radare2, YARA, Volatility, Wireshark en mode forensics.
remnux.org → VM OVA → VMware/VBox
🎯 Sandbox vs Analyse Manuelle — quand choisir quoi ?
SANDBOX
Triage rapide
Malware inconnu, besoin de IOCs rapides, pas de temps pour l'analyse manuelle. 10 minutes → rapport complet. Risque : sandbox-aware malware qui change de comportement.
MANUELLE
Investigation deep
Malware APT avancé, anti-sandbox, comportement conditionnel. Attribution, comprendre l'algorithme complet, reverse du protocole C2. Peut prendre des heures/jours.
COMBINÉE
Best practice
Sandbox d'abord pour les IOCs rapides et la classification. Analyse manuelle pour les samples critiques ou les nouvelles familles. Approche professionnelle SOC/CERT.
// Slide 07 · Persistance · C2 · Architecture malware

Persistance & Communication C2

Comprendre l'architecture des malwares modernes — les deux piliers de tout RAT/backdoor

🔒 Mécanismes de persistance Linux
  • Crontab — @reboot /tmp/.hidden_payload → exécuté au démarrage
  • ~/.bashrc / ~/.profile — injection de commande au login shell
  • Systemd service — créer un faux service légitime-sounding
  • LD_PRELOAD — hooker les fonctions système pour cacher le malware
  • Rootkit LKM — module kernel qui cache les processus/fichiers
  • Chercher dans strace : open("/etc/crontab"), write(".bashrc")
📡 Protocoles C2 courants
  • HTTP/HTTPS — requêtes GET/POST vers le C2. User-agent custom identifiable dans strace/Wireshark
  • DNS tunneling — données encodées dans les requêtes DNS TXT. Très discret, contourne les firewalls
  • IRC / Telegram — botnets anciens (IRC) ou modernes (Telegram API)
  • Tor / I2P — anonymisation du serveur C2 → plus difficile à bloquer
  • Identifier dans Ghidra : imports de socket(), connect(), send(), recv()
# Détecter la persistance avec strace
$ strace -e trace=file,process ./malware 2>&1 | grep -E "cron|bashrc|systemd|/tmp|\.hidden"
 openat(AT_FDCWD, "/tmp/.x", O_WRONLY|O_CREAT) = 4
 openat(AT_FDCWD, "/home/user/.bashrc", O_RDWR) = 5
 → Persistance via .bashrc détectée !

MITRE ATT&CK — Classifier les Comportements

Le framework de référence pour documenter et communiquer sur les techniques des attaquants

📚 Qu'est-ce que MITRE ATT&CK ?
  • Framework ouvert de Tactics, Techniques & Procedures (TTPs) des attaquants réels
  • 14 Tactiques : Initial Access → Execution → Persistence → Privilege Escalation → Defense Evasion → Credential Access → Discovery → Lateral Movement → Collection → C2 → Exfiltration → Impact
  • 500+ Techniques chacune avec sous-techniques, exemples réels, groupes APT
  • Chaque comportement observé peut être mappé à un ID : T1055 = Process Injection
  • Référence : attack.mitre.org — entièrement gratuit
🗺️ Mapper notre malware CTF
T1027
Obfuscation
XOR encoding du flag et des strings
T1055
Defense Evasion
ptrace anti-debug pour détecter l'analyse
T1053
Persistence
Crontab entry créée pour survive reboot
T1071
C2
HTTP beacon vers domaine encodé
"Mapper tes trouvailles sur MITRE ATT&CK transforme une analyse technique en intelligence opérationnelle. C'est ce qui distingue un rapport de reverse engineering d'un rapport de Threat Intelligence."

Rédiger un Rapport d'Analyse Malware Professionnel

La compétence qui distingue le reverser amateur du analyste professionnel

📄 Structure du rapport pro
1
Executive Summary — 5 lignes max. Ce que fait le malware, le niveau de risque, les actions immédiates requises. Pour le RSSI/management.
2
Metadata — Hash SHA256/MD5, type de fichier, date de découverte, source du sample
3
Analyse statique — Strings, imports, sections, entropie, famille détectée
4
Analyse dynamique — Comportements observés : réseau, fichiers, processus, registry
5
IOCs — Tableau : Type | Valeur | Description | Confiance
6
MITRE ATT&CK — Tableau des TTPs mappés avec IDs et descriptions
7
Recommandations — Règles YARA, règles SIEM/EDR, actions de remédiation
✅ Bonnes pratiques rédactionnelles
  • Séparer faits et interprétations — "Le binaire appelle ptrace()" vs "Il détecte probablement GDB"
  • Quantifier la confiance — HIGH/MEDIUM/LOW pour chaque IOC et conclusion
  • Screenshots annotés — Ghidra, GDB, trafic réseau, comportements observés
  • IOCs actionnables — Format STIX/TAXII pour l'intégration SIEM
  • Reproductibilité — Liste des outils, versions, commandes exactes utilisées
  • Modèles gratuits : MISP, OpenCTI, Cortex Analyzer pour automatiser

Arsenal Complet — Outils d'Analyse Malware

Les outils de référence organisés par phase d'analyse

🏷️ YARA
Création et matching de règles de détection. Standard industriel pour la signature de malwares. Intégré dans tous les SIEM/EDR.
yara -r rules/ samples/
yara rule.yar malware.bin
🧠 Volatility 3
Forensics mémoire RAM. Extraire processus, connections réseau, clés de registre, injections mémoire depuis un dump RAM.
vol -f mem.dmp windows.pslist
vol -f mem.dmp linux.bash
🦠 Capa
Outil FLARE de FireEye. Identifie automatiquement les capacités d'un binaire (download files, create mutex, inject code) et les mappe sur MITRE ATT&CK.
capa malware.bin
→ rapport MITRE automatique
📡 Wireshark
Capture et analyse trafic réseau. Identifier les connexions C2, protocoles utilisés, données exfiltrées. Mode forensics : ouvrir des PCAPs de sandbox.
wireshark -r sandbox.pcap
Filtre : http.request.method == POST
🔍 PEStudio / PEiD
Analyse PE (Windows). Imports, strings, sections, certificats, anomalies. Détection de packers, version info, manifests. Indispensable pour les malwares Windows.
pestudio malware.exe
→ rapport complet en GUI
🌐 MISP
Plateforme de partage de Threat Intelligence. Stocker, partager et consommer des IOCs. Standard dans les SOC/CERT/CSIRT professionnels.
misp-project.org
→ Open source, auto-hébergeable
// Slide 12 · Éthique · Légalité · Responsabilité

Éthique & Responsabilité de l'Analyste Malware

Les règles non négociables — protéger, ne pas nuire, rester dans les clous

✅ Ce qui est toujours légal
  • CTF — tout est permis dans le périmètre du challenge
  • Malware reçu en incident — analyser un malware qui a infecté ton organisation est légal et nécessaire
  • Samples publics — MalwareBazaar, VirusTotal, TheZoo — samples partagés pour la recherche
  • Sandbox isolation totale — la VM n'a aucun accès réseau vers l'extérieur → aucun risque d'activation
  • Bug bounty — RE dans le périmètre du programme de bug bounty
🚫 Les lignes rouges
  • JAMAIS exécuter un malware sur une machine connectée à un réseau réel
  • JAMAIS partager des samples actifs sans chiffrement et mots de passe (standard : zip "infected")
  • JAMAIS contacter les serveurs C2 d'un malware — même pour "voir" ce qu'ils répondent
  • JAMAIS utiliser les techniques apprises pour attaquer des systèmes sans autorisation
  • Signaler toujours un malware actif aux autorités compétentes (ANSSI, CERT-FR, INTERPOL)
"Le reverse engineer défensif est l'égal de l'attaquant en compétences — mais son supérieur en éthique. La connaissance sans responsabilité n'est pas de la cybersécurité, c'est juste du hacking. — SooPHA Learn"

Le Livrable Final — Ce qu'un Analyste Rend Vraiment

Le reverse engineering n'est pas fini quand tu comprends le sample : il est fini quand tu peux l'expliquer

📝 Structure d'un vrai rapport
  • Contexte — origine du sample, périmètre, date, hash
  • Résumé exécutif — ce qu'il fait en 5 lignes compréhensibles
  • Analyse technique — anti-debug, persistance, réseau, obfuscation
  • IOCs — IPs, domaines, chemins, mutex, hashes, règles YARA
  • Recommandations — containment, détection, hunting, remédiation
🚀 Suite logique
  • Tu as maintenant les briques du parcours complet : triage, asm, Ghidra, GDB, obfuscation, anti-debug.
  • La meilleure suite n'est pas un autre cours théorique : c'est un cas de synthèse complet.
  • Le capstone RE·07 sert précisément à ça : enchaîner tout le workflow sur un même sample.
Comprendre un malware est une compétence. Produire un rapport clair, actionnable et défendable devant un client ou un SOC, c'est le niveau professionnel.
