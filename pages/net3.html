DNS — L'Annuaire d'Internet

Domain Name System — traduire soopha-learn.com en 172.66.x.x en moins de 50ms

📋 Types d'enregistrements DNS
A       domaine → adresse IPv4
AAAA   domaine → adresse IPv6
CNAME  alias → domaine canonique
MX     serveur de messagerie email
TXT    données texte (SPF, DKIM, vérif)
NS     serveurs de noms autoritaires
SOA    Start of Authority — infos de zone
PTR    IP → domaine (DNS inverse)
SRV    localiser un service (SIP, XMPP)
CAA    autorités habilitées à émettre des certs SSL

$ dig soopha-learn.com A
$ dig soopha-learn.com MX
$ dig -x 172.66.x.x  # PTR reverse
🔄 Résolution DNS récursive
1Ton PC→Résolveur FAI"C'est quoi l'IP de soopha-learn.com ?"
2Résolveur→Root Server (.)"Qui gère .com ?"
3Root Server→Résolveur"Les TLD .com servers : a.gtld-servers.net"
4Résolveur→TLD .com Server"Qui gère soopha-learn.com ?"
5TLD Server→Résolveur"NS: melany.ns.cloudflare.com"
6Résolveur→Cloudflare NS"IP de soopha-learn.com ?"
7Cloudflare→Résolveur → PC"172.66.x.x — TTL 300s"
// Slide 04 · Attaques DNS · Cache Poisoning · DNS Hijacking · DNSSEC

Attaques DNS — Quand l'Annuaire Ment

Le DNS est critique et souvent peu sécurisé — comprendre les attaques pour mieux défendre

DNS Cache Poisoning
Injecter de fausses réponses DNS dans le cache d'un résolveur. Tous les utilisateurs de ce résolveur sont redirigés vers un faux site. Attaque Kaminsky (2008) : exploit du port UDP prévisible.
✅ Fix : DNSSEC · randomisation du port source · DNS over HTTPS (DoH)
DNS Hijacking
Modifier les enregistrements DNS directement chez le registrar en compromettant le compte d'administration. Rediriger un domaine entier vers une infrastructure malveillante. Très utilisé par des APT.
✅ Fix : 2FA obligatoire sur le compte registrar · Registry Lock
DNS Amplification (DDoS)
Envoyer des requêtes DNS avec IP source forgée (IP de la victime) vers des serveurs DNS ouverts. Les réponses (100x plus grandes que les requêtes) inondent la victime. Facteur d'amplification × 50.
✅ Fix : Désactiver les open resolvers · BCP38 anti-spoofing
DNS Tunneling
Encapsuler des données non-DNS dans des requêtes DNS pour exfiltrer des données ou créer un canal C2. Fonctionne même derrière des firewalls qui bloquent tout sauf le port 53. Outil : iodine, dnscat2.
✅ Fix : DNS inspection · analyser la taille et fréquence des requêtes
Typosquatting
Enregistrer des domaines similaires à des sites légitimes (g00gle.com, soopha-leam.com). Les utilisateurs qui font une faute de frappe atterrissent sur un site malveillant — phishing, malware.
✅ Fix : Surveiller les domaines similaires · DMARC · formation utilisateurs
Zone Transfer (AXFR)
Si le serveur DNS est mal configuré, un attaquant peut demander un transfert de zone complet — obtenir tous les sous-domaines, IPs internes, architecture réseau en une seule requête.
✅ Fix : Autoriser AXFR uniquement aux serveurs NS secondaires légitimes

// Slide 06 · TLS · Handshake · Certificats · Chaîne de confiance · QCM requis

TLS — Comment HTTPS Chiffre Vraiment

Transport Layer Security — le protocole qui sécurise 95% des communications web mondiales

🤝 TLS Handshake — établir le tunnel chiffré
1
ClientHello — le navigateur envoie les versions TLS supportées, les cipher suites, un nombre aléatoire (client random)
2
ServerHello — le serveur choisit la version TLS + cipher suite + envoie son certificat X.509
3
Vérification du certificat — le navigateur vérifie la signature de l'AC (Autorité de Certification) dans sa liste de confiance
4
Échange de clé — Diffie-Hellman ou RSA → génération du session key partagée (sans jamais la transmettre)
5
Tunnel établi — toutes les communications chiffrées avec AES-256-GCM (symmetric encryption)
🏅 Headers de sécurité HTTP essentiels
# Forcer HTTPS pour 1 an
Strict-Transport-Security: max-age=31536000

# Protéger contre le clickjacking
X-Frame-Options: DENY

# Bloquer le MIME sniffing
X-Content-Type-Options: nosniff

# Contrôler les sources autorisées
Content-Security-Policy: default-src 'self'

# Limiter les infos de référent
Referrer-Policy: strict-origin

# Vérifier avec
$ curl -I https://soopha-learn.com

SMTP & Sécurité Email — Stopper le Spoofing

Simple Mail Transfer Protocol — comprendre comment les emails voyagent et comment les sécuriser

📧 Le voyage d'un email
Expéditeur → MUA (Outlook/Gmail)
  ↓ Submission port 587 (STARTTLS)
MTA expéditeur (Mail Transfer Agent)
  ↓ Lookup DNS MX du destinataire
  ↓ Connexion SMTP port 25
MTA destinataire
  ↓ Livraison dans la boîte
Destinataire → récupère via IMAP/993

# Tester manuellement SMTP
$ telnet mail.example.com 25
EHLO test.com
MAIL FROM: <test@test.com>
RCPT TO: <victim@example.com>
🛡️ SPF + DKIM + DMARC — la triade anti-spoofing
  • SPF (Sender Policy Framework) — enregistrement TXT DNS qui liste les serveurs autorisés à envoyer pour ton domaine. Un email d'une IP non listée = échec SPF
    v=spf1 include:_spf.google.com ~all
  • DKIM (DomainKeys Identified Mail) — signature cryptographique ajoutée à chaque email. Le destinataire vérifie la signature via la clé publique en DNS. Prouve que l'email n'a pas été modifié
  • DMARC — définit quoi faire si SPF ET DKIM échouent : none (surveiller), quarantine (spam), reject (bloquer). Envoie des rapports sur les tentatives de spoofing
    v=DMARC1; p=reject; rua=mailto:dmarc@soopha.com
  • Sans ces 3 protections : n'importe qui peut envoyer des emails au nom de ton domaine

SSH — L'Accès Distant Sécurisé

Secure Shell — le protocole indispensable de tout administrateur Linux et pentester

🔑 Authentification par clé — la bonne pratique
# 1. Générer une paire de clés
$ ssh-keygen -t ed25519 -C "beni@soopha.com"
→ ~/.ssh/id_ed25519 (clé privée — JAMAIS PARTAGER)
→ ~/.ssh/id_ed25519.pub (clé publique — OK à partager)

# 2. Copier la clé publique sur le serveur
$ ssh-copy-id user@serveur.com
# → ajoute dans ~/.ssh/authorized_keys

# 3. Se connecter sans mot de passe
$ ssh user@serveur.com

# 4. Désactiver l'auth par mot de passe
PasswordAuthentication no /etc/ssh/sshd_config
🔧 SSH — Au-delà de la connexion
# Copier des fichiers
$ scp fichier.txt user@serveur:/tmp/
$ rsync -avz ./dossier user@serveur:/backup/

# Tunneling local (SOCKS proxy)
$ ssh -D 9050 user@serveur.com
# → tout le trafic via le serveur SSH

# Port forwarding local
$ ssh -L 8080:localhost:80 user@serveur
# → localhost:8080 = serveur:80

# Exécuter une commande à distance
$ ssh user@serveur "df -h && uptime"
"Hardening SSH : changer le port (ex: 2222), désactiver root login, utiliser ed25519 (plus sûr que RSA-2048), activer fail2ban, limiter les IPs autorisées avec AllowUsers. Ces 5 mesures éliminent 99% des attaques bruteforce automatisées."

FTP vs SFTP vs FTPS — Ne Plus Jamais Utiliser FTP

File Transfer Protocol — comprendre pourquoi FTP est une catastrophe sécuritaire

FTP — Port 20/21
Protocole de 1971 — aucun chiffrement. Identifiants (login/mot de passe) et données transmis en clair. N'importe qui sur le réseau peut les capturer avec Wireshark. Deux modes : actif (problèmes NAT) et passif.
⚠ BANNIR DÉFINITIVEMENT — remplacer par SFTP ou FTPS
SFTP — Port 22
SSH File Transfer Protocol — pas un FTP chiffré mais un protocole de transfert de fichiers intégré à SSH. Tout est chiffré. Un seul port. Authentification par clé SSH possible. Le standard recommandé.
sftp user@serveur.com · Filezilla (mode SFTP) · WinSCP
FTPS — Port 990 (Explicit: 21)
FTP over TLS/SSL — le vrai FTP chiffré. Deux modes : Explicit (STARTTLS sur port 21) et Implicit (port 990). Compatible avec les infrastructures FTP existantes. Complexe à configurer avec le NAT et les firewalls.
Acceptable si SFTP impossible — FileZilla supporte les deux
SCP — Port 22
Secure Copy Protocol — transfert de fichiers via SSH, simple et efficace. Pas d'interface, juste une commande. Déprécié dans OpenSSH au profit de SFTP mais toujours très utilisé.
scp fichier.txt user@host:/dest/ · scp -r dossier/
RSYNC — Port 873 ou SSH
Synchronisation efficace de fichiers — ne transfère que les différences (delta). Via SSH pour le chiffrement. Idéal pour les sauvegardes incrémentales, la synchronisation de serveurs.
rsync -avz --delete src/ user@host:/dest/
Démo Wireshark : FTP vs SFTP
En capturant du trafic FTP dans Wireshark, on voit : USER admin → PASS MonMotDePasse123 en clair dans les paquets. En SFTP : données chiffrées, impossible à lire.
filter: ftp → voir les credentials en clair · ssh → chiffré
// Slide 10 · ARP · Address Resolution Protocol · ARP Spoofing · MITM

ARP — La Résolution MAC et ses Failles

Address Resolution Protocol — trouver l'adresse MAC d'une IP locale et l'attaque MITM classique

🔍 Comment ARP fonctionne
  • Quand tu pingues 192.168.1.1, ton OS a besoin de l'adresse MAC du routeur pour construire la trame Ethernet
  • ARP Request — broadcast : "Qui a l'IP 192.168.1.1 ? Réponds à AA:BB:CC:11:22:33"
  • ARP Reply — le routeur répond : "C'est moi ! Mon MAC est 11:22:33:44:55:66"
  • La correspondance IP↔MAC est stockée dans la table ARP (cache pendant ~2 min)
  • ARP est stateless et sans authentification — n'importe qui peut répondre à la place du routeur
$ arp -a          # voir la table ARP
$ arping 192.168.1.1  # requête ARP manuelle
⚔️ ARP Spoofing / ARP Poisoning
  • L'attaquant envoie de fausses ARP Replies sans qu'on lui ait posé de question (Gratuitous ARP)
  • Il dit à Alice : "L'IP du routeur (192.168.1.1) c'est MON MAC"
  • Il dit au routeur : "L'IP d'Alice (192.168.1.10) c'est MON MAC"
  • Résultat : tout le trafic d'Alice passe par l'attaquant → MITM (Man in the Middle)
  • Sur WiFi ouvert avec un hub, c'est trivial. Outil : arpspoof, Ettercap, Bettercap
# Détecter ARP spoofing
$ arp -a | grep -c "192.168.1.1"
# Si >1 MAC pour la même IP → ARP spoof!

SPF · DKIM · DMARC — La Triade Anti-Spoofing

Sans ces 3 protections, n'importe qui peut envoyer des emails en ton nom — voici comment les configurer

🔧 Configuration concrète pour soopha-learn.com
# 1. Enregistrement SPF (dans DNS TXT)
v=spf1 include:_spf.google.com include:sendgrid.net ip4:41.243.x.x -all
# → autorise Google, SendGrid, et l'IP fixe. -all = rejeter tout le reste

# 2. Enregistrement DKIM (dans DNS TXT, clé publique)
mail._domainkey.soopha-learn.com TXT "v=DKIM1; k=rsa; p=MIIBIjAN..."
# → le serveur mail signe chaque email avec la clé privée correspondante

# 3. Enregistrement DMARC (dans DNS TXT)
_dmarc.soopha-learn.com TXT "v=DMARC1; p=reject; rua=mailto:dmarc@soopha.com"
# → rejeter les emails qui échouent SPF ET DKIM · recevoir des rapports

# Vérifier la config
$ dig soopha-learn.com TXT | grep spf
$ dig _dmarc.soopha-learn.com TXT
SPF = qui peut envoyer DKIM = signature du contenu DMARC = quoi faire si ça échoue

Protocoles Modernes — L'Évolution de la Sécurité Réseau

Les protocoles qui remplacent les anciens et renforcent la confidentialité du réseau

DoH — DNS over HTTPS
Les requêtes DNS chiffrées via HTTPS (port 443). Empêche le FAI et les intermédiaires de voir quels domaines tu consultes. Cloudflare (1.1.1.1), Google (8.8.8.8) et Mozilla le supportent.
Activer dans Firefox : Paramètres → DNS over HTTPS
DoT — DNS over TLS
DNS chiffré via TLS sur le port 853. Alternative à DoH — plus simple à configurer au niveau du réseau/routeur. Cloudflare, Quad9 (9.9.9.9) le supportent.
systemd-resolved · Unbound · port 853
QUIC / HTTP/3
Protocole de transport de Google basé sur UDP (pas TCP). Intègre TLS 1.3 nativement, 0-RTT connection (moins de latence), multiplexage sans head-of-line blocking. Utilisé par 30%+ d'Internet.
curl --http3 https://google.com · port UDP 443
WireGuard
VPN moderne — 4000 lignes de code vs 70 000 pour OpenVPN. ChaCha20 + Poly1305 + Curve25519. Performance 2-3x supérieure à OpenVPN. Intégré au kernel Linux depuis 5.6.
wg genkey · wg pubkey · port UDP 51820
STARTTLS vs SSL/TLS implicite
STARTTLS = connexion en clair puis upgrade chiffré (ports 25, 587, 143). SSL/TLS implicite = chiffré dès le départ (ports 465, 993, 995). STARTTLS vulnérable au downgrade attack.
Préférer ports implicites : 993 (IMAP/S) · 465 (SMTPS)
mTLS — Mutual TLS
TLS bidirectionnel — non seulement le serveur prouve son identité, mais le client aussi. Utilisé entre microservices, dans les architectures Zero Trust, l'API banking. Le serveur rejette les clients sans certificat valide.
Kubernetes Istio · API M-Pesa · architecture Zero Trust

Récapitulatif — Sécurisé vs À Bannir

La carte de décision rapide pour choisir le bon protocole à chaque situation

🔴 Bannir définitivement
  • FTP (ports 20/21) — remplacer par SFTP ou FTPS
  • Telnet (port 23) — remplacer par SSH
  • HTTP (port 80) — forcer HTTPS avec HSTS
  • SMTP sans chiffrement (port 25 ouvert) — utiliser 587+STARTTLS ou 465
  • WEP/WPA WiFi — passer à WPA3 ou WPA2+AES minimum
  • MD5/SHA1 pour les certificats — utiliser SHA-256 minimum
  • SSL 2.0/3.0, TLS 1.0/1.1 — désactiver, utiliser TLS 1.2+ uniquement
  • Mot de passe root SSH — authentification par clé obligatoire
✅ Standards recommandés
  • SFTP (port 22) — transfert de fichiers chiffré
  • SSH (port 22) avec clé ed25519 — accès distant
  • HTTPS (port 443) avec TLS 1.3 — web
  • IMAP/S (port 993) — réception email
  • SMTP/S (port 465) ou 587+STARTTLS — envoi email
  • SPF + DKIM + DMARC — anti-spoofing email
  • DNS over HTTPS/TLS — résolution DNS privée
  • WireGuard ou OpenVPN — VPN moderne
  • Headers de sécurité HTTP : HSTS, CSP, X-Frame-Options
