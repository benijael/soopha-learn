// 01 · Cartographie

L'écosystème DFIR : Digital Forensics & Incident Response

DFIR — Les deux disciplines
🔬 DIGITAL FORENSICS

Science de la collecte et l'analyse de preuves numériques. Reconstituer ce qui s'est passé après un incident. Disque, RAM, réseau, cloud. Recevable en justice.

🚨 INCIDENT RESPONSE

Processus de réponse en temps réel à une attaque. Contenir, éradiquer, rétablir et tirer les leçons. Limiter l'impact business et éviter la récidive.

// Forensics = le passé → IR = le présent.
Les deux ensemble = DFIR.
Les 6 phases du cycle IR (NIST SP 800-61)
01 Préparation Plans, outils, équipe, runbooks
02 Détection SIEM, EDR, alertes, signalement
03 Analyse Qualification, scope, timeline
04 Containment Isolation — sans éteindre
05 Éradication Supprimer la persistance, patcher
06 Post-incident Rapport, leçons, amélioration
Les 4 sources de preuves numériques
  • Disque / Storage — Fichiers, MFT, journaux NTFS, LNK, prefetch, registre Windows, historiques shell. Persistant mais modifiable.
  • Mémoire RAM — Processus actifs, connexions réseau, credentials en mémoire, clés de chiffrement, injections. Volatile — disparaît à l'extinction.
  • Réseau / PCAP — Trafic capturé, flux NetFlow, logs firewall, DNS, proxy. Reconstituer les communications C2.
  • Logs & Cloud — Logs Windows Event, Syslog, CloudTrail, Entra ID Audit. Source de vérité chronologique.
Ordre de volatilité — RFC 3227

Toujours collecter du plus volatile au moins volatile. Une preuve perdue est perdue pour toujours.

1. RAM / Registres CPU → secondes
2. Cache / ARP / routing → minutes
3. Processus / connexions→ variable
4. Disque local → jours
5. Logs distants / cloud → semaines
6. Archives / sauvegardes → mois/années
// 02 · Fondamentaux

Les piliers de la forensics numérique

⛓ Chaîne de custody — l'intégrité des preuves
Une preuve numérique sans chaîne de custody est irrecevable devant un tribunal. Si tu ne peux pas prouver que tu ne l'as pas modifiée, elle ne vaut rien.
01
Hash avant acquisition — MD5 + SHA-256 du support original. C'est l'empreinte de référence.
02
Acquisition sur support vierge — Image bit-à-bit avec write blocker hardware. Jamais sur l'original.
03
Hash de l'image — Doit être identique au hash original. Preuve d'intégrité.
04
Documentation — Qui, quoi, quand, où, comment. Chaque action horodatée et signée.
OriginalSHA256: a3f4b2c1d8e9f0...✓ Ref
Image .ddSHA256: a3f4b2c1d8e9f0...✓ Conforme
Copie travailSHA256: a3f4b2c1d8e9f0...✓ Conforme
Outils d'acquisition
dd / dcfldd · Linux FTK Imager · Windows Guymager · GUI Linux WinPmem · RAM dump DumpIt · RAM Windows LiME · RAM Linux kernel
# Acquisition disque bit-à-bit
dd if=/dev/sda of=disk.dd bs=4M conv=noerror,sync
# Hash de vérification
sha256sum /dev/sda > hash_original.txt
sha256sum disk.dd > hash_image.txt
diff hash_original.txt hash_image.txt
Artefacts Windows clés — ce que l'attaquant laisse
Prefetch → Programmes exécutés (C:\Windows\Prefetch)
LNK Files → Fichiers récemment ouverts
Jump Lists → Fichiers et apps récentes par utilisateur
MFT → Métadonnées de TOUS les fichiers NTFS
Registre SYSTEM → Services, autoruns, USB connectés
SAM / SECURITY → Hashes de mots de passe locaux
Event Logs → 4624 login, 4688 process, 4698 task
Amcache.hve → Exécutables récents + hash SHA-1
ShimCache → Historique d'exécution de programmes
Timeline analysis — la clé de toute investigation

Reconstituer une chronologie unifiée de tous les événements : logs système, accès fichiers, connexions réseau, events Windows. Identifier le patient zéro et l'ordre exact des actions.

log2timeline · plaso Timesketch Hayabusa Eric Zimmerman Tools
# Générer une timeline unifiée avec plaso
log2timeline.py timeline.plaso disk.dd
psort.py -o l2tcsv timeline.plaso > timeline.csv

Scénario : activation d'un incident P1
06:47
Alerte EDR critique — CrowdStrike détecte une injection de processus sur un serveur de production. L'analyste de nuit active le war room et appelle l'équipe DFIR.
07:15
Briefing de crise — RSSI, DSI, juriste. Activation du plan de réponse aux incidents. Attribution des rôles : Lead IR, Forensics, Communications, Legal.
07:30
Isolation réseau — Couper les machines suspectes du réseau (firewall ACL, VLAN isolation) sans les éteindre. Préserver la RAM et les connexions actives.
08:00
Acquisition mémoire RAM — DumpIt sur les 3 machines isolées. Chaque dump est hashé et horodaté. Copie sur support forensique dédié hors réseau.
09:00
Analyse Volatility — Processus suspects, connexions C2 actives, DLL injections, credentials en mémoire. Premier pivot : identification de l'IP C2.
11:00
Investigation EDR & logs — Requêtes CloudTrail, logs AD (4624/4625/4688/4698), Sysmon. Reconstruction de la kill chain.
14:00
Rapport d'avancement — Point avec la direction : périmètre confirmé, données potentiellement exfiltrées ?, décision sur la notification RGPD (72h).
17:00
Containment complet — Reset credentials compromis, suppression persistence, blocage IOCs dans le SIEM. Début de la phase de reconstruction.
La règle d'or : isoler SANS éteindre
Éteindre une machine compromise en cours d'analyse, c'est brûler la scène de crime. La RAM, les processus actifs, les connexions C2 en cours, les clés de déchiffrement — tout disparaît en quelques secondes.
  • Isoler réseau — Couper le câble ou désactiver la NIC via EDR
  • Dumper la RAM — Avant toute autre action forensique
  • Ne jamais éteindre — Sauf si l'attaquant chiffre activement les données
  • Ne pas relancer — Chaque reboot écrase les artefacts volatils

// 04 · Cas Concret
🔴 CAS CONCRET #1

Analyse mémoire RAM — Mimikatz détecté

Volatility 3 · Credential Dump · Process Injection · Windows 10
Contexte — Alerte EDR
[CrowdStrike] CRITICAL — Credential Theft Detected
Host : WS-FINANCE-12 · Windows 10 22H2
User : marie.dupont (Domain Admin)
Process : svchost.exe (PID 4412) → LSASS access
Time : 2024-03-15 03:22:41 UTC
Technique: T1003.001 · LSASS Memory
Questions à résoudre
  • Quel processus a accédé à LSASS ? Est-il légitime ?
  • Y a-t-il eu injection dans un processus parent légitime ?
  • Quels credentials ont été dumpés ?
  • Y a-t-il des connexions réseau suspectes en mémoire ?
  • Depuis quand l'attaquant est-il présent sur la machine ?
Volatility 3 — Commandes d'investigation
# 1. Lister les processus actifs au moment du dump
vol -f ram.dmp windows.pslist
# 2. Détecter les injections de code (hollow/injection)
vol -f ram.dmp windows.malfind
# 3. Connexions réseau au moment du dump
vol -f ram.dmp windows.netstat
# 4. DLLs chargées par un process suspect
vol -f ram.dmp windows.dlllist --pid 4412
# 5. Dump du processus injecté pour analyse
vol -f ram.dmp windows.dumpfiles --pid 4412
# 6. Strings Mimikatz dans la mémoire
strings ram.dmp | grep -i "sekurlsa\|mimikatz\|wdigest"
Résultats de l'analyse — Findings
F1
Process creux (hollowing) — svchost.exe PID 4412 spawné par explorer.exe (anormal — svchost doit être enfant de services.exe). malfind détecte une région RWX avec du code injecté.
F2
Connexion C2 active — netstat révèle une connexion ESTABLISHED vers 185.220.x.x:443 depuis le process injecté. IP appartenant à un exit node Tor.
F3
Strings Mimikatz — "sekurlsa::logonpasswords", "lsadump::dcsync" trouvés dans la mémoire du process. Credentials NTLM potentiellement exfiltrés.
F4
Pivot latéral — Dans le registre mémoire, connexions SMB vers 3 autres machines du domaine dans les 30 minutes précédant l'alerte.
F5
Persistence — Clé Run dans le registre en mémoire : HKCU\Software\Microsoft\Windows\CurrentVersion\Run pointant vers %TEMP%\svc.exe
Remédiation immédiate
  • Isoler les 3 machines identifiées comme cibles de pivot
  • Reset forcé de TOUS les mots de passe du domaine (compromise confirmé)
  • Révoquer tous les tickets Kerberos (krbtgt reset ×2)
  • Bloquer l'IP C2 sur tous les firewalls + IOC dans le SIEM
  • Supprimer la clé de registre de persistence + svc.exe
  • Notification RGPD CNIL sous 72h si données personnelles exposées

KAPE — Triage rapide des artefacts

KAPE (Kroll Artifact Parser and Extractor) collecte en quelques minutes tous les artefacts critiques d'un système Windows. Il existe des targets prédéfinies pour chaque use case forensique.

# Collecte rapide des artefacts IR
kape.exe --tsource C: \
  --tdest D:\Evidence \
  --target !SANS_Triage,EventLogs, \
    Prefetch,Amcache,NTDS \
  --module !EZParser
# Résultat : CSV parsés prêts pour Timesketch
Eric Zimmerman Tools — Parse les artefacts
MFTECmd · MFT Parser PECmd · Prefetch RECmd · Registre EvtxECmd · Event Logs ShellBagsExplorer WxTCmd · Timeline
Lessons learned — Remédiation

  • Désactiver l'exécution PHP dans le répertoire uploads
  • WAF en mode blocage avec règles upload
  • Sysmon déployé pour loguer les créations de processus
  • Centraliser les logs IIS dans le SIEM avec rétention 90j

🔴 CAS CONCRET #3

Incident Response P0 — Attaque ransomware en cours

Playbook IR complet · Velociraptor · Patient zéro · Rapport légal
07h03 — Le téléphone sonne
Le DSI appelle : "La moitié des serveurs affichent une note de rançon."
Fichiers chiffrés avec extension .locked.
Demande : 800 000$ en Bitcoin. Timer : 72h.
Le groupe opère depuis 3 semaines sans être détecté.
Premières décisions — les 30 premières minutes
T+0
Isolation réseau total — Couper tous les segments du réseau. Mieux vaut une interruption de service que la propagation continue.
T+5
War room activé — RSSI, DG, juriste, communication. Décision : NE PAS payer (favorise la récidive, aucune garantie).
T+15
Conserver les machines chiffrées — Ne pas rebuilder immédiatement. La note de rançon et les fichiers chiffrés sont des preuves.
T+30
Identification du ransomware — ID Ransomware (nomoreransom.org) : BlackBasta. Vérifier si un déchiffreur existe (spoiler : non).
Velociraptor — Chasse du patient zéro à grande échelle

Velociraptor permet d'interroger simultanément des milliers de machines en temps réel pour trouver le patient zéro sans analyser chaque machine individuellement.

/* Hunt VQL — trouver le premier .locked créé */
SELECT Fqdn, FullPath, Mtime
FROM glob(globs="C:/**/*.locked")
ORDER BY Mtime
LIMIT 1
/* Résultat : WS-COMPTA-03 — 2024-03-12 03:17 */
/* Patient zéro identifié en 4 minutes sur 350 machines */
  • Patient zéro : WS-COMPTA-03 — poste du service comptabilité
  • Vecteur initial : Macro Excel malveillante reçue par email J-21
  • Séjour attaquant : 21 jours — exfiltration de 450GB avant chiffrement
Rapport d'incident final — Structure légale
01
Executive Summary — 1 page. Impact business, timeline clé, conclusion sur l'attribution. Pour la direction et les assureurs.
02
Timeline technique détaillée — Chaque action de l'attaquant horodatée avec la source de preuve. Attestable en justice.
03
Indicators of Compromise (IOC) — Hashes, IPs, domaines C2, règles YARA, règles Sigma. Partagés avec le SIEM et les partenaires.
04
Root cause analysis — Pourquoi l'attaque a réussi ? Quelle défaillance process/technique/humaine ?
05
Plan de remédiation priorisé — Actions immédiates, court terme, long terme. Avec responsables et délais.
06
Obligations légales — Notification CNIL (72h RGPD), dépôt de plainte police, notification aux assureurs cyber.
Checklist reconstruction post-ransomware
  • Rebuilder depuis les sauvegardes immuables les plus récentes (hors réseau)
  • Patcher toutes les CVE exploitées avant reconnexion
  • Changer TOUS les mots de passe du domaine (krbtgt ×2)
  • Déployer EDR sur 100% des endpoints avant reconnexion
  • Tester les backups AVANT l'incident — pas pendant
// 07 · Outils

Les outils DFIR en profondeur

⭐ Volatility 3
Open Source

Le framework de référence mondiale pour l'analyse de mémoire RAM. Supporte Windows, Linux, macOS. Plugins pour processus, réseau, injections, registre, hives. Standard des CERT et CSIRT mondiaux.

  • Totalement gratuit et open source
  • Communauté de plugins très active
  • Indispensable pour toute certification DFIR
  • Nécessite de connaître les profils OS (internals)
pslistmalfindnetstatdlllisthivelistcmdline
Velociraptor
Open Source

Plateforme de collecte et chasse sur endpoint à grande échelle. Agent léger déployé sur tous les endpoints, interrogeable en temps réel via VQL (Velociraptor Query Language). Standard des grandes cellules IR.

  • Interroger 10 000 machines simultanément
  • Collecte forensique distante sans déplacement
  • Artefacts KAPE intégrés nativement
  • Complexe à déployer en infrastructure from scratch
Autopsy / FTK
Confirmé

GUI complète pour l'analyse forensique de disques. Autopsy (open source, basé sur The Sleuth Kit) et FTK (commercial) permettent de naviguer dans les systèmes de fichiers, récupérer les fichiers supprimés, analyser le registre, les logs, les artefacts web.

  • Autopsy totalement gratuit
  • Récupération de fichiers supprimés (unallocated space)
  • Génération de rapports HTML/PDF pour tribunal
Ghidra / IDA Pro
Expert

Reverse engineering de malwares. Ghidra (gratuit, NSA) et IDA Pro (référence commerciale) décompilent les binaires malveillants pour comprendre leur comportement : C2 hardcodé, capacités d'évasion, algorithme de chiffrement.

  • Ghidra totalement gratuit et très puissant
  • Indispensable pour analyser un ransomware custom
  • Courbe d'apprentissage longue (assembleur x86/x64)

Forensics réseau & analyse de logs Windows

Wireshark — Analyse PCAP forensique

Un PCAP capturé pendant l'incident est une mine d'or. On peut reconstituer les communications C2, extraire les fichiers transférés, identifier les protocoles utilisés.

# Filtres Wireshark essentiels en DFIR
http.request.method == "POST" # Uploads
dns.qry.name contains "pastebin" # C2 via DNS
tcp.flags.syn==1 && tcp.flags.ack==0 # Scan
# Exfil volumineuse
smb2.filename contains ".exe" # Lateral move
# Extraire les fichiers d'un PCAP
File → Export Objects → HTTP/SMB

Hayabusa — Détection rapide sur Event Logs

Hayabusa analyse des gigaoctets de logs Windows Event (.evtx) en quelques minutes et génère une timeline d'alertes basée sur des règles Sigma. Parfait pour le triage initial d'un incident.

# Analyser tous les event logs d'un dossier
hayabusa-x86_64 csv-timeline \
  --directory C:\Windows\System32\winevt\Logs \
  --output timeline.csv \
  --min-level medium
# Output : CSV avec alertes Sigma triées par timestamp
# Importer dans Timesketch pour visualisation
Anti-forensics — Ce que l'attaquant fait pour effacer ses traces
  • Effacer les event logs — wevtutil cl System / Security (Event 1102)
  • Timestomping — Modifier les dates MACB des fichiers pour brouiller la timeline
  • Supprimer les prefetch — Effacer C:\Windows\Prefetch\ pour masquer les exécutions
  • Overwrite du disque — sdelete, cipher /w pour empêcher la récupération
  • La parade : Centraliser les logs (SIEM) avant l'incident. Ce qui est en dehors de la machine ne peut pas être effacé localement.

// Récap · DFIR

Ce qu'il faut retenir

✅ LES 5 RÉFLEXES DU FORENSICIEN
  • Ordre de volatilité — Toujours du plus volatile (RAM) au moins volatile (disque)
  • Ne jamais éteindre — Isoler réseau, dumper RAM, analyser. Dans cet ordre.
  • Hash partout — Chaque preuve est hashée avant et après. Intégrité inviolable.
  • Documenter en temps réel — Chaque action horodatée. La mémoire est faillible.
  • Centraliser les logs — Ce qui est dans le SIEM ne peut pas être effacé localement.
⚠ LES 3 ERREURS QUI DÉTRUISENT UNE INVESTIGATION
  • Éteindre la machine — La RAM est perdue. Les connexions C2 disparaissent. Les clés de chiffrement évaporées.
  • Analyser sur l'original — Toujours travailler sur une copie forensique. L'original ne se touche qu'avec un write blocker.
  • Rebuilder avant d'investiguer — On détruit les preuves. Le patient zéro reste inconnu. L'attaquant reviendra.
