// 00 · Définition

La Forensics numérique — c'est quoi ?

🔬 DÉFINITION
"L'art de collecter, préserver et analyser des preuves numériques pour reconstituer ce qui s'est passé sur un système informatique — avant, pendant et après une intrusion."
  • On analyse l'ordinateur de la victime, pas de l'attaquant
  • Tout doit être légalement recevable — chaîne de custody obligatoire
  • La première règle : ne jamais modifier les preuves
  • Discipline à la croisée du droit, de l'informatique et de l'investigation
🌍 POURQUOI C'EST CRUCIAL EN AFRIQUE
  • Fraudes Mobile Money — millions FCFA détournés, preuves numériques essentielles pour les poursuites
  • Ransomware sur PME africaines — comprendre l'attaque pour ne plus être victime
  • Cyberespionnage institutionnel — administrations et banques ciblées
  • Les tribunaux africains reconnaissent de plus en plus les preuves numériques
  • CSIRT nationaux en création — Gabon, Sénégal, CI, Cameroun
💡 Un expert DFIR local a une valeur ajoutée unique — il connaît le contexte africain.
// 01 · Méthodologie

Les 4 phases du DFIR

01
Identification
Déterminer qu'un incident a eu lieu. Périmètre, systèmes touchés, timeline.
02
Préservation
Acquérir les preuves sans les altérer. Write-blockers, hash MD5/SHA256, chaîne de custody.
03
Analyse
Examiner les artefacts : fichiers, logs, mémoire, réseau. Reconstituer l'attaque.
04
Rapport
Documenter les trouvailles. Rapport technique + résumé exécutif. Recevable en justice.
⚠️ LA RÈGLE D'OR — ORDER OF VOLATILITY
Acquérir les preuves du plus volatile au moins volatile — les données RAM disparaissent à l'extinction, le disque dur survit.
1. RAM (volatile) → 2. Processus / réseau → 3. Disque dur → 4. Logs système → 5. Archives / backups
// 02 · Forensics Disque

🗂️ Forensics disque — Acquérir & Analyser

ACQUISITION — COPIE FORENSIQUE
On ne travaille jamais sur l'original. On crée une image bit-à-bit et on vérifie l'intégrité avec un hash.
# Créer une image forensique avec dd
$ dd if=/dev/sda of=disque.img bs=4M status=progress

# Mieux : dcfldd (avec hachage automatique)
$ dcfldd if=/dev/sda of=disque.img hash=sha256 hashlog=hash.txt

# Vérifier l'intégrité
$ sha256sum disque.img
# → doit correspondre au hash d'origine
ANALYSE — CE QU'ON CHERCHE
  • Fichiers supprimés — récupérables car seul l'index est effacé, pas les données
  • Timeline des accès — timestamps MAC (Modified, Accessed, Created)
  • Métadonnées — auteur d'un document Word, coordonnées GPS d'une photo
  • Données cachées — steganographie, ADS (Alternate Data Streams Windows)
  • Artefacts navigateur — historique, cookies, mots de passe en cache
Autopsy FTK Imager Sleuth Kit Foremost Binwalk
// 03 · Forensics Mémoire

🧠 Forensics RAM — La mémoire ne ment pas

POURQUOI LA RAM EST PRÉCIEUSE
  • Contient les processus en cours — y compris les malwares
  • Clés de chiffrement en clair en mémoire
  • Connexions réseau actives au moment de l'incident
  • Mots de passe et tokens d'authentification non chiffrés
  • Les malwares fileless n'existent qu'en RAM — invisibles sur le disque
CAPTURE
# Capturer la RAM (Windows — WinPmem)
C:\> winpmem.exe memdump.raw

# Capturer la RAM (Linux — LiME)
$ insmod lime.ko "path=/tmp/ram.lime format=lime"
VOLATILITY — L'OUTIL DE RÉFÉRENCE
# Identifier le profil OS
$ vol.py -f memdump.raw imageinfo

# Lister les processus
$ vol.py -f memdump.raw --profile=Win10x64 pslist

# Processus cachés (rootkit)
$ vol.py -f memdump.raw psscan

# Connexions réseau actives
$ vol.py -f memdump.raw netscan

# Extraire un fichier de la mémoire
$ vol.py -f memdump.raw dumpfiles -Q 0x3e8

# Hashes des mots de passe Windows (SAM)
$ vol.py -f memdump.raw hashdump
// 04 · Forensics Réseau

📡 Forensics réseau — Lire les captures PCAP

C'EST QUOI UN FICHIER .PCAP ?
Un fichier .pcap (Packet Capture) enregistre tout le trafic réseau. En forensics, c'est l'équivalent d'une vidéosurveillance du réseau.
  • Capturer avec Wireshark, tcpdump, tshark
  • Peut contenir des fichiers transférés reconstituables
  • Révèle les commandes C2 (Command & Control) d'un malware
  • Permet de voir l'exfiltration de données
# Capturer le trafic en live
$ tcpdump -i eth0 -w capture.pcap

# Filtrer par IP suspecte
$ tshark -r capture.pcap -Y "ip.addr==185.220.101.47"
ANALYSE WIRESHARK — FILTRES FORENSICS
# Trafic HTTP (données en clair)
http

# DNS — domaines contactés
dns

# Connexions vers IP externe suspecte
ip.dst == 185.220.101.47

# Fichiers transférés (FTP, HTTP)
ftp-data || http.content_type contains "application"

# Reconstituer un fichier transféré
File → Export Objects → HTTP

# Suivre une session complète
Clic droit → Follow → TCP Stream
💡 Outil complémentaire : NetworkMiner — reconstruit automatiquement les fichiers transférés depuis un .pcap

// 06 · Artefacts Linux

🐧 Artefacts Linux — Traces et investigation

📋 FICHIERS CLÉS À ANALYSER
/var/log/auth.logConnexions SSH, sudo, su — qui a fait quoi
~/.bash_historyCommandes exécutées — souvent effacé par l'attaquant
/etc/crontab + /etc/cron.d/Persistance via tâches planifiées
/tmp/ et /dev/shm/Dossiers favoris des malwares — écriture sans privilèges
/etc/passwd + /etc/shadowComptes créés par l'attaquant (UID 0 suspect)
~/.ssh/authorized_keysBackdoor SSH — clé publique ajoutée pour persistance
⚙️ COMMANDES D'INVESTIGATION
# Qui s'est connecté et quand ?
$ last -F | head -30

# Connexions SSH échouées
$ grep "Failed" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn

# Fichiers modifiés dans les 24h
$ find / -mtime -1 -type f 2>/dev/null | grep -v proc

# Processus suspects avec connexions réseau
$ ss -tulnp | grep -v "127.0.0.1"

# Fichiers SUID (escalade privilèges)
$ find / -perm -4000 -ls 2>/dev/null

# Historique même si effacé
$ strings /proc/*/environ 2>/dev/null | grep HIST
// 07 · Timeline Analysis

📅 Reconstruire la timeline d'une attaque

POURQUOI LA TIMELINE EST ESSENTIELLE
"Sans timeline, on a des preuves isolées. Avec elle, on a une histoire cohérente qui prouve comment l'attaque s'est déroulée."
  • Corréler les timestamps de plusieurs sources — logs, fichiers, réseau, RAM
  • Identifier le patient zéro — premier système compromis
  • Comprendre le vecteur d'entrée — phishing, exploit, insider
  • Mesurer le dwell time — combien de temps l'attaquant est resté
ATTENTION AUX TIMESTAMPS
Un attaquant peut falsifier les timestamps (timestomping). Croiser les sources valide l'authenticité.
⚙️ CONSTRUIRE LA TIMELINE
# Plaso — agrège toutes les sources en une timeline
$ log2timeline.py -z UTC output.plaso /chemin/evidence/
$ psort.py -z UTC output.plaso > timeline.csv

# Mactime (Sleuth Kit) — timeline depuis un disque
$ fls -r -m / disque.img | mactime -d > timeline.txt

# Filtrer les événements d'une fenêtre temporelle
$ grep "2024-03-15" timeline.csv | grep -v "normal"
Sources à corréler : logs système + événements Windows + timestamps fichiers + logs réseau + captures PCAP

// 09 · Labo PCAP

⚡ Labo — Analyser un .pcap d'attaque

MISSION — 30 MINUTES
Tu reçois un fichier capture.pcap d'un réseau suspect. Trouve ce qui s'est passé.
01
Ouvrir dans Wireshark → Statistics → Conversations — quelles IPs communiquent le plus ?
02
Filtrer le DNS : dns → quels domaines suspects sont résolus ?
03
Chercher du HTTP : http → File → Export Objects → HTTP — des fichiers ont-ils été transférés ?
04
Identifier l'IP attaquante : gros volume de paquets vers une cible ? Port scan → tcp.flags.syn == 1 && tcp.flags.ack == 0
05
Reconstruire la session suspecte : Clic droit sur un paquet → Follow → TCP Stream

// 10 · Labo Volatility

🧠 Labo — Analyser un dump mémoire

MISSION — 30 MINUTES
Tu reçois un memdump.raw d'une machine Windows compromise. Trouve le malware.
01
Identifier le profil : vol.py -f memdump.raw imageinfo
02
Lister les processus : pslist puis pstree — y a-t-il un processus avec un nom bizarre ?
03
Comparer pslist et psscan — les processus cachés par un rootkit n'apparaissent que dans psscan
04
Connexions réseau du malware : netscan — vers quelle IP contacte-t-il ?
05
Extraire le binaire suspect : procdump -p [PID] -D ./output/ puis analyser sur VirusTotal
// CHECKLIST — INDICES D'INFECTION
  • Processus avec nom ressemblant à un processus légitime : svchost32.exe, explorer32.exe
  • Processus parent anormal : cmd.exe lancé par un navigateur web
  • Connexion réseau vers une IP géolocalisée dans un pays suspect
  • Processus caché dans psscan mais absent de pslist → rootkit
  • Grande quantité de handles sur des fichiers système → injection de code
  • Utiliser malfind — détecte automatiquement le code injecté en mémoire
# malfind — détection code malveillant injecté
$ vol.py -f memdump.raw malfind --dump-dir ./malware_dump/
// 11 · Rapport & Afrique

📄 Le rapport DFIR & contexte Afrique

STRUCTURE D'UN RAPPORT DFIR
  • Résumé exécutif — 1 page, pour la direction et les juristes. Non technique.
  • Périmètre de l'analyse — quels systèmes, quelle période, quelles limitations
  • Chronologie de l'incident — timeline horodatée en UTC
  • Vecteur d'attaque initial — comment l'attaquant est entré
  • Preuves techniques — screenshots, hash des fichiers, commandes exécutées
  • Indicateurs de compromission (IoC) — IPs, domaines, hash de malwares
  • Recommandations — actions correctives prioritaires
🌍 FORENSICS EN AFRIQUE — OPPORTUNITÉS
  • CSIRT nationaux en création — Gabon (CERT-Gabon), Sénégal (CERT-SN), CI (CERT-CI), Cameroun
  • Les banques africaines (Ecobank, BGFI, Coris) recrutent des analystes DFIR locaux
  • Fraudes Mobile Money → besoin urgent d'experts pour les enquêtes judiciaires
  • Manque de cadre légal harmonisé — preuve numérique reconnue au cas par cas
  • Outils DFIR majeurs sont gratuits et open source — accessibles depuis l'Afrique
