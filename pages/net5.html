Wireshark — L'Outil Incontournable du Réseau

Analyser, capturer, décoder — tout le trafic réseau sous les yeux en temps réel

🖥️ Les 5 zones de l'interface
  • Barre de filtres (haut) — filtre d'affichage en temps réel, fond vert = filtre valide, rouge = erreur de syntaxe
  • Liste des paquets (milieu haut) — un paquet par ligne, colorisé par protocole
  • Détail du paquet (milieu bas) — arborescence des couches OSI, chaque champ cliquable
  • Vue hexadécimale (bas) — octets bruts avec correspondance ASCII, surlignage synchronisé
  • Barre de statut (bas) — nombre de paquets capturés/affichés, durée, taille
🎨 Colorisation des protocoles
HTTPVert clair — trafic web non chiffré
DNSBleu clair — requêtes de résolution
TCP SYNMauve — établissement de connexion
ARPJaune — résolution MAC locale
TCP RSTRouge — connexion réinitialisée
ICMPGris-vert — ping et messages ICMP
TLSBleu foncé — trafic chiffré HTTPS

Filtres d'Affichage — Trouver l'Aiguille dans la Botte de Foin

La compétence clé de Wireshark — écrire des filtres précis pour isoler ce qui t'intéresse

📌 Filtres essentiels par protocole
# Protocole seul
http          tous les paquets HTTP
dns          toutes les requêtes DNS
tls          trafic TLS/HTTPS
arp          trames ARP
tcp          tout TCP
udp          tout UDP
icmp         ping et messages ICMP

# Par adresse IP
ip.addr == 192.168.1.10
ip.src == 10.0.0.1
ip.dst == 8.8.8.8

# Par port
tcp.port == 443
tcp.dstport == 22
🔗 Opérateurs et combinaisons
# Opérateurs logiques
&& ou and   ET logique
|| ou or    OU logique
! ou not    négation

# Filtres combinés pratiques
http and ip.src == 192.168.1.5
HTTP venant d'une IP précise

tcp.port == 22 and ip.addr == 10.0.0.0/8
SSH depuis le réseau interne

dns and !ip.addr == 192.168.1.1
DNS sans le routeur local

http.request.method == "POST"
Formulaires web soumis uniquement

tcp.flags.syn == 1 and tcp.flags.ack == 0
SYN sans ACK → détection de scan
// Slide 04 · Filtres BPF · Capture Filters · tcpdump

Filtres BPF — Capturer Uniquement ce dont tu as Besoin

Berkeley Packet Filter — appliqué AVANT la capture pour réduire la taille des .pcap

⚡ Display Filter vs Capture Filter
  • Display Filter — appliqué APRÈS la capture. Tu as tous les paquets, tu filtres l'affichage. Syntaxe Wireshark. Modifiable à chaud.
  • Capture Filter (BPF) — appliqué PENDANT la capture. Seuls les paquets qui correspondent sont capturés. Syntaxe tcpdump. Non modifiable à chaud.
  • Pour les analyses courtes : pas de filtre de capture, filtrer l'affichage ensuite
  • Pour les captures longues (production, 24h+) : BPF obligatoire sinon le .pcap explose en taille
📋 Syntaxe BPF essentielle
# Capturer uniquement du HTTP
tcp port 80

# DNS uniquement
udp port 53

# Depuis/vers une IP
host 192.168.1.10
src host 10.0.0.1
dst host 8.8.8.8

# Un réseau entier
net 192.168.1.0/24

# Combinaisons
tcp port 443 or tcp port 80
not port 22 tout sauf SSH
host 10.0.0.5 and port 3306
"Différence clé : tcp port 80 est de la syntaxe BPF (capture), alors que tcp.port == 80 est de la syntaxe Wireshark (display). Les deux font la même chose mais à des moments différents — confondre les deux = filtre qui ne fonctionne pas."

Follow Stream — Reconstruire une Conversation Complète

La fonctionnalité la plus puissante de Wireshark pour comprendre ce qui s'est dit

🔄 Comment l'utiliser
1
Cliquer sur un paquet du flux qui t'intéresse (ex: un paquet HTTP vers le serveur cible)
2
Clic droit → Follow → TCP Stream (ou HTTP Stream, UDP Stream...)
3
Une fenêtre affiche l'échange complet — rouge = client → serveur, bleu = serveur → client
4
Possibilité de sauvegarder en brut, texte, ou hexadécimal pour analyse offline
5
La liste des paquets se filtre automatiquement sur ce stream — noter le filtre généré
💡 Ce que Follow Stream révèle
// Follow HTTP Stream sur un login en clair
POST /login HTTP/1.1
Host: admin.example.com
Content-Type: application/x-www-form-urlencoded

username=admin&password=P@ssw0rd123

HTTP/1.1 302 Found
Location: /dashboard
Set-Cookie: session=eyJhbGci...

// Le mot de passe est visible en clair !
// → Preuve qu'il faut forcer HTTPS

Naviguer dans une Capture .pcap

Les menus et outils Wireshark qui font gagner du temps lors d'une analyse forensique

Statistics → Protocol Hierarchy
Vue synthétique de tous les protocoles présents dans la capture avec leur proportion en %. Permet d'identifier immédiatement le protocole dominant et les anomalies.
Ex : 85% UDP = possible exfiltration DNS ou jeu · 60% ARP = suspect
Statistics → Conversations
Toutes les paires IP qui communiquent — triées par volume. Identifier immédiatement quelle IP génère le plus de trafic. Utile pour trouver un C2 ou une exfiltration.
Trier par Bytes desc → machine qui exfiltre en tête
Statistics → IO Graph
Graphique du débit réseau dans le temps. Pic de trafic à 3h du matin = exfiltration programmée. Plateau constant = tunnel (C2, VPN non autorisé).
Pic de trafic DNS à heure fixe → DNS tunneling
Analyze → Expert Information
Wireshark liste automatiquement les anomalies détectées : retransmissions TCP, connexions réinitialisées, erreurs. Un bon point de départ pour l'analyse forensique.
Beaucoup de TCP RST → scan de port ou firewall qui bloque
File → Export Objects → HTTP
Extraire tous les fichiers transmis en HTTP depuis la capture : images, scripts, archives, exécutables. Très utilisé en forensics pour récupérer des payloads malveillants.
Extraire des .exe ou .zip téléchargés depuis une capture
Edit → Find Packet
Rechercher dans les paquets par chaîne de caractères, valeur hex, ou filtre d'affichage. Trouver rapidement "password", "flag", "secret" dans une capture CTF.
String search → "SOOPHA{" pour trouver un flag CTF dans un pcap
// Slide 07 · tcpdump · CLI · Serveur distant · Sans GUI

tcpdump — Wireshark en Ligne de Commande

Capturer sur un serveur Linux distant sans interface graphique — indispensable pour les admins et pentesters

⚡ Commandes tcpdump essentielles
# Lister les interfaces disponibles
$ tcpdump -D

# Capturer sur eth0 (verbeux)
$ tcpdump -i eth0 -v

# Sauvegarder dans un .pcap
$ tcpdump -i eth0 -w capture.pcap

# Avec filtre BPF
$ tcpdump -i eth0 -w http.pcap 'tcp port 80'

# Afficher en ASCII (no résolution DNS)
$ tcpdump -i eth0 -A -n 'port 80'

# Limiter à 100 paquets
$ tcpdump -i eth0 -c 100 -w out.pcap

# Lire un .pcap existant
$ tcpdump -r capture.pcap -n
$ tcpdump -r capture.pcap 'tcp port 443'
🔧 Workflow capture distante → analyse locale
1
SSH sur le serveur distant : ssh user@serveur.com
2
Lancer la capture tcpdump : tcpdump -i eth0 -w /tmp/cap.pcap
3
Laisser tourner le temps nécessaire, puis Ctrl+C
4
Rapatrier le .pcap localement : scp user@serveur:/tmp/cap.pcap ./
5
Ouvrir dans Wireshark et analyser avec les filtres d'affichage
# Option avancée : pipe direct vers Wireshark
ssh user@srv "tcpdump -i eth0 -U -s0 -w -" | wireshark -k -i -
// Slide 08 · Analyse CTF .pcap · Méthodologie · QCM requis

Analyser un .pcap CTF — Méthodologie Systématique

Face à un fichier .pcap inconnu — les étapes dans l'ordre pour ne rien rater

1
Protocol Hierarchy (Statistics menu) — quels protocoles sont présents ? HTTP en clair ? DNS avec des requêtes longues ? SMTP ? Identifier le vecteur principal.
2
Conversations — quelles IPs communiquent le plus ? Trouver les acteurs principaux (attaquant, victime, C2). Trier par bytes.
3
Expert Information — anomalies automatiquement détectées. Beaucoup de TCP RST, retransmissions, connexions réinitialisées ?
4
Filtrer par protocole — si du HTTP est visible : http puis Follow TCP Stream sur chaque conversation. Chercher les credentials, les données exfiltrées, les flags.
5
Export Objects — si des fichiers ont été transférés (images, archives, exécutables), les extraire et analyser séparément.
6
Find Packet — rechercher des chaînes : "flag", "password", "SOOPHA{", "secret". Ou en hex si les données sont encodées.
7
DNS long queries — filtrer avec dns and frame.len > 100 — des sous-domaines très longs peuvent indiquer un DNS tunneling avec des données exfiltrées.

Détecter des Attaques dans une Capture

Les signatures réseau de chaque attaque — reconnaître l'ennemi dans le trafic

🔍 Scan de ports Nmap
Signature : de nombreux SYN vers des ports différents d'une même IP destination, sans suite de SYN-ACK. Les ports fermés répondent RST.
Filtre : tcp.flags.syn==1 && tcp.flags.ack==0
→ si 1 IP fait 1000 SYN en 1s = scan
🎭 ARP Spoofing (MITM)
Signature : une même MAC répond à plusieurs requêtes ARP pour des IPs différentes. Ou l'adresse MAC de la passerelle change soudainement dans les ARP Reply.
Filtre : arp.opcode == 2
→ chercher doublons MAC/IP suspects
🔨 Brute Force SSH
Signature : des centaines de connexions TCP port 22 depuis la même IP, avec TCP RST ou FIN rapide. Chaque tentative = un SYN/SYN-ACK/ACK puis RST ou échange court.
Filtre : tcp.dstport==22 && tcp.flags.syn==1
→ compter les connexions par IP source
🌐 DNS Tunneling
Signature : des requêtes DNS avec des sous-domaines très longs (50+ caractères), beaucoup de type TXT ou NULL, vers un seul domaine externe suspect.
Filtre : dns.qry.name contains "." && frame.len > 150
→ sous-domaines encodés = data exfiltrée
💥 SYN Flood DDoS
Signature : torrent de SYN vers une IP/port sans aucun ACK final. Les IPs sources sont souvent forgées (spoofées). Le serveur cible ne répond plus aux connexions légitimes.
Filtre : tcp.flags==0x002
→ des milliers de SYN par seconde
📤 Exfiltration de données
Signature : trafic volumétrique sortant vers une IP externe inhabituelle. Protocoles inhabituels (ICMP avec payload, DNS avec gros TXT). Souvent la nuit ou en dehors des heures de bureau.
Statistics → Conversations → trier par bytes desc
→ IP externe en tête = suspect

Cas Pratique — Analyser une Capture d'Attaque Réelle

Un scénario complet : du .pcap à l'identification de la compromission

📋 Scénario : machine compromise sur le réseau 192.168.1.0/24
# Étape 1 : Ouvrir le .pcap dans Wireshark
# Statistics → Protocol Hierarchy révèle :
TCP : 45% · HTTP : 30% · DNS : 15% · ARP : 10%
→ 10% ARP est SUSPECT sur un réseau de 5 machines

# Étape 2 : Filtrer les trames ARP
arp
→ 192.168.1.50 répond à TOUS les ARP Requests
→ même MAC pour 192.168.1.1 (routeur) ET .20 (victime)
→ ARP Spoofing confirmé — .50 est l'attaquant

# Étape 3 : Voir ce que l'attaquant intercepte
http and ip.addr == 192.168.1.20
# Follow TCP Stream sur le login HTTP
POST /login → username=admin&password=Secret2024!
→ Credentials interceptés en clair — MITM réussi

# Étape 4 : Chercher la persistance
dns and ip.src == 192.168.1.20
→ requêtes DNS vers c2.malware-domain.ru toutes les 60s
→ Malware avec beacon C2 installé sur la victime

Filtres Avancés — Aller Plus Loin

Les filtres complexes qui font la différence lors d'une analyse forensique poussée

🎯 Filtres forensics spécialisés
# Requêtes DNS avec noms longs (tunneling)
dns.qry.name matches "^[a-z0-9]{30,}"

# Credentials HTTP en clair
http.request.method == "POST" and http contains "password"

# Cookies de session
http.cookie or http.set_cookie

# ICMP avec payload (tunneling possible)
icmp and data.len > 32

# Connexions vers IPs non RFC1918 (Internet)
not ip.dst matches "192\.168\.|10\.|172\.(1[6-9]|2[0-9]|3[01])\."

# Téléchargements d'exécutables
http.content_type contains "application/octet-stream"

# Retransmissions TCP (réseau dégradé ou IDS)
tcp.analysis.retransmission

# Paquets avec TTL anormal (IP spoofing ?)
ip.ttl < 10
🔧 Outils complémentaires
  • tshark — Wireshark CLI, idéal pour les scripts et le parsing automatisé
    tshark -r cap.pcap -Y 'http' -T fields -e http.host -e http.request.uri
  • NetworkMiner — reconstruction automatique des fichiers et sessions depuis un .pcap. Extraction OS fingerprinting.
  • Zeek (Bro) — génère des logs structurés (JSON) depuis des captures. Utilisé dans les SIEM pour la corrélation d'événements réseau.
  • Suricata avec pcap — analyser une capture avec les règles IDS offline
    suricata -c suricata.yaml -r capture.pcap
  • En CTF : tshark + grep + Python = trio gagnant pour les .pcap complexes

Wireshark en CTF — Les Techniques Gagnantes

Les challenges réseau CTF suivent des patterns récurrents — les connaître accélère tout

🏴 Patterns CTF .pcap classiques
  • Flag en HTTP clair — Follow TCP Stream sur les requêtes HTTP, chercher SOOPHA{ dans les réponses ou les requêtes POST
  • Flag encodé en Base64 — Follow Stream, copier les données, décoder. Chercher les chaînes = et == à la fin
  • Flag dans les paquets DNS — les requêtes TXT contiennent parfois des données. Filtrer dns.txt
  • Fichier caché en HTTP — Export Objects → HTTP pour extraire et inspecter
  • Flag en ICMP — les ICMP echo peuvent transporter un payload. Chercher dans le data des ICMP
  • Credentials FTP — Follow TCP Stream sur port 21 → USER et PASS en clair
🔧 Script tshark pour CTF
# Extraire tous les hosts HTTP visités
$ tshark -r cap.pcap -Y http.request -T fields -e http.host | sort -u

# Extraire les usernames/passwords FTP
$ tshark -r cap.pcap -Y "ftp" -T fields -e ftp.request.command -e ftp.request.arg

# Lister toutes les requêtes DNS
$ tshark -r cap.pcap -Y dns.qry.name -T fields -e dns.qry.name | sort -u

# Extraire tous les fichiers HTTP
$ tshark -r cap.pcap --export-objects http,./extracted/

# Chercher un flag dans le pcap
$ strings capture.pcap | grep -i "soopha{"
$ strings capture.pcap | grep -E "[A-Z]+\{[^\}]+\}"
// Slide 13 · Capturer sur WiFi · Mode Monitor · Limites légales

Captures WiFi & Contexte Africain

Capturer sur le sans-fil — et pourquoi le WiFi public africain est un risque majeur

📡 Capture WiFi — Mode Monitor
  • En mode normal, Wireshark ne capture que le trafic de ta machine + broadcast
  • En mode promiscuous, il capture tout le trafic du même VLAN/segment switch (si hub, pas switch)
  • Mode Monitor (WiFi) — capture TOUT le trafic de l'air sur le canal radio, même les trames d'autres réseaux
    sudo airmon-ng start wlan0 → wlan0mon
  • Mode Monitor nécessite une carte WiFi compatible (Alfa AWUS036ACH recommandée)
  • Capturer le WiFi d'autrui sans autorisation = ILLÉGAL — uniquement sur ton propre réseau ou en lab
🌍 Risques WiFi public en Afrique
  • Les hotspots WiFi dans les aéroports, hôtels, cafés de Dakar, Kinshasa, Libreville, Abidjan sont souvent non chiffrés ou WEP
  • Tout trafic HTTP est lisible par n'importe qui sur le même réseau — logins, sessions, emails
  • Les Evil Twin attacks sont fréquents : un attaquant crée un hotspot WiFi avec le même nom que le WiFi de l'hôtel
  • Protection : utiliser un VPN (WireGuard) sur tout réseau public — même HTTPS n'est pas suffisant contre les DNS leaks
  • Extension HTTPS Everywhere + DNS over HTTPS comme minimum de base
