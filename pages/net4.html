VLANs — Segmenter Sans Câbler

Virtual Local Area Network — créer des réseaux logiques séparés sur la même infrastructure physique

🔀 Pourquoi segmenter avec des VLANs
  • Sans VLAN : tous les appareils sur le même switch voient tous les broadcasts → un employé peut sniffer le trafic du DAF
  • Avec VLANs : les domaines de broadcast sont séparés — VLAN 10 (Compta) ne voit pas le trafic de VLAN 20 (IT)
  • La séparation est logique, pas physique — un seul câble trunk porte plusieurs VLANs
  • Standard 802.1Q — ajoute un tag de 4 octets dans la trame Ethernet identifiant le VLAN (VLAN ID 1-4094)
  • VLANs = sécurité + performance + flexibilité + réduction des coûts de câblage
🏷️ Exemple de segmentation VLAN typique
10SERVERS10.10.10.0/24Serveurs internes — ERP, fichiers, base de données
20USERS10.10.20.0/24Postes de travail employés
30MGMT10.10.30.0/24Administration réseau (SSH vers switches)
40WIFI-CORP10.10.40.0/24WiFi des employés authentifiés
50WIFI-GUEST10.10.50.0/24WiFi visiteurs — Internet uniquement, isolé
60DMZ10.10.60.0/24Serveurs exposés Internet (web, mail, VPN)
70IOT10.10.70.0/24Caméras, imprimantes, capteurs — isolés
// Slide 03 · Trunk · Access · Inter-VLAN · Config Cisco

VLANs — Configuration Technique

Ports access vs trunk · Inter-VLAN routing · Configuration Cisco et Linux

⚙️ Port Access vs Port Trunk
  • Port Access — connecté à un PC ou serveur. Appartient à un seul VLAN. La trame sort sans tag 802.1Q (le device ne voit pas les VLANs)
  • Port Trunk — connecté à un autre switch ou routeur. Transporte plusieurs VLANs. Les trames sont taguées 802.1Q avec leur VLAN ID
  • Native VLAN — le VLAN non tagué sur un trunk (défaut : VLAN 1). Changer le Native VLAN pour éviter le VLAN hopping
  • VLAN Hopping — attaque qui permet de sortir de son VLAN en forgeant des double tags 802.1Q
🔧 Config Cisco IOS (switch)
# Créer les VLANs
SW1# vlan 10
SW1# name SERVERS

# Port Access (vers PC)
SW1# interface fa0/1
SW1# switchport mode access
SW1# switchport access vlan 20

# Port Trunk (vers routeur)
SW1# interface gi0/1
SW1# switchport mode trunk
SW1# switchport trunk allowed vlan 10,20,30
SW1# switchport trunk native vlan 99
# Inter-VLAN routing sur Linux (Router-on-a-Stick)
$ ip link add link eth0 name eth0.10 type vlan id 10
$ ip addr add 10.10.10.1/24 dev eth0.10
$ ip link add link eth0 name eth0.20 type vlan id 20
$ ip addr add 10.10.20.1/24 dev eth0.20
# Le trafic inter-VLAN passe par le routeur — contrôlé par des ACL/firewall

Zones de Sécurité — Structurer le Réseau

Chaque zone a un niveau de confiance et des règles de flux précises — la base de toute architecture sécurisée

🌐 INTERNET Réseau non maîtrisé — confiance = 0. Tout ce qui vient d'Internet est suspect par défaut. Aucun flux entrant autorisé sauf sur les services exposés en DMZ. Confiance : 0/10
↕ FIREWALL EXTERNE (règles strictes)
🟡 DMZ Zone démilitarisée — serveurs exposés à Internet mais isolés du réseau interne. Si compromis, l'attaquant ne peut pas aller plus loin. Serveurs web, mail, VPN, reverse proxy. Confiance : 3/10
↕ FIREWALL INTERNE (segmentation)
🔵 INTERNE Réseau des employés — postes de travail, imprimantes, partages de fichiers. Flux contrôlés entre départements via VLANs. Accès Internet filtré par proxy. Pas d'accès direct à la zone Critique. Confiance : 6/10
↕ SEGMENTATION (ACL/VLAN)
🟢 CRITIQUE Base de données, ERP, données financières, sauvegardes. Accès ultra-restreint — seuls quelques serveurs et admins autorisés. Monitoring intensif. Chiffrement obligatoire des données au repos. Confiance : 9/10
↕ ISOLATION TOTALE
🟣 MANAGEMENT Administration des équipements réseau et sécurité — switches, firewalls, hyperviseurs. Accès exclusivement via jump server (bastion). MFA obligatoire. Aucune navigation Internet depuis cette zone. Confiance : 10/10

Firewalls — Les 4 Générations

Chaque génération apporte une intelligence supplémentaire — comprendre ce que chacun peut et ne peut pas faire

1. Packet Filter — Stateless
Analyse chaque paquet indépendamment — IP source/dest, port, protocole. Ne comprend pas le contexte d'une connexion. Très rapide mais limité. Vulnérable au IP spoofing.
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
nftables, ACL Cisco (ipchains)
2. Stateful Inspection
Maintient une table des connexions actives (état). Un paquet de retour légitime est autorisé automatiquement. Bloque les paquets hors contexte. Standard depuis les années 2000.
Cisco ASA, pfSense, Windows Firewall
iptables -m state --state ESTABLISHED,RELATED
3. NGFW — Next Generation
Deep Packet Inspection (DPI) — inspecte le contenu applicatif. Identification de l'application (pas juste le port). IPS intégré, détection de malwares, SSL inspection, user-identity awareness.
Palo Alto, Fortinet FortiGate, Cisco Firepower
Check Point, Sophos XG, pfSense + Suricata
4. WAF — Web Application Firewall
Spécialisé couche 7 HTTP/HTTPS. Bloque les attaques web : SQLi, XSS, CSRF, LFI, RFI, traversal de répertoire. Comprend le protocole HTTP en profondeur. Complémentaire au NGFW.
ModSecurity, AWS WAF, Cloudflare WAF
F5 BIG-IP ASM, Imperva · règles OWASP CRS
Cloud Firewall / Security Groups
Firewall défini par logiciel dans les clouds (AWS Security Groups, Azure NSG, GCP Firewall Rules). Attaché à chaque instance. Stateful par défaut. Géré par API/IaC (Terraform).
AWS Security Group · Azure NSG · GCP Firewall
Deny all by default, allow port 443 from 0.0.0.0/0
IDS vs IPS
IDS (Intrusion Detection System) — détecte et alerte sans bloquer. IPS (Intrusion Prevention System) — détecte ET bloque en temps réel. Souvent intégré dans les NGFW. Outils : Snort, Suricata, Zeek.
Suricata · Snort · Zeek (Bro) · Wazuh · OSSEC

Architecture DMZ — Exposer sans Sacrifier

Demilitarized Zone — la zone tampon entre Internet hostile et le réseau interne précieux

🔥 Single Firewall DMZ
Internet
  │
[ Firewall unique ] — 3 interfaces :
  ├── interface WAN → Internet
  ├── interface DMZ → 10.10.60.0/24
  │    ├── Serveur Web (Apache/Nginx)
  │    ├── Serveur Mail (Postfix)
  │    └── Reverse Proxy (Nginx/HAProxy)
  └── interface LAN → 192.168.1.0/24
        ├── Postes utilisateurs
        └── Serveurs internes

⚠ Si le firewall est compromis, tout tombe
🏆 Dual Firewall DMZ (Gold Standard)
Internet
  │
[ Firewall EXTERNE ] ← règles strictes
  │ Allow 80,443 from ANY to DMZ
  │ Deny ALL from Internet to LAN
  │
DMZ — 10.10.60.0/24
  ├── Web Server, Mail, VPN, Proxy
  │
[ Firewall INTERNE ] ← règles strictes
  │ Allow DMZ:8080 to DB:3306
  │ Deny ALL from DMZ to LAN
  │
LAN — 192.168.1.0/24
✅ Compromission DMZ ≠ compromission LAN
"La DMZ, c'est le sas d'entrée d'une ambassade. Les visiteurs entrent dans le sas (DMZ), mais ne peuvent pas accéder directement aux bureaux sensibles (LAN). Si un visiteur est dangereux, il est contenu dans le sas."

Règles de Firewall — L'Art de la Politique Deny-All

La règle d'or : tout ce qui n'est pas explicitement autorisé est interdit — jamais l'inverse

📋 Exemple de ruleset firewall — Architecture DMZ dual firewall
// FIREWALL EXTERNE — Internet → DMZ
ALLOWANYDMZ:10.10.60.xTCP 80,443; trafic web public
ALLOWANYDMZ:10.10.60.xUDP 1194; VPN clients
ALLOWADMIN_IPMGMT_ZONETCP 22; SSH admin seulement
DENYANYLAN_192.168.xANY; jamais direct Internet → LAN
DENYANYANYANY; DENY ALL — règle finale implicite
// FIREWALL INTERNE — DMZ → LAN
ALLOWDMZ:WebSrvLAN:DBTCP 3306; web → base de données
ALLOWLANDMZTCP 443; accès interne aux apps web
ALLOWESTABLISHEDANYANY; connexions établies (stateful)
DENYDMZLANANY; DMZ ne peut pas initier vers LAN
DENYANYANYANY; DENY ALL final
// Slide 08 · iptables · nftables · Linux firewall · Pratique

iptables & nftables — Le Firewall Linux

Maîtriser le firewall Linux — indispensable pour tout admin système et pentester

🔥 iptables — commandes essentielles
# Voir les règles actuelles
$ iptables -L -n -v --line-numbers

# Politique par défaut DENY ALL
$ iptables -P INPUT DROP
$ iptables -P FORWARD DROP
$ iptables -P OUTPUT ACCEPT

# Autoriser SSH (port 22)
$ iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Autoriser connexions établies
$ iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Autoriser loopback
$ iptables -A INPUT -i lo -j ACCEPT

# Rate limiting SSH (anti bruteforce)
$ iptables -A INPUT -p tcp --dport 22 -m limit --limit 3/min -j ACCEPT

# Sauvegarder les règles
$ iptables-save > /etc/iptables/rules.v4
⚡ nftables — le successeur moderne
# nftables remplace iptables, ebtables, arptables
$ nft list ruleset

# Exemple de config nftables
table inet filter {
 chain input {
  type filter hook input priority 0;
  policy drop;
  iifname lo accept;
  ct state established,related accept;
  tcp dport 22 accept;
  tcp dport {80, 443} accept;
  icmp type echo-request limit rate 5/second accept;
 }
}

$ nft -f /etc/nftables.conf

Routage — Trouver le Meilleur Chemin

Comment les routeurs décident où envoyer les paquets — statique vs dynamique

📊 Table de routage — lire et comprendre
$ ip route show
default via 192.168.1.1 dev eth0              ← route par défaut
10.10.20.0/24 via 10.10.1.254 dev eth0       ← route statique VLAN 20
192.168.1.0/24 dev eth0 proto kernel         ← réseau local (auto)

# Ajouter une route statique
$ ip route add 10.0.0.0/8 via 192.168.1.254

# Route persistante (Debian/Ubuntu)
# /etc/netplan/01-config.yaml :
routes: [{to: 10.0.0.0/8, via: 192.168.1.254}]
🔄 Routage dynamique — OSPF & BGP
  • Routage statique — admin configure manuellement chaque route. Simple pour petits réseaux, ne s'adapte pas aux pannes
  • OSPF (Open Shortest Path First) — protocole interne (IGP). Algorithme Dijkstra. Convergence rapide. Utilisé dans les réseaux d'entreprise et les campus
  • BGP (Border Gateway Protocol) — protocole d'Internet. Relie les systèmes autonomes (AS). Utilisé par tous les FAI africains. Vulnérable au BGP Hijacking
  • EIGRP, RIP — protocoles Cisco propriétaires ou anciens, moins utilisés
  • BGP Hijacking — annoncer frauduleusement des préfixes IP pour détourner le trafic (attaque nationale, espionnage)
// Slide 10 · Load Balancer · HA · VRRP · Redondance

Load Balancing & Haute Disponibilité

Ne jamais avoir un seul point de défaillance — l'architecture qui reste debout

Load Balancer — Couche 4 (TCP)
Distribue le trafic TCP/UDP entre plusieurs serveurs selon leur IP:port. Algorithmes : Round Robin, Least Connections, IP Hash. Aucune connaissance du contenu HTTP.
HAProxy (mode TCP) · LVS · AWS NLB · Nginx stream
Load Balancer — Couche 7 (HTTP)
Inspect le contenu HTTP — route selon l'URL, les headers, les cookies. Peut faire du SSL termination, de la réécriture d'URL, du caching, de la compression.
Nginx, HAProxy (mode HTTP), Traefik, AWS ALB, Cloudflare
VRRP — Virtual Router Redundancy
Deux routeurs partagent une IP virtuelle. Le routeur Master répond. Si le Master tombe, le Backup prend automatiquement l'IP virtuelle. Failover en quelques secondes. Standard RFC 5798.
keepalived sur Linux · Cisco HSRP (propriétaire) · GLBP
Bonding / LACP — Agrégation de liens
Combiner plusieurs interfaces réseau physiques en une interface logique — plus de débit (LACP/802.3ad) et tolérance aux pannes d'un câble. Standard entre serveurs et switches.
bonding Linux (mode 802.3ad) · Cisco EtherChannel
STP — Spanning Tree Protocol
Élimine les boucles réseau dans les topologies redondantes entre switches. Bloque les ports redondants et les débloque en cas de panne. Versions : STP, RSTP (rapide), MSTP.
show spanning-tree vlan 10 (Cisco) · RSTP converge en <1s
Anycast — Routage vers le plus proche
La même adresse IP annoncée depuis plusieurs points de présence (PoP) dans le monde. Le trafic est automatiquement routé vers le PoP le plus proche. Utilisé par Cloudflare, les DNS publics.
Cloudflare 1.1.1.1 · Google 8.8.8.8 · Anycast BGP

Zero Trust — "Ne Jamais Faire Confiance, Toujours Vérifier"

Le paradigme qui remplace le modèle château-fort — indispensable avec le cloud et le télétravail

🏰 Ancien modèle vs Zero Trust
  • Ancien modèle "château fort" — périmètre réseau = frontière de confiance. Ce qui est à l'intérieur est de confiance. Une seule brèche = compromission totale
  • En 2024, le périmètre n'existe plus — cloud, télétravail, BYOD, SaaS ont tout dissous
  • Zero Trust — aucun utilisateur, aucun device, aucun réseau n'est de confiance par défaut, même à l'intérieur
  • Chaque accès est vérifié : qui ? quel device ? depuis où ? pour quoi ?
  • Principe du moindre privilège — accès minimal nécessaire, rien de plus
🔑 Les 5 piliers du Zero Trust
  • Identité — MFA obligatoire, vérification continue (pas juste à la connexion)
  • Device — vérification de la conformité du device (MDM, EDR, patch level)
  • Réseau — micro-segmentation, cryptage de tout le trafic est-ouest
  • Application — accès par application, pas par réseau (ZTNA remplace VPN)
  • Données — classification, DLP, chiffrement au repos et en transit
  • Solutions : Cloudflare Zero Trust, Zscaler, Palo Alto Prisma, Microsoft Entra
"Le VPN dit : 'Entre dans mon réseau et tu peux tout voir.' ZTNA dit : 'Prouve qui tu es, depuis quel device, et je te donne accès uniquement à cette application spécifique.' Voilà pourquoi les entreprises africaines en croissance adoptent Zero Trust."
// Slide 12 · Micro-segmentation · East-West · Defense in Depth · QCM requis

Micro-segmentation & Defense in Depth

Chaque serveur est son propre périmètre — stopper le mouvement latéral d'un attaquant

↔️ Trafic Nord-Sud vs Est-Ouest
  • Trafic Nord-Sud — entre Internet et le datacenter. Protégé par le firewall périmétrique. Bien contrôlé depuis longtemps.
  • Trafic Est-Ouest — entre serveurs à l'intérieur du datacenter. Représente 70-80% du trafic total. Historiquement peu ou pas filtré.
  • Un attaquant qui entre dans un serveur peut se déplacer librement dans le datacenter si le trafic est-ouest n'est pas filtré (lateral movement)
  • Micro-segmentation — chaque workload a son propre firewall. Un serveur web ne peut parler qu'au serveur d'app, pas directement à la BDD.
  • Solutions : VMware NSX, Cisco ACI, Calico (Kubernetes NetworkPolicy)
🛡️ Defense in Depth — Couches de défense
1
Périmètre — Firewall externe, WAF, DDoS protection (Cloudflare)
2
Réseau interne — VLANs, firewall interne, IDS/IPS
3
Hôte — EDR/antivirus, patch management, durcissement OS
4
Application — WAF applicatif, validation des inputs, authentification forte
5
Données — Chiffrement, DLP, contrôle d'accès granulaire (IAM)

Architecture Réseau Typique — PME Africaine

Un exemple concret adapté aux contraintes et réalités du contexte africain en 2026

🏗️ Architecture recommandée (budget moyen)
Fibre/4G FAI (2 opérateurs pour redondance)
  ↓
Routeur pfSense ou Mikrotik
  · Firewall stateful
  · VPN WireGuard (télétravail)
  · Load balancing 2 FAI
  ↓
Switch managé (Mikrotik CRS ou Cisco SG)
  · VLAN 10: Serveurs (NAS, ERP)
  · VLAN 20: Postes employés
  · VLAN 30: WiFi invités (isolé)
  · VLAN 40: Management
  · VLAN 50: Caméras/IoT
  ↓
WAF Cloudflare (plan gratuit → Pro)
  · DDoS protection
  · SSL/TLS automatique
  · Cache CDN Afrique
💡 Conseils pratiques contexte africain
  • Double opérateur — MTN + Airtel/Orange pour la redondance. pfSense gère le failover automatique
  • Cloudflare gratuit — DDoS, WAF basique, CDN, DNS, SSL. Indispensable pour toute PME exposée sur Internet
  • Mikrotik — excellent rapport qualité/prix, très répandu en Afrique, communauté active en français
  • UPS obligatoire — les coupures de courant détruisent les équipements réseau sans onduleur
  • Backup 3-2-1 — 3 copies, 2 supports différents, 1 hors site (cloud chiffré)
  • NAS Synology — stockage centralisé, VPN intégré, snapshots automatiques, accessible budget africain
  • Éviter les équipements contrefaits (faux Cisco) — firmware inconnu = backdoor potentielle
