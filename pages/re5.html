L'Obfuscation — Rendre le Binaire Illisible

Comprendre les motivations et apprendre à détecter les signaux d'alerte

🎭 Pourquoi obfusquer ?
  • Malwares — cacher URLs C2, clés de chiffrement, payloads pour éviter l'AV
  • Licences logiciels — protéger les algorithmes de validation anti-crack
  • CTF — empêcher la résolution triviale par strings ou ltrace
  • DRM — protection anti-copie dans jeux et médias
  • L'obfuscation ralentit, elle n'arrête pas — c'est de la sécurité par l'obscurité
🔍 Signaux d'alerte obfuscation
  • strings presque vide — peu de chaînes lisibles dans le binaire → obfuscation certaine
  • Haute entropie Ghidra — zones > 7.5 bits/byte → données chiffrées ou packées
  • Tableaux de bytes dans .rodata — longues séquences de bytes sans sens apparent
  • Boucle XOR/ADD/SUB sur tableau — routine de déchiffrement inline
  • Premier réflexe : Ghidra → Analysis → Entropy → localiser les zones suspectes
"L'obfuscation, c'est un coffre-fort avec la serrure visible. Tu sais qu'il y a quelque chose dedans, tu vois le mécanisme, et avec les bons outils, tu l'ouvres. La question n'est pas si, mais combien de temps."

XOR — Propriétés Fondamentales

Pourquoi XOR est l'opération préférée des auteurs de malwares depuis 40 ans

⚡ Les 4 propriétés clés
  • Réversible avec la même clé — A ^ K = B → B ^ K = A. Encode ET décode avec la même opération et la même clé
  • A ^ A = 0 — XOR d'une valeur avec elle-même vaut toujours 0
  • A ^ 0 = A — XOR avec 0 est une identité, la valeur ne change pas
  • Commutativité — A ^ B = B ^ A, ordre indifférent
  • Un seul algoritme suffit pour encoder ET décoder → code compact et discret
🔬 Exemple : "SOOPHA" XOR 0x42
CharASCII hexXOR 0x42Encodé
'S'0x530x53 ^ 0x420x11
'O'0x4F0x4F ^ 0x420x0D
'O'0x4F0x4F ^ 0x420x0D
'P'0x500x50 ^ 0x420x12
'H'0x480x48 ^ 0x420x0A
'A'0x410x41 ^ 0x420x03
# Python — encode puis décode avec la même fonction
$ python3 -c "k=0x42;s='SOOPHA';e=[c^k for c in s.encode()];d=[c^k for c in e];print(bytes(e),bytes(d))"
b'\x11\r\r\x12\n\x03'  b'SOOPHA'  ← même fonction, XOR K deux fois = identité
// Slide 04 · Brute force · Known-plaintext · QCM requis

Casser le XOR Monobyte — 3 Techniques

La clé fait 1 byte = 256 possibilités max. Aucun XOR monobyte ne résiste plus de 5 secondes

🔓 Trois méthodes du plus rapide au plus robuste
── MÉTHODE 1 : Known-plaintext (la plus rapide) ─────────────
# Si le flag commence par "SOOPHA{" → on connaît les 7 premiers bytes
encoded = [0x11, 0x0d, 0x0d, 0x12, 0x0a, 0x03, 0x79]
key = encoded[0] ^ ord('S')  # 0x11 ^ 0x53 = 0x42
print(f"Clé trouvée : 0x{key:02x}")           # 0x42 !

── MÉTHODE 2 : Brute force 256 clés ────────────────────────
for key in range(256):
  decoded = bytes([b ^ key for b in encoded])
  if b'SOOPHA' in decoded or decoded.isascii():
    print(f"KEY=0x{key:02x}: {decoded}")

── MÉTHODE 3 : Frequency analysis ──────────────────────────
# Le byte le plus fréquent ≈ chiffrement de l'espace (0x20)
most_common = max(set(encoded), key=encoded.count)
key = most_common ^ 0x20  # hypothèse : texte anglais/ASCII
⚡ xortool — automatique
$ pip install xortool
$ xortool -c 20 encoded.bin
 Key: 0x42 (length: 1)
 Decrypted: SOOPHA{...}
🎯 Indices dans Ghidra
  • Repérer l'instruction XOR reg, 0x42 dans une boucle
  • La constante dans l'instruction XOR = la clé
  • L'adresse du tableau = position dans .rodata

XOR Multibyte & Rolling XOR

Les variantes avancées — plus difficiles à casser, très présentes dans les malwares réels

🔑 XOR Multibyte — clé de N bytes
# La clé se répète cycliquement
# Clé = "Base" = [0x42, 0x61, 0x73, 0x65]

key = [0x42, 0x61, 0x73, 0x65]
decoded = bytes([
  b ^ key[i % len(key)]
  for i, b in enumerate(encoded)
])

# i%4 → cycle 0,1,2,3,0,1,2,3...
# Clé visible dans Ghidra : tableau de 4 bytes
🔄 Rolling XOR — clé évolutive
# La clé change à chaque byte !
# Nouvelle clé = byte chiffré précédent

key = 0x42  # clé initiale seed
result = []
for b in encoded:
  decoded_byte = b ^ key
  result.append(decoded_byte)
  key = b  # nouvelle clé = ciphertext
print(bytes(result))
🔬 Comment détecter multibyte vs monobyte dans Ghidra
1
Monobyte : instruction XOR byte [rdi+rax], 0x42 — constante immédiate dans l'instruction
2
Multibyte : MOVZX ecx, byte [key_array+rax%N] puis XOR [data+rax], ecx — accès à un tableau de clés
3
Rolling : après le XOR, MOV [key], byte [data+rax] — la clé est mise à jour avec le ciphertext
// Slide 06 · Base64 · ROT13 · Caesar · QCM requis

Base64, ROT & Caesar — Identifier et Décoder

Les encodages classiques présents dans CTF et malwares — les reconnaître au premier coup d'œil

Base64
Alphabet A-Z a-z 0-9 +/=. Toujours un multiple de 4 chars. Se termine par 0, 1 ou 2 '='. Entropie ~6 bits/char.
U09PUEJB → "SOOPHA" (base64)
echo "U09PUEJB" | base64 -d
ROT13
Rotation de 13 lettres. Idempotent : ROT13(ROT13(x)) = x. Fréquent dans les CTF basiques et les easter eggs de développeurs.
FBBUUN → "SOOPHA" (rot13)
echo "FBBUUN" | tr 'A-Za-z' 'N-ZA-Mn-za-m'
Caesar (ROT-N)
Décalage alphabétique de N positions. Brute force : tester 25 possibilités. Très facile à casser avec CyberChef → ROT Brute Force.
VRRGOB → "SOOPHA" (rot-3 inverse)
CyberChef → ROT13 → Amount: 3
Hex encoding
Données représentées en hexadécimal. Toujours des paires 0-9 a-f. Parfois avec préfixe 0x ou séparateurs : ou espace.
53 4f 4f 50 48 41 → "SOOPHA"
python3 -c "print(bytes.fromhex('534f4f504841'))"
URL encoding
Caractères encodés en %XX. Commun dans les malwares qui communiquent via HTTP et dans les challenges web + RE hybrides.
%53%4F%4F%50%48%41 → "SOOPHA"
python3 -c "from urllib.parse import unquote; print(unquote('%53%4F...'))"
Base32
Alphabet A-Z 2-7 uniquement. Toujours majuscule. Padding avec =. Moins courant que Base64 mais présent dans certains malwares.
KNQXI2LT → "SOOPHA" (approx)
python3 -c "import base64; print(base64.b32decode('...'))"
# CyberChef en ligne — https://gchq.github.io/CyberChef/
# Glisser les données → "Magic" → détecte automatiquement l'encodage
$ echo "U09PUEJBX3Rlc3Q=" | base64 -d
SOOPHA_test

CyberChef — L'Outil de Décodage Ultime

L'outil en ligne de la GCHQ — indispensable dans toute analyse RE

// Slide 08 · Stack Strings · Anti-strings · QCM requis

Stack Strings — L'Obfuscation Anti-strings

La technique préférée des malwares modernes pour cacher les chaînes de caractères

🧩 Qu'est-ce qu'une stack string ?
// Code C original
char *url = "http://c2.evil.io";
// string visible dans .rodata
// strings ./malware → révèle l'URL
// Stack string obfusquée
mov byte [rsp+0], 0x68 ; 'h'
mov byte [rsp+1], 0x74 ; 't'
mov byte [rsp+2], 0x74 ; 't'
mov byte [rsp+3], 0x70 ; 'p'
; strings → RIEN !
🔍 Détecter les stack strings
  • Dans Ghidra : longue séquence de MOV byte [rbp-N], IMM consécutifs
  • Chaque constante correspond à un caractère ASCII (0x20 à 0x7E)
  • Flare-IDA plugin FLOSS (outil gratuit de FireEye) : extrait automatiquement les stack strings
  • GDB : BP après la construction → x/s $rsp révèle la string assemblée
🛠️ FLOSS — extraction automatique
$ pip install flare-floss
$ floss ./malware.exe
 FLOSS extracted strings:
 http://c2.evil.io/gate.php
 Mozilla/5.0 (Windows NT)
 SOOPHA{stack_strings_found}

Identifier XOR dans Ghidra — Patterns Visuels

Reconnaître une routine de déchiffrement XOR au premier coup d'œil dans le décompilateur

🔬 Les 3 patterns XOR à reconnaître dans Ghidra
── PATTERN 1 : XOR monobyte inline ─────────────────────────
// Décompilateur Ghidra :
local_len = 0x1e;                   // longueur = 30
i = 0;
while (i < local_len) {
  DAT_00104020[i] = DAT_00104020[i] ^ 0x42;
  i++;
}                                    // clé visible = 0x42

── PATTERN 2 : XOR fonction dédiée ─────────────────────────
decrypt_payload(DAT_00104020, 0x1e, key_array, 4);
// → suivre decrypt_payload → voir l'algo XOR à l'intérieur

── PATTERN 3 : Comparaison après XOR ───────────────────────
if ((input[i] ^ 0x42) != expected[i]) { return 0; }
// → XOR appliqué à l'input avant comparaison → clé = 0x42
📍 Localiser le tableau encodé
  • Note l'adresse du tableau (ex: DAT_00104020)
  • Ghidra → Data → Create Array → byte[N] → rend la mémoire lisible
  • Script Ghidra Python → extraire les bytes et décoder automatiquement
🐍 Script extraction Ghidra
>>> addr = toAddr(0x00104020)
>>> raw = [getByte(addr.add(i))&0xff for i in range(30)]
>>> flag = ''.join(chr(b^0x42) for b in raw)
>>> print(flag)
// Slide 10 · Packing · UPX · Entropie · QCM requis

Packing & Entropie — Le Niveau Supérieur

Quand XOR ne suffit plus — les packers compressent et chiffrent tout le binaire

📦 Qu'est-ce qu'un packer ?
  • Un packer compresse ET chiffre le binaire original → le vrai code n'existe qu'en mémoire au runtime
  • Le stub (déchiffreur) décompresse le vrai code → l'exécute → analyse statique ne voit que le stub
  • UPX — le packer le plus courant (open source, utilisé par legit apps ET malwares)
  • Themida, VMProtect, Enigma — packers commerciaux anti-RE avancés
  • Binaire packé : strings vide + entropie très haute + taille anormalement petite
📊 Mesurer l'entropie
# Avec file + binwalk
$ binwalk -E ./crackme
 ENTROPY (8 bits/byte) → packé !
 ENTROPY (6 bits/byte) → code normal

# Détecter UPX
$ upx -t ./crackme
 Testing ./crackme [OK] → UPX détecté

# Décompresser UPX
$ upx -d ./crackme -o unpacked
 Unpacked 1 file → analyse normale !
🎯 Stratégie face à un binaire packé
OPTION A
UPX → dépack
Si c'est UPX (détecté par upx -t ou strings "UPX" dans le binaire) → upx -d → le binaire original est récupéré → analyse normale avec Ghidra + GDB
OPTION B
Dump mémoire
Pour les packers custom : GDB → laisser le stub se déchiffrer → breakpoint sur l'OEP (Original Entry Point) → dump de la mémoire avec process_dump ou pmap + dd
OPTION C
Sandbox
Soumettre à Any.run, Cuckoo, REMnux → observer le comportement sans analyser le packer. Les IOCs (URLs, fichiers créés) sont souvent suffisants

// Slide 12 · Techniques avancées · Polymorphisme · Chiffrement

Techniques d'Obfuscation Avancées

Au-delà du XOR — les méthodes des malwares de niveau APT

🔁 Code polymorphe
Le code se réécrit lui-même à chaque exécution. Les octets des instructions changent mais la sémantique reste identique. Rend la signature AV impossible.
Approche RE : analyser le comportement (dynamique), pas les octets
🪃 Junk code insertion
Instructions inutiles insérées entre le vrai code : NOP, MOV rax, rax, PUSH/POP. Gonfle le binaire et rend la lecture difficile. Très courant dans les crackmes CTF.
Ghidra : Function Graph → identifier les blocs sans effets
🔀 Control flow flattening
Le flux de contrôle normal (if/while/for) est aplati via une dispatch table. Toutes les branches passent par un état central. LLVM-obfuscator utilise cette technique.
Reconnaître : switch géant avec états → tracer chaque état
🔢 Arithmetic substitution
Remplacer les opérations simples par des équivalents complexes : x+y → x - (~y) - 1, NOT x → -x - 1. Le compilateur ne peut pas simplifier ces formes obfusquées.
Identifier : opérations arithmétiques sur des masques étranges
"Face à l'obfuscation avancée, la dynamique prime : peu importe comment le code est écrit, ce qui arrive en mémoire à runtime est toujours le vrai texte. Un breakpoint au bon endroit vaut mieux que 3 heures de lecture de code obfusqué."
// Slide 13 · Diagnostic rapide · Méthode obfuscation

Diagnostiquer une Obfuscation en 60 Secondes

Le bon réflexe n'est pas de tout casser — c'est d'abord de reconnaître le motif

strings quasi vide
Souvent stack strings, XOR, chiffrement ou packing.
Indice : peu de texte lisible ≠ programme vide
Boucle + clé fixe
Signature classique d'un XOR simple.
buf[i] ^= 0x37
Tableau d'octets opaque
Probable payload encodé ou configuration cachée.
db 0x11,0x72,0x5A...
Construction caractère par caractère
Très souvent stack strings anti-strings.
mov byte ptr [rbp-0x20], 0x68
✅ Méthode d'attaque
  • Identifier la famille : encodage, XOR, stack strings, packing.
  • Choisir l'outil : Python, CyberChef, Ghidra, GDB, entropie.
  • Valider vite sur un petit extrait avant d'automatiser tout le flux.
⚠️ Pièges
  • Confondre Base64 et chiffrement.
  • Brute-forcer trop tôt alors qu'un known-plaintext suffit.
  • Oublier que la vraie valeur n'apparaît parfois qu'au runtime.
