Qu'est-ce que le Reverse Engineering ?

Comprendre sans le code source — L'art de lire les traces du compilateur

🔍 Définition technique
  • Analyser un binaire compilé pour reconstruire sa logique interne sans son code source
  • Le compilateur traduit le C/C++/Rust en instructions machine — le RE fait l'inverse
  • On travaille sur l'assembleur brut et sur le pseudo-code produit par les décompilateurs
  • Discipline au croisement entre programmation bas niveau, algorithmique et patience
⚖️ Cadre légal — toujours vérifier
  • CTF — toujours légal, c'est l'objectif du challenge
  • Malware analysis — légal pour la défense et la recherche
  • Interopérabilité — légal dans de nombreuses juridictions (directive UE)
  • Bug bounty — légal dans le périmètre du programme
  • Logiciels propriétaires sans autorisation — ILLÉGAL dans la plupart des pays
  • DRM et protections anti-contournement — DMCA, directive EUCD
"Le reverse engineering, c'est de l'archéologie numérique. Tu lis les traces laissées par le compilateur pour reconstruire l'intention originale du développeur — sans jamais avoir vu le plan d'origine."

Pourquoi Faire du Reverse Engineering ?

Les 6 raisons légitimes qui font du RE une compétence clé en cybersécurité

🏴 CTF — Capture The Flag
Les challenges RE sont présents dans tous les CTF. Un crackme ou un binaire avec un flag caché — tu dois trouver comment il valide le mot de passe.

🔍 Vulnerability Research
Trouver des 0-day dans des logiciels dont le code source n'est pas disponible. Analyser les patches pour identifier les vulnérabilités corrigées.

// Slide 04 · Compilation · C → Binaire · La transformation

La Chaîne de Compilation

Comment ton code C devient un exécutable binaire — et pourquoi c'est réversible

1. Code source
.c / .cpp
Le code que le développeur écrit — lisible, haut niveau, avec des noms de variables explicites et des commentaires.
2. Préprocesseur
#include #define
Résout les #include, remplace les macros #define, gère les #ifdef. Produit un fichier C "étendu" sans macros.
3. Compilateur
gcc / clang
Traduit le C en assembleur x86-64. Les optimisations (-O2, -O3) transforment radicalement la structure du code — les boucles disparaissent, les fonctions s'inlinent.
4. Assembleur
.o / objets
Transforme l'assembleur textuel en code machine binaire — des octets. Chaque instruction devient 1 à 15 octets selon x86-64.
5. Linker
ELF final
Combine tous les fichiers objets, résout les références croisées, lie les bibliothèques système (libc). Produit l'ELF exécutable final.
# La chaîne complète en une commande
$ gcc -o challenge challenge.c  # compile + link
$ gcc -S challenge.c            # stopper après l'assembleur → .s
$ gcc -c challenge.c            # stopper après assemblage → .o
$ gcc -O2 -o challenge challenge.c # avec optimisations niveau 2

Le Format ELF — Anatomie d'un Exécutable Linux

Executable and Linkable Format — le format standard de Linux, Android et BSD

📋 ELF Header — Les métadonnées
  • Magic bytes : 0x7F 45 4C 46 (\x7FELF) — identifier un ELF à l'œil
  • Class : ELF32 (32-bit) ou ELF64 (64-bit)
  • Type : ET_EXEC (exécutable), ET_DYN (library/PIE), ET_REL (objet)
  • Machine : x86-64 (0x3E), ARM (0x28), MIPS (0x08)
  • Entry point : adresse de la première instruction à exécuter
📦 Sections principales
  • .text — code exécutable (instructions machine) — r-x
  • .rodata — données en lecture seule (strings, constantes) — r--
  • .data — variables globales initialisées (tableaux XOR) — rw-
  • .bss — variables globales non initialisées, rempli de zéros — rw-
  • .plt / .got — résolution dynamique des fonctions externes (printf, strcmp)
  • .symtab / .strtab — table des symboles (noms de fonctions si non stripped)
$ readelf -h ./challenge
ELF Header:
  Magic:  7f 45 4c 46 02 01 01 00 00 00 00 00 00 00 00 00
  Class:                   ELF64
  Type:                    DYN (Position-Independent Executable file)
  Entry point address:    0x1060

Stripped vs Not Stripped — Impact sur l'Analyse

La présence ou l'absence des symboles change tout à ton workflow

✅ NOT STRIPPED — Le rêve du reverse engineer
  • Les noms de fonctions sont présents dans la table des symboles : check_flag, validate_input, decode_xor
  • Ghidra et GDB affichent les vrais noms → navigation immédiate
  • Les noms de variables locales peuvent aussi être présents (avec -g pour les debug symbols)
  • Très fréquent en CTF débutant et intermédiaire — volontaire pour faciliter l'apprentissage
❌ STRIPPED — La réalité du monde réel
  • La section .symtab est supprimée avec strip ./binary ou -s à la compilation
  • Ghidra nomme les fonctions FUN_00401060, FUN_00401130...
  • Tu dois déduire le rôle de chaque fonction depuis son comportement
  • Malwares, logiciels commerciaux, binaires release = toujours stripped
$ file ./not_stripped
not_stripped: ELF 64-bit LSB pie executable, x86-64, dynamically linked, not stripped
$ file ./stripped_binary
stripped_binary: ELF 64-bit LSB pie executable, x86-64, dynamically linked, stripped
$ nm ./not_stripped | grep check
0000000000001280 T check_flag
// Slide 07 · Premiers réflexes · Triage d'un binaire inconnu

Les Premiers Réflexes face à un Binaire Inconnu

Le workflow de triage — 5 commandes avant toute analyse approfondie

1
file ./binary — identifier le type, l'architecture, stripped ou non, statique ou dynamique
→ ELF 64-bit LSB pie executable, x86-64, dynamically linked, not stripped
2
strings ./binary | grep -i "flag\|key\|pass\|soopha\|secret" — chercher les données hardcodées immédiatement
→ strings révèle souvent le flag directement sur les challenges débutants
3
readelf -s ./binary — lister les symboles (fonctions) si non stripped
→ chercher check_flag, validate, compare, verify dans la liste
4
ldd ./binary — afficher les bibliothèques dynamiques requises
→ libcrypto.so = chiffrement · libssl.so = TLS · libpthread.so = threads
5
./binary — exécuter et observer le comportement dans une VM isolée
→ "Enter the flag: " → indique qu'il y a une comparaison de chaîne à traquer
// Slide 08 · strings avancé · Options · QCM requis

strings — Plus Puissant qu'il n'y Paraît

L'outil le plus sous-estimé du reverse engineer — maîtrise toutes ses options

⚙️ Options importantes
# Longueur minimale (défaut : 4 chars)
$ strings -n 8 ./binary

# Afficher l'offset de chaque string
$ strings -t x ./binary
 0x2008 Enter the flag:

# Chercher dans une section spécifique
$ strings -d ./binary  # .data uniquement

# Combiner avec grep pour filtrer
$ strings ./binary | grep -E "[A-Z]{6}\{" # format flag
🎯 Ce que strings peut révéler
  • Flags hardcodés — directement lisibles dans .rodata ou .data
  • Messages d'erreur — "Wrong password", "Access denied" → logique du programme
  • URLs et IPs — serveurs C2 d'un malware, endpoints d'une API
  • Clés et secrets — clés XOR, seeds, parfois même des clés privées
  • Chemins de fichiers — /tmp/payload, /etc/passwd → comprendre les actions
  • Limite : strings encodés (XOR, Base64) n'apparaissent pas en clair
// Slide 09 · readelf · objdump · Inspection approfondie

readelf & objdump — Inspecter l'ELF en Profondeur

Les deux outils CLI natifs pour tout savoir sur un binaire sans l'ouvrir dans un GUI

readelf -h
ELF Header — architecture, type, entry point, taille des sections
readelf -h ./binary
readelf -S
Toutes les sections ELF avec leurs offsets, tailles et permissions (R/W/X)
readelf -S ./binary | grep -E "\.text|\.data|\.rodata"
readelf -s
Table des symboles — noms de fonctions, types (FUNC/OBJECT), adresses
readelf -s ./binary | grep FUNC
readelf -d
Section dynamic — bibliothèques requises (NEEDED), flags, RPATH
readelf -d ./binary | grep NEEDED
objdump -d
Désassembler la section .text complète en syntaxe AT&T (défaut) ou Intel (-M intel)
objdump -d -M intel ./binary | less
objdump -s -j .rodata
Dump hexadécimal d'une section spécifique — voir les données brutes
objdump -s -j .data ./binary
$ readelf -s ./re_challenge_1 | grep FUNC
  54: 0000000000001280  87 FUNC GLOBAL DEFAULT 14 check_flag
  55: 0000000000001060  48 FUNC GLOBAL DEFAULT 14 main
// Slide 10 · Formats binaires · ELF vs PE vs Mach-O · QCM requis

Formats Binaires — Au-delà de l'ELF

Chaque OS a son format — le RE s'adapte à tous

ELF — Linux / Android / BSD
Executable and Linkable Format. Magic bytes : \x7FELF. Utilisé pour les exécutables Linux, les .so (shared objects), les fichiers .o.
Outils : Ghidra, radare2, GDB, objdump, readelf
PE — Windows
Portable Executable. Magic bytes : MZ (0x4D5A). Format des .exe, .dll, .sys Windows. Structure différente mais concepts similaires à ELF.
Outils : Ghidra, IDA, x64dbg, PE-bear, CFF Explorer
Mach-O — macOS / iOS
Mach Object. Magic bytes : 0xFEEDFACE (32-bit) ou 0xFEEDFACF (64-bit). Format des exécutables Apple, universal binaries (fat binary pour ARM64 + x86_64).
Outils : Ghidra, Hopper, otool, nm
DEX / APK — Android
Dalvik Executable. Magic bytes : dex\n. Les APK Android sont des ZIP contenant des .dex. Le code est compilé en bytecode Dalvik ou ART (non x86-64 natif).
Outils : jadx, apktool, Bytecode Viewer, dex2jar
JAR / Class — Java
Bytecode JVM. Magic bytes : 0xCAFEBABE. Plus facile à décompiler que l'assembleur natif — les noms de classes et méthodes souvent conservés (sauf obfuscation ProGuard).
Outils : jadx, CFR, Procyon, JD-GUI
PYC — Python compilé
Bytecode Python (.pyc). Magic bytes variables selon version Python. Décompilable quasi-parfaitement vers le Python source avec uncompyle6 ou decompile3.
Outils : uncompyle6 · decompile3 · pycdas

Environnement de Travail — Tout Installer

La bonne VM, les bons outils — ne jamais analyser un malware sur sa machine principale

🖥️ VM recommandée
  • REMnux — distribution Linux spécialisée RE et malware analysis, tous outils pré-installés
  • Kali Linux — Ghidra disponible via apt install ghidra
  • VirtualBox / VMware — snapshots avant chaque analyse malware
  • Isoler la VM du réseau principal pour l'analyse malware
  • Partager seulement le dossier /challenges entre hôte et VM
🔧 Installer les outils essentiels
# Sur Kali / Ubuntu
$ sudo apt install ghidra gdb python3
$ sudo apt install binutils file ltrace strace

# pwndbg (extension GDB recommandée)
$ git clone https://github.com/pwndbg/pwndbg
$ cd pwndbg && ./setup.sh

# radare2
$ sudo apt install radare2

// Slide 13 · Workflow RE · Méthode · Fil rouge

Méthodologie de Triage — Le Workflow du Reverse Engineer

Avant Ghidra, avant GDB : prendre de bonnes décisions dans le bon ordre

PHASE 01
Identifier
Type de binaire, architecture, stripping, dépendances, hash, contexte de récupération.
PHASE 02
Trier
Chercher les signaux rapides : strings utiles, sections anormales, imports suspects, présence d'obfuscation.
PHASE 03
Formuler
Poser 2 ou 3 hypothèses de travail : crackme simple, packer, loader, malware réseau, validateur de licence.
PHASE 04
Choisir
Statique d'abord si possible ; dynamique si le sample se révèle au runtime, se déchiffre en mémoire, ou résiste à la lecture seule.
PHASE 05
Documenter
Noter les offsets, fonctions, hypothèses et indices dès le début. Le RE sans notes devient vite chaotique.
🎯 Réflexe à ancrer
  • Ne lance pas un binaire inconnu juste “pour voir”.
  • Ne t'arrête pas à strings : absence de résultat ≠ absence de logique.
  • Commence toujours par un triage minimum reproductible.
🧪 Fil rouge du parcours
  • Imagine un sample nommé lockbox : tu vas d'abord l'identifier ici…
  • …lire ses fonctions en asm dans RE·02…
  • …le cartographier dans Ghidra, puis l'exécuter dans GDB.
// Slide 14 · Conclusion · Mise en pratique · QCM final

Ce que tu dois retenir avant RE·02

Un bon reverse engineer ne commence pas par cliquer — il commence par cadrer

✅ Compétences acquises
  • Tu sais définir le reverse engineering et son cadre d'usage légitime.
  • Tu comprends la chaîne source → compilation → binaire.
  • Tu sais faire un triage initial propre avec file, strings, readelf, ldd.
  • Tu sais quand basculer vers la statique ou la dynamique.
⚠️ Erreurs fréquentes
  • Lancer trop tôt un binaire inconnu sur son poste principal.
  • Croire que not stripped veut dire “facile”.
  • Confondre encodage, chiffrement, packing et simple absence de chaînes lisibles.
La suite logique, c'est RE·02 : apprendre à lire ce que le compilateur a vraiment produit. Sans assembleur, Ghidra ne sera jamais qu'un GPS flou.
