L'Analyse Dynamique — Quand et Pourquoi ?

Les cas où l'analyse statique atteint ses limites — la dynamique prend le relais

🚧 Limites de la statique
  • Binaires packés/chiffrés — le vrai code est déchiffré en mémoire au runtime
  • Valeurs runtime inconnues — input utilisateur, contenu de fichiers, réponses réseau
  • Branches complexes — difficile de suivre toutes les conditions sans exécuter
  • Auto-modification — le binaire modifie son propre code pendant l'exécution
  • Ghidra seul ne peut pas voir ce qui se passe APRÈS le déchiffrement
⚡ Force de la dynamique
  • Voir les valeurs réelles dans les registres et la mémoire au moment précis
  • Lire le flag avant la comparaison — même si chiffré dans le binaire
  • Modifier les registres pour bypasser les conditions sans recompiler
  • Observer les appels système (fichiers, réseau, processus) en temps réel
  • Combinée avec Ghidra : outil ultime — statique pour la carte, dynamique pour la boussole
"Sur un CTF RE, si strings et Ghidra ne révèlent pas le flag directement — mets un breakpoint avant le strcmp, lance GDB, et lis $rsi. 9 fois sur 10, le flag est là, en clair, dans le registre."

GDB + pwndbg — Le Duo Indispensable

GDB seul est austère — pwndbg lui donne une interface moderne et puissante

📦 Installation
# GDB (généralement déjà installé)
$ sudo apt install gdb
$ gdb --version
 GNU gdb 13.2

# pwndbg — surcouche Python sur GDB
$ git clone https://github.com/pwndbg/pwndbg
$ cd pwndbg && ./setup.sh

# Vérifier l'installation
$ gdb ./crackme
pwndbg> # pwndbg actif ✅
🎯 Pourquoi pwndbg ?
  • Vue registres colorée — RAX, RIP, RFLAGS affichés automatiquement
  • Disassembly automatique — le code autour du RIP à chaque step
  • Stack visuelle — contenu de la pile affiché proprement
  • Commandes étendues — cyclic, vmmap, heap
  • REPL Python — scripter le debugger en Python directement
  • Alternatives : peda, GEF (même principe, styles différents)

// Slide 04 · Breakpoints · Run · Navigation · QCM requis

Breakpoints — Figer l'Exécution au Bon Moment

Le breakpoint est l'arme principale du reverse engineer dynamique

🎯 Les commandes essentielles GDB
# Ouvrir le binaire dans GDB
$ gdb ./crackme_04
pwndbg>

# Poser un breakpoint sur une fonction
pwndbg> b check_flag         # par nom (si non stripped)
pwndbg> b *0x00401280        # par adresse exacte
pwndbg> b *main+0x3a        # relatif à main
 Breakpoint 1 at 0x401280

# Lancer le programme avec des arguments
pwndbg> run SOOPHA_test_flag
 Starting program... Breakpoint 1 hit → PAUSE ici

# Voir tous les breakpoints
pwndbg> info breakpoints     # liste + status
pwndbg> delete 1            # supprimer BP #1
pwndbg> disable 1          # désactiver BP #1
⚡ Types de breakpoints
  • b addr — software breakpoint (instruction INT3 injectée)
  • hb addr — hardware breakpoint (registre DR0-DR3 CPU)
  • tb addr — temporary breakpoint (se supprime après 1 hit)
  • condition — b check_flag if $rdi==5
🔄 Commandes de contrôle
  • c / continue — reprendre jusqu'au prochain BP
  • n / next — instruction suivante (step over)
  • s / step — entrer dans les fonctions (step into)
  • finish — exécuter jusqu'à la fin de la fonction courante
// Slide 05 · Registres · Mémoire · x/ command

Inspecter Registres et Mémoire

La commande x/ — l'outil le plus puissant de GDB pour lire la mémoire

🔬 La commande x/ — syntaxe complète
# Syntaxe : x/[N][f][u] ADDRESS
# N = nombre d'unités · f = format · u = unit size

# Lire une chaîne en clair (LE FORMAT LE PLUS UTILE)
pwndbg> x/s $rdi
 0x7ffd...: "SOOPHA_test_flag"
pwndbg> x/s $rsi
 0x404020: "SOOPHA{dynamic_rules_gdb}" ← LE FLAG !

# Lire des bytes hexadécimaux (dump mémoire)
pwndbg> x/16xb $rsi
 0x404020: 0x53 0x4f 0x4f 0x50 0x48 0x41 0x7b ...

# Lire des qwords (64-bit) sur la stack
pwndbg> x/8gx $rsp
 0x7ffd...: 0x00007ffd9a8b1234 0x0000000000000001

# Lire des instructions assembleur depuis une adresse
pwndbg> x/10i $rip
 => 0x401280: cmp eax, 0x1d
# Format x/ — mémo complet
x=hex d=decimal s=string i=instruction c=char
b=byte h=halfword(2) w=word(4) g=giant(8)
# Exemple : x/16xb = 16 bytes en hexa · x/4gx = 4 qwords hexa
// Slide 06 · Bypass strcmp · Modifier registres · QCM requis

Bypass d'une Comparaison — La Technique Clé CTF

Deux approches pour bypasser check_flag sans connaître le flag à l'avance

🎯 Méthode 1 — Lire le flag dans les registres
# Identifier l'appel à strcmp dans Ghidra
# call strcmp @ 0x40128e

# Mettre un BP juste avant
pwndbg> b *0x40128e
pwndbg> run mon_input
 Breakpoint → juste avant strcmp

# RDI = s1 (mon input), RSI = s2 (le flag attendu)
pwndbg> x/s $rdi
 "mon_input"
pwndbg> x/s $rsi
 "SOOPHA{dynamic_gdb_master}" ← FLAG !
⚡ Méthode 2 — Modifier EAX après cmp
# BP après le retour de check_flag
pwndbg> b *main+0x42
pwndbg> run mauvais_flag
 Breakpoint hit — EAX = 0 (false)

# Forcer EAX = 1 (true) → succès garanti
pwndbg> set $eax = 1
pwndbg> continue
 "Congratulations! Access granted."

# Alternative : modifier le Zero Flag
pwndbg> set $eflags = $eflags | 0x40 # ZF=1
"La méthode 1 est préférable en CTF car elle donne le vrai flag. La méthode 2 bypass juste la vérification — tu passes, mais tu n'as pas le flag pour le soumettre. Toujours préférer lire le flag à bypasser."
// Slide 07 · ni · si · finish · Stepping

Stepping — Avancer Instruction par Instruction

Contrôler l'exécution avec précision — chaque instruction compte

🚶 Les 4 modes de stepping
  • ni (next instruction) — exécute une instruction, SANS entrer dans les fonctions appelées (step over)
  • si (step instruction) — exécute une instruction, ENTRE dans les fonctions appelées (step into)
  • finish — exécute jusqu'au RET de la fonction courante, s'arrête au retour
  • until *addr — exécute jusqu'à atteindre une adresse spécifique
  • Règle : utiliser ni par défaut, si seulement pour entrer dans une fonction précise
⌨️ Workflow de stepping pratique
pwndbg> b main
pwndbg> run input_test
 Breakpoint 1 → main+0

pwndbg> ni          # avancer une instruction
pwndbg> ni 5        # avancer 5 instructions
pwndbg> si          # entrer dans check_flag
pwndbg> finish      # sortir de check_flag
 Value returned is $rax = 0x0 (false)
💡 Inspecter les registres après chaque step
pwndbg> info registers  # tous les registres
pwndbg> print $rax      # valeur d'un registre précis
pwndbg> regs           # pwndbg : vue colorée compacte
pwndbg> context        # pwndbg : vue complète (regs+stack+code)
// Slide 08 · Watchpoints · Surveillance mémoire · QCM requis

Watchpoints — Surveiller la Mémoire

Être alerté automatiquement quand une variable est lue ou modifiée

👁️ Watchpoints — commandes et usages
# Watchpoint sur une variable — s'arrête quand elle change
pwndbg> watch *0x404020       # address du flag encodé
 Hardware watchpoint 2: *0x404020

# Watchpoint en lecture (rwatch)
pwndbg> rwatch *0x404020      # s'arrête quand lue

# Watchpoint lecture+écriture (awatch)
pwndbg> awatch *0x404020      # les deux

# Exemple de sortie après déclenchement
 Hardware watchpoint 2: *0x404020
 Old value = 0x42
 New value = 0x53            # 'S' de SOOPHA{...}
 decrypt_flag () at check.c:42
🎯 Cas d'usage watchpoints
  • Binaire qui déchiffre le flag en mémoire au runtime → watch sur le buffer
  • Identifier quelle fonction modifie une variable globale suspecte
  • Suivre l'écriture d'une clé XOR générée dynamiquement
  • Combiné avec x/s après déclenchement → voir le résultat déchiffré
⚠️ Limites des watchpoints
  • Maximum 4 hardware watchpoints simultanés (limitation CPU)
  • Les software watchpoints sont très lents (vérification à chaque instruction)
  • Astuce : utiliser l'adresse du résultat de malloc() pour les buffers heap
  • info watchpoints pour voir l'état actuel
// Slide 09 · ltrace · strace · Tracer sans debugger

ltrace & strace — Tracer sans Debugger

Observer les appels de bibliothèques et système — souvent plus rapide que GDB

📚 ltrace — Library calls
# Tracer tous les appels de bibliothèques
$ ltrace ./crackme mauvais_flag

 printf("Enter flag: ")
 fgets(0x7fff..., 256, stdin) = 0x7fff...
 strcmp("mauvais_flag\n", "SOOPHA{ltrace_wins}") = -1
 puts("Wrong flag, try again.")

# strcmp révèle : s1=input · s2=FLAG !
$ ./crackme SOOPHA{ltrace_wins}
 Congratulations! ✅
⚙️ strace — Syscalls
# Tracer tous les appels système
$ strace ./crackme input_test

 execve("./crackme", [...]) = 0
 mmap(NULL, 4096, ...) = 0x7f...
 open("/etc/flag.txt", O_RDONLY) = 3
 read(3, "SOOPHA{file_based_flag}", 64)
 write(1, "Wrong!", 6)

# strace révèle : flag lu depuis fichier !
⚡ Quand utiliser ltrace vs strace vs GDB ?
ltrace
strcmp visible
Essayer en premier si le binaire compare des chaînes. Révèle les deux arguments de strcmp en clair. 30 secondes pour avoir le flag.
strace
I/O suspecte
Le binaire lit des fichiers, fait des connexions réseau, utilise des variables d'environnement. strace montre tout le comportement système.
GDB
Logique complexe
Chiffrement custom, algorithme non-trivial, anti-debug. GDB donne un contrôle total mais demande plus d'effort.
// Slide 10 · GDB Script · Python · Automatisation · QCM requis

Scripting GDB — Automatiser l'Analyse

Python + GDB = analyse automatisée — aller au-delà des commandes manuelles

🐍 Script GDB Python — extraire un flag XOR automatiquement
# Fichier : solve.py — script GDB Python
import gdb

# Définir une commande personnalisée
class SolveFlag(gdb.Command):
  def __init__(self):
    super().__init__("solve", gdb.COMMAND_USER)

  def invoke(self, arg, from_tty):
    # Mettre un BP sur check_flag
    gdb.execute("b check_flag")
    gdb.execute("run dummy_input")
    # Lire RSI (2e arg = flag attendu)
    rsi = gdb.parse_and_eval("$rsi")
    flag = gdb.execute("x/s $rsi", to_string=True)
    print(f"[*] FLAG: {flag}")

SolveFlag()

# Utilisation dans GDB
pwndbg> source solve.py
pwndbg> solve
 [*] FLAG: SOOPHA{gdb_python_auto}
# Autre usage : .gdbinit — commandes auto au démarrage
$ cat ~/.gdbinit
 set disassembly-flavor intel     # syntaxe Intel par défaut
 set pagination off               # pas de pause "more"
 define hook-stop                 # auto à chaque BP
  regs                             # afficher registres auto
 end
// Slide 11 · Choisir le bon breakpoint · Checklist

Choisir le Bon Breakpoint

Un bon breakpoint économise 20 minutes de stepping inutile

🎯 Breakpoints utiles
  • break main — démarrer proprement et prendre le contrôle tôt
  • break *0xADDR — viser une adresse repérée dans Ghidra
  • break strcmp — intercepter une validation de flag ou mot de passe
  • catch syscall ptrace — idéal contre l'anti-debug Linux
  • break if conditionnel — s'arrêter seulement quand une valeur t'intéresse
🧰 Checklist runtime
  • run pour lancer
  • ni pour avancer sans entrer
  • si pour entrer dans l'appel
  • info registers pour lire l'état CPU
  • x/s, x/20gx pour lire la mémoire
  • bt pour comprendre la pile d'appels
Ne pose pas des breakpoints “au hasard”. Pose-les là où une décision importante est prise : comparaison, déchiffrement, anti-debug, appel système.

// Slide 13 · Anti-debug · Problèmes courants · Solutions

Problèmes Courants en Analyse Dynamique

Les obstacles typiques et comment les contourner

⛔ ASLR — Adresses aléatoires
Address Space Layout Randomization randomise les adresses à chaque exécution. Les adresses Ghidra ne correspondent plus aux adresses GDB.
pwndbg> vmmap → adresse réelle
offset = runtime - ghidra_base
⛔ Stripped — Pas de symboles
Les fonctions n'ont pas de noms — GDB ne peut pas faire "b check_flag". Il faut utiliser les adresses exactes trouvées dans Ghidra.
b *0x401280 → adresse absolue
Ou : b *main+0x3a → relatif
⚠️ ptrace anti-debug
Le binaire détecte GDB via ptrace et se ferme. Solution : patcher l'appel ptrace dans Ghidra avant d'analyser dynamiquement.
NOP les instructions ptrace
catch syscall ptrace → nop
⚠️ Segfault immédiat
Le binaire crash avant d'atteindre le flag. Souvent un bug volontaire ou une dépendance manquante. Analyser le backtrace GDB.
pwndbg> bt → backtrace complet
Vérifier ldd — bibliothèques manquantes
💡 ASLR désactiver (local)
Pour faciliter l'analyse locale, désactiver l'ASLR. Attention : ne jamais faire ça sur une machine de prod.
echo 0 | sudo tee /proc/sys/kernel/randomize_va_space
💡 PIE offset calcul
Avec PIE activé, l'offset entre l'adresse Ghidra et l'adresse runtime est constant pour une session. vmmap donne la base.
base = vmmap[0].start
real_addr = ghidra_addr - 0x400000 + base
