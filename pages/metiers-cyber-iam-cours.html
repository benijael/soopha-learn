L'écosystème complet : IAM, IGA, PAM, AM, IDaaS

IAM — Identity & Access Management

Le cadre global. Qui sont les utilisateurs ? Comment sont-ils créés ? Quel cycle de vie les régit ? IAM est le terme générique qui couvre toutes les sous-disciplines ci-dessous.

// IAM = IGA + AM + PAM. Le tout, pas une partie.
IGA — Identity Governance & Administration

La gouvernance : qui a accès à quoi, est-ce légitime, qui a validé ? IGA ajoute les processus d'audit, de recertification des accès, de gestion des rôles métier et de demande d'accès via workflow. C'est le cœur du quotidien d'un consultant IGA/IAM.

One IdentitySailPointSaviynt
PAM — Privileged Access Management

Les comptes à hauts privilèges : admins, root, service accounts, comptes de service. Le PAM sécurise ces accès critiques via un coffre-fort de mots de passe (vaulting), une rotation automatique des secrets, un enregistrement de sessions et une approbation juste-à-temps (JIT).

CyberArkBeyondTrustDelinea
AM — Access Management (Authentification)

Comment je prouve que je suis moi ? SSO (Single Sign-On), MFA, fédération d'identités via SAML 2.0, OAuth 2.0 / OIDC. L'AM est le pont entre l'utilisateur et les applications.

OktaEntra IDPing IdentityForgeRock
IDaaS — Identity as a Service

L'IAM entier dans le cloud, en mode SaaS. Pas de serveurs à gérer. Okta, Entra ID, Ping Identity sont les leaders. Idéal pour les entreprises en migration cloud ou les startups.

Comment ça s'articule sur un flux réel ?
RH saisit un recrutement → SIRH notifie
IGA reçoit l'événement → crée l'identité
IAM provisionne → AD + messagerie
AM prend le relais → SSO + MFA activés
PAM si compte sensible → vaulting + JIT
// 02 · Fondamentaux

Les 4 piliers conceptuels de l'IAM

🔄 JML — Joiner · Mover · Leaver

Le cycle de vie complet d'une identité en entreprise :

  • Joiner — Nouvelle embauche : création du compte, attribution des droits selon le profil de poste, notification à l'utilisateur
  • Mover — Changement de poste/pays/département : modification des droits, révocation des anciens, pas d'accumulation
  • Leaver — Départ : désactivation immédiate de tous les accès, archivage du compte, transfert des données
// Le Leaver non géré = ghost account = faille majeure
🏗 RBAC · ABAC · ReBAC
  • RBAC — Role-Based Access Control. Les droits suivent le rôle métier. Simple, le plus répandu, limite : rigide.
  • ABAC — Attribute-Based. Les droits dépendent d'attributs multiples (département, pays, heure, classification données). Flexible mais complexe à maintenir.
  • ReBAC — Relation-Based (modèle Google Zanzibar). Droits basés sur les relations entre objets. Utilisé par Google Drive, Notion.
⚖️ Least Privilege & Need-to-Know
  • Chaque identité a uniquement les droits nécessaires à sa fonction — ni plus, ni moins
  • Need-to-Know : accès accordé seulement si la personne a besoin de cette information pour son travail
  • Réduit drastiquement la surface d'attaque en cas de compromission d'un compte
// RGPD Art. 5.1.c : minimisation des données. Least Privilege = conformité by design.
🚫 SoD — Séparation des tâches

Celui qui crée ne peut pas valider. Celui qui valide ne peut pas exécuter. Empêche la fraude interne et les erreurs humaines.

  • Exemple bancaire : le comptable saisit un virement → le responsable valide → la banque exécute. 3 entités distinctes.
  • En IAM : l'administrateur qui crée un compte ne peut pas auto-approuver les droits sensibles

🔴 CAS CONCRET #1

Onboarding automatisé à grande échelle

Scénario réel · Groupe industriel Afrique Francophone · 200 nouvelles recrues simultanées
Le problème sans IAM automatisé
→ 200 mails/tickets manuels vers l'IT
→ Création des comptes AD à la main
→ Délai moyen : 3 à 5 jours ouvrés
→ Droits accordés "au feeling", pas selon le poste
→ Zéro traçabilité des attributions
→ Coût caché : ~3h IT par onboarding manuel

Sur 200 recrutements × 3h = 600h IT perdues. En freelance à 700€/j : 52 500€ de gaspillage.

La solution — Onboarding event-driven
01
Connecteur SIRH — Sync automatique avec SAP HCM / Workday via API REST ou SCIM. Dès qu'une fiche employé est créée côté RH, l'IGA en est informé en quasi-temps réel.
02
Role Mining — Le poste ("Acheteur Senior · Dakar") mappe automatiquement vers un bundle de droits prédéfini : groupes AD, messagerie, ERP, accès VPN, dossier réseau.
03
Provisioning multi-cible — Création simultanée du compte AD, boîte mail, profil ERP, badge logique. Notification automatique à l'employé avec ses credentials provisoires.
04
Workflow d'approbation — Pour les accès sensibles (données financières, RH) : le N+1 reçoit une demande d'approbation automatique. Pas de droits sans validation humaine.

Extrait code VB.NET — Règle de provisioning
' Mapping Pays → Bundle de droits
Public Overrides Sub Provision(person As IPerson)
  Dim country As String = person.GetValue("CountryCode")
  Select Case country
    Case "GA" : AssignRole(person, "BUNDLE_GABON_STD")
    Case "SN" : AssignRole(person, "BUNDLE_SENEGAL_STD")
    Case "CI" : AssignRole(person, "BUNDLE_COTEDIVOIRE_STD")
    Case Else : AssignRole(person, "BUNDLE_INTERNATIONAL")
  End Select
  SendWelcomeNotification(person)
End Sub

🔴 CAS CONCRET #2

Le Ghost Account — l'attaque qui vient de l'intérieur

Scénario composite · Vrai vecteur d'attaque · Analyse technique complète
Le scénario
Kofi, directeur financier, quitte l'entreprise le 31 mars.
Son compte AD reste actif le 1er juin.
Un ancien prestataire, qui connaît ses credentials, s'y connecte.
Il exporte 3 ans d'historique client depuis l'ERP.
L'entreprise ne le détecte que 6 semaines plus tard, sur alerte SIEM.
  • Cause #1 : le départ n'a pas été notifié à l'IT par les RH dans les délais
  • Cause #2 : aucun processus Leaver automatisé
  • Cause #3 : pas de revue périodique des comptes inactifs
  • Cause #4 : droits excessifs (accès complet ERP pour tout l'historique)
  • Impact : violation RGPD · risque pénal · perte client · coût incident 200K+€
Réalité terrain en Afrique Francophone
🌍

Dans de nombreuses structures, la DSI est informée du départ après coup, parfois jamais. Les comptes prestataires sont encore pires : leur contrat se termine mais leurs accès VPN restent ouverts des mois. Sans processus Leaver automatisé couplé au SIRH, ce scénario se reproduit à chaque départ non tracé.

Processus Leaver automatisé — Étape par étape
J-5
Alerte managériale — Le manager est notifié : transférer les fichiers, déléguer les responsabilités, transférer la boîte mail.
J-1
Pré-désactivation — Révocation des accès non critiques. Génération d'un rapport de tous les accès actifs.
J0 AM
Désactivation cascade — AD désactivé → SSO révoqué → tokens OAuth invalidés → VPN coupé → sessions actives tuées. Tout, simultanément, en automatique.
J+1
Vérification — Rapport d'audit automatique confirmant que 100% des accès sont coupés. Envoi au RSSI.
J+30
Archivage — Compte archivé (pas supprimé) pour 5 ans, conformité légale et investigations éventuelles.
SQL — Détection des ghost accounts (audit hebdo)
-- Comptes AD actifs sans connexion depuis 60 jours
SELECT p.DisplayName, p.SamAccountName,
  p.LastLogonTimestamp, hr.ExitDate,
  DATEDIFF(day,p.LastLogonTimestamp,GETDATE()) AS DaysInactive
FROM ADSAccount p
LEFT JOIN HRPerson hr ON p.UID_Person = hr.UID_Person
WHERE p.AccountDisabled = 0
AND ( DATEDIFF(day,p.LastLogonTimestamp,GETDATE()) > 60
  OR hr.ExitDate < GETDATE()
  OR hr.UID_Person IS NULL )
ORDER BY DaysInactive DESC

🔴 CAS CONCRET #3

Campagne de Recertification des Accès

Exigence ISO 27001 · SOX · RGPD Art. 5 · Exercice annuel redouté
Le "Privilege Creep" — pourquoi c'est dangereux
Les droits s'accumulent comme de la dette technique.
Marie entre comme Assistante RH en 2020 → promotion Responsable Formation → mutation en Finance.
En 2024 : elle a encore accès à la paie, aux offres d'emploi et aux budgets.
Violation du principe de moindre privilège. Violation SoD. Violation RGPD.

Étude Gartner : en moyenne, les employés conservent 60% d'accès inutiles 12 mois après une mutation.

Sans IGA — Le processus manuel (réalité)
1. Export Excel 5000 lignes depuis AD
2. Email à 80 managers avec pièce jointe
3. Attente 3 semaines… 40% sans réponse
4. Relances manuelles infinies par l'équipe IT
5. Modifications manuelles des droits
6. Rapport inexploitable par l'auditeur
→ 3 mois de travail · résultat non certifiable
Avec IGA — Campagne pilotée en 3 jours
01
Lancement en 1 clic — Périmètre, reviewers, délais, règles d'escalade configurés depuis l'interface IGA. Toutes les données AD sont déjà synchronisées.
02
Portail manager self-service — Chaque manager voit ses collaborateurs et leurs accès actuels. Il approuve ou révoque en 1 clic, avec commentaire obligatoire pour les révocations.
03
Escalade automatique — Pas de réponse sous 72h → alerte N+2. Pas de réponse sous 5j → alerte RSSI + escalade dirigeant.
04
Application instantanée — Chaque révocation est exécutée immédiatement dans l'AD et les systèmes cibles. Pas de délai entre la décision et l'effet.
05
Rapport d'audit certifiable — Export PDF/Excel avec toutes les décisions, timestamps, justifications, signatures électroniques. Prêt pour KPMG ou Deloitte le lendemain.

// 07 · Protocoles

Les protocoles techniques que tout consultant IAM doit maîtriser

LDAP / Active Directory

Le protocole historique d'annuaire. Active Directory est l'implémentation Microsoft de LDAP. Structure hiérarchique : Forêt → Domaine → OU → Utilisateur/Groupe.

// Requête LDAP — chercher un utilisateur
(&(objectClass=user)
  (sAMAccountName=mbata.beni))
// Attributs clés IAM
distinguishedName · memberOf
userAccountControl · pwdLastSet
SAML 2.0 — Fédération d'identités Web

Protocole XML de fédération entre un IdP (Identity Provider) et un SP (Service Provider). Utilisé pour le SSO entre entreprises ou vers des apps cloud (Salesforce, Office 365).

Utilisateur → clique sur "Se connecter"
SP → redirige vers l'IdP (Entra ID)
IdP → authentifie + génère Assertion XML
SP → valide l'assertion → accès accordé
// Aucun mot de passe ne transite côté SP
OAuth 2.0 + OIDC — Autorisation moderne

OAuth 2.0 = délégation d'autorisation (une app accède à tes données en ton nom). OIDC (OpenID Connect) ajoute l'authentification au-dessus d'OAuth 2.0. Le duo standard des apps modernes et mobiles.

App demande → Authorization Code
User autorise → IdP retourne code
App échange → Access Token + ID Token
App utilise → Access Token pour API
// OIDC = OAuth 2.0 + ID Token (JWT)
SCIM 2.0 — Provisioning automatisé

Le protocole REST/JSON de provisioning. Standardise la création, modification et suppression d'identités entre un IGA et les applications cibles. Remplace les connecteurs custom.

POST /Users → Créer un utilisateur
PATCH /Users/id → Modifier des attributs
DELETE /Users/id → Désactiver le compte
GET /Groups → Lister les groupes
// Supporté nativement : Okta, Entra, Slack, Box

Ce qu'il faut retenir

✅ LES 5 VÉRITÉS IAM
  • JML — Joiner, Mover, Leaver : le cycle de vie de toute identité
  • Source de vérité unique — Le SIRH pilote, l'IGA exécute, jamais l'inverse
  • Least Privilege — Jamais plus de droits que nécessaire, jamais "au cas où"
  • Automatisation — L'humain valide, la machine provisionne et désactivie
  • Recertification — Les accès sans revue régulière deviennent des risques certifiés
⚠ LES 3 ERREURS QUI COÛTENT CHER
  • Déployer une solution IGA sans cartographier les rôles métier d'abord
  • Traiter l'IAM comme un projet IT — c'est un projet d'entreprise (RH, Finance, Métiers)
  • Ne pas automatiser le Leaver — chaque départ manuel est un ghost account en sursis
